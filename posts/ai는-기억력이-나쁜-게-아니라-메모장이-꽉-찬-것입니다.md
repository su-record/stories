# AI는 기억력이 나쁜 게 아니라, 메모장이 꽉 찬 것입니다

## AI가 잊는 순간은 갑자기 옵니다

![AI는 기억력이 나쁜 게 아니라, 메모장이 꽉 찬 것입니다](/images/ai는-기억력이-나쁜-게-아니라-메모장이-꽉-찬-것입니다/fig1.png)


AI가 갑자기 딴소리를 하는 것처럼 보일 때가 있습니다. 40분 전에 정한 조건을 지운 듯이 말합니다.

쇼핑몰 랜딩 페이지를 같이 만든다고 해보겠습니다. 초반에 무료배송 조건을 이렇게 정했습니다.

- 5만 원 이상 구매 시 무료배송
- 제주, 도서산간 지역은 제외
- 첫 구매 쿠폰과 중복 적용 불가

문제는 그다음입니다. 대화가 70번째 메시지를 넘은 뒤 모델이 이런 문장을 씁니다.

> 전 지역 무료배송과 첫 구매 쿠폰을 동시에 제공합니다.

틀린 답입니다. 그렇다고 모델이 갑자기 게을러진 건 아닙니다. 앞에서 정한 조건이 현재 작업 공간 밖으로 밀려났을 가능성이 큽니다.

2026년 5월 11일 기준 공개 설명에서도 컨텍스트 윈도우는 대체로 이렇게 설명됩니다. 모델이 한 번에 기억하고 처리할 수 있는 텍스트의 총량입니다.

여기서 중요한 말은 한 번에입니다. AI는 대화 전체를 사람처럼 떠올리지 않습니다. 현재 창 안에 들어온 텍스트를 바탕으로 다음 토큰을 계산합니다.

---

## 컨텍스트 윈도우는 작업대입니다

컨텍스트 윈도우는 하드디스크라기보다 작업대에 가깝습니다. 작업대 위에 올라온 종이만 볼 수 있습니다.

대화, 시스템 지시문, 첨부 문서, 코드, 이전 답변, 현재 질문이 모두 같은 공간을 씁니다. 모델이 답변을 쓸 자리도 남아 있어야 합니다.

아래 숫자는 실제 제품별 한계가 아닙니다. 긴 대화에서 예산이 어떻게 소모되는지 보여주기 위한 예시입니다.

| 항목 | 내용 | 예상 토큰 |
|---|---:|---:|
| 시스템 지시문 | 말투, 안전 규칙, 도구 사용법 | 1,500 |
| 프로젝트 문서 | 요구사항, 정책, 예외 조건 | 18,000 |
| 회의록 6개 | 결정 사항, 미정 사항, 액션 아이템 | 42,000 |
| 누적 대화 | 질문과 답변 80회 | 56,000 |
| 현재 요청 | 새 랜딩 페이지 작성 요청 | 2,000 |
| 답변 예약 공간 | 모델이 답변을 쓸 공간 | 8,000 |
| 합계 | 128k 창에 거의 도달 | 127,500 |

128k 컨텍스트 모델이라도 남는 공간은 500토큰뿐입니다. 여기에 10,000토큰짜리 새 문서를 붙이면 무언가는 빠집니다.

제품마다 처리 방식은 다릅니다. 오래된 메시지를 잘라내기도 하고, 일부를 요약하기도 합니다. 중요도가 낮다고 판단한 내용을 약하게 반영하는 경우도 있습니다.

사용자 눈에는 증상이 비슷합니다. 앞에서 말한 조건을 잊습니다. 이름을 바꿉니다. 금지한 표현을 다시 씁니다.

## 긴 대화가 망가지는 세 가지 패턴

첫 번째는 요구사항이 대화 속에만 있는 경우입니다. 이때 모델은 마지막 몇 턴의 분위기에 쉽게 끌립니다.

초반에 정한 제약보다 방금 사용자가 말한 문장이 더 강하게 작동합니다. 대화가 길어질수록 자주 보이는 현상입니다.

두 번째는 문서와 대화를 한 창에 계속 쌓는 방식입니다. 30페이지 문서를 붙이고, 그 위에 수정 지시를 50번 더합니다.

처음에는 잘 됩니다. 그래서 더 위험합니다. 어느 순간부터 모델은 오래된 표, 예외 조항, 숫자를 놓치기 시작합니다.

세 번째는 에이전트 작업에서 상태를 채팅에 맡기는 경우입니다. 자동화에서는 이 방식이 특히 약합니다.

Google Gemini Spark 같은 24/7 에이전트형 도구를 떠올리면 이해하기 쉽습니다. 태스크, 스킬, 스케줄을 이용해 Gmail과 캘린더 작업을 자동화합니다.

이때 지난 실행 결과를 채팅 기록에만 두면 위험합니다. 밤새 실행된 작업의 상태가 다음 실행 때 빠질 수 있습니다.

| 상황 | 나쁜 방식 | 나은 방식 |
|---|---|---|
| 회의 요약 | 채팅에 계속 붙임 | 결정 사항 문서로 분리 |
| 이메일 자동화 | 이전 대화에 의존 | 스프레드시트에 처리 상태 저장 |
| 일정 관리 | 모델이 기억한다고 가정 | 캘린더 이벤트 ID를 외부에 기록 |
| 코드 수정 | 긴 로그 전체 재전송 | 실패 구간과 재현 절차만 전달 |

AI에게 기억하라고 말하는 것만으로는 부족합니다. 기억해야 할 형태를 만들어 줘야 합니다.

## 관리법은 요약, 상태 문서, 캐싱입니다

첫 번째 방법은 체크포인트 요약입니다. 저는 대화가 10회에서 15회 정도 쌓이면 중간 정리를 시킵니다.

요약은 감상문이 되면 안 됩니다. 다음 작업에 필요한 상태만 남겨야 합니다. 아래 형식이면 다시 쓰기 편합니다.

```text
지금까지의 작업을 다음 형식으로 압축합니다.

1. 최종 목표
2. 확정된 결정 사항
3. 변경된 요구사항
4. 금지 사항
5. 남은 질문
6. 다음 답변에서 반드시 반영할 기준

추측은 쓰지 않습니다.
숫자, 이름, 예외 조건은 원문 그대로 유지합니다.
```

두 번째 방법은 상태 문서입니다. 긴 프로젝트에서는 채팅보다 문서가 중심이 되는 편이 안전합니다.

```yaml
goal: B2B SaaS 랜딩 페이지 초안 작성
audience: 20명 이하 스타트업 대표
confirmed:
  price: 월 49,000원부터
  free_shipping: 해당 없음
  trial: 14일
constraints:
  - 과장 표현 금지
  - 보안 인증은 현재 준비 중으로 표기
  - 고객사 이름은 공개 승인된 3곳만 사용
open_questions:
  - 환불 정책 문구 확정 필요
last_checkpoint: 2026-06-30 14:20
```

이 문서를 매번 대화 상단에 붙이면 모델의 단기기억을 보조할 수 있습니다. 길이는 300에서 800토큰 정도가 적당합니다.

세 번째 방법은 컨텍스트 캐싱입니다. 반복해서 쓰는 긴 문서나 규칙을 캐시해두는 방식입니다.

캐싱은 무한 기억이 아닙니다. 같은 참고자료를 매번 다시 읽히는 비용과 지연을 줄이는 장치에 가깝습니다.

정책 문서, 브랜드 보이스, API 명세처럼 자주 바뀌지 않는 자료에 잘 맞습니다. 매번 달라지는 대화 상태는 요약 문서로 관리하는 편이 낫습니다.

| 관리법 | 적합한 대상 | 주의점 |
|---|---|---|
| 체크포인트 요약 | 긴 대화, 기획 회의 | 누락된 숫자를 반드시 검수 |
| 상태 문서 | 제품 요구사항, 자동화 상태 | 원본과 충돌하면 문서를 수정 |
| 컨텍스트 캐싱 | 정책, 명세, 스타일 가이드 | 최신 변경 사항은 별도로 전달 |
| 작업 분할 | 코드 리팩터링, 리서치 | 단계별 산출물을 파일로 남김 |

AI의 단기기억은 대화가 길어질수록 불안정해집니다.

앞 내용을 잊는 이유가 항상 성능 저하는 아닙니다. 컨텍스트 윈도우라는 작업 공간이 가득 찼기 때문일 수 있습니다.

긴 작업에서는 대화만 믿기보다 상태를 따로 관리하는 편이 안전합니다. 요약하고, 문서로 고정하고, 반복 자료는 캐싱해 둡니다.
