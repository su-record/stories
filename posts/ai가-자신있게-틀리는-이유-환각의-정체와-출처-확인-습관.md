AI는 모른다고 멈추기보다, 먼저 그럴듯한 문장을 만들어냅니다.

저도 이 지점에서 몇 번 당했습니다. 초안은 매끄럽고 문장도 자연스러웠습니다. 문제는 숫자였습니다.

제품 출시일, 가격, 인용문이 실제 문서와 달랐습니다. 틀린 내용이었습니다. 더 위험했던 건 그 답이 꽤 자신 있어 보였다는 점입니다.

---

## 환각은 거짓말이 아니라 빈칸 채우기입니다

![AI가 자신있게 틀리는 이유: 환각의 정체와 출처 확인 습관](/images/ai가-자신있게-틀리는-이유-환각의-정체와-출처-확인-습관/fig1.png)


환각(hallucination)은 AI가 없거나 틀린 정보를 진짜처럼 만들어내는 현상입니다. 사실이 아닌 내용을 사실처럼 말합니다.

사람은 모르면 “모른다”고 말할 수 있습니다. AI도 그렇게 답할 수는 있습니다. 다만 기본 작동 방식은 다릅니다. 다음에 올 가능성이 높은 단어를 고릅니다.

이 차이가 큽니다. AI는 진실을 직접 확인하지 않습니다. 문장 패턴, 학습 데이터, 질문 맥락을 바탕으로 답을 구성합니다.

| 상황 | 겉으로 보이는 답 | 실제 위험 |
|---|---|---|
| 출처 없는 통계 요청 | 정확한 백분율처럼 응답 | 만들어진 수치일 수 있음 |
| 최신 제품 기능 질문 | 그럴듯한 기능 목록 제시 | 출시 전 소문이 섞일 수 있음 |
| 논문·보고서 요약 | 저자명과 연도 포함 | 존재하지 않는 논문일 수 있음 |
| 검색 결과 없음 | 비슷한 사례로 대체 | 질문 자체를 바꿔 답할 수 있음 |

예를 들어 “2026년 특정 경기 결과와 순위”를 물었는데, 출처에서 match scores, results, standings가 확인되지 않았다고 해보겠습니다. 안전한 답은 “출처에서 찾지 못했습니다”입니다.

환각이 섞인 답은 다릅니다. 팀명, 점수, 순위를 만들어 냅니다. 문장은 자연스럽습니다. 그래서 더 위험합니다.

## 왜 AI는 틀렸는데도 확신하는가

주된 원인은 세 가지입니다. 데이터의 편향, 학습 과정의 한계, 모델 설계상의 문제입니다.

학습 데이터가 불완전하면 모델은 빈칸을 추론으로 메웁니다. 그 추론이 쓸모 있을 때도 있습니다. 그렇다고 사실 검증을 통과한 결론은 아닙니다.

| 원인 | 실제로 벌어지는 일 | 사용자가 보는 증상 |
|---|---|---|
| 데이터 불완전 | 최신 정보나 사내 자료가 없음 | 오래된 사실을 현재형으로 말함 |
| 데이터 편향 | 자주 등장한 패턴을 과대평가 | 유명한 사례만 정답처럼 반복 |
| 목표 함수 한계 | 그럴듯한 답변을 보상 | 모른다는 답을 피함 |
| 문맥 압축 | 긴 문서 일부를 놓침 | 앞뒤 조건이 뒤섞임 |

제가 쓰는 콘텐츠 자동화 시스템 sutory도 같은 문제를 만납니다. sutory는 hermes 기반 시스템이고, 토리와 미니가 콘텐츠를 운영합니다.

여기에 내부 지식베이스를 붙이면 답은 빨라집니다. 다만 내부 지식이 공식 출처를 대신할 수는 없습니다.

예를 들어 지식베이스에는 Google Gemini Spark가 노코드 AI 자동화 솔루션이라고 적혀 있습니다. 태스크, 스킬, 스케줄을 통해 Gmail과 캘린더 등과 연동한다고도 되어 있습니다.

여기까지는 보유 지식의 범위입니다. 문제는 그 다음입니다. 모델이 가격, 정식 출시일, 지원 국가를 덧붙이기 시작하면 환각 가능성이 생깁니다.

확인되지 않은 2026-04-27 같은 날짜를 “출시일”로 단정하면 틀릴 수 있습니다. 날짜가 검색 결과에 있었다는 사실과 제품 출시일이라는 사실은 다릅니다.

## RAG를 붙여도 환각은 사라지지 않습니다

RAG는 모델이 외부 문서를 검색한 뒤 답하게 만드는 방식입니다. 많은 팀이 이 방식으로 환각을 줄입니다.

방향은 맞습니다. 그렇다고 해결책은 아닙니다. RAG도 틀릴 수 있습니다. 특히 문서 구조가 깨질 때 문제가 커집니다.

| RAG 실패 지점 | 예시 | 결과 |
|---|---|---|
| 문서 청크 분리 오류 | 표 제목과 수치가 다른 조각으로 분리 | 잘못된 수치를 연결 |
| PDF 파싱 오류 | 각주와 본문이 섞임 | 인용 범위를 오해 |
| 검색어 불일치 | 질문 표현과 문서 표현이 다름 | 관련 없는 문서 선택 |
| 권한·버전 혼재 | 최신 문서와 구버전 동시 검색 | 폐기된 정책을 답변 |

공식 IR 자료 PDF를 넣고 “이 문서 안에서만 답하라”고 해도 끝이 아닙니다. 표가 깨지면 근거도 같이 깨집니다.

예를 들어 매출 표에서 2025년 열과 2026년 열이 어긋나면 답은 자연스럽게 틀립니다. 모델이 표 구조를 완전히 이해했다고 볼 수 없습니다.

그래서 RAG 답변에는 근거 문장 위치가 필요합니다. 파일명, 페이지, 문장 인용이 같이 나와야 합니다.

아래처럼 지시하면 최소한의 방어선이 생깁니다.

```text
역할: 검증 보조자
범위: 업로드한 PDF 안에서만 답한다.
규칙:
1. 답변마다 파일명과 페이지를 적는다.
2. 원문 문장을 1개 이상 인용한다.
3. 문서에 없으면 '문서에서 찾지 못했습니다'라고 답한다.
4. 추정, 일반 지식, 외부 기억을 섞지 않는다.
```

이 프롬프트도 완전하지는 않습니다. 그래도 기준을 “그럴듯한 답”에서 “검증 가능한 답”으로 옮겨 줍니다.

## 출처 확인은 습관이 아니라 작업 절차입니다

좋은 질문보다 더 중요한 건 확인 절차입니다. 저는 답변을 받을 때 네 단계를 고정해 둡니다.

| 단계 | 질문 | 실패 신호 |
|---|---|---|
| 1 | 이 답의 1차 출처는 무엇인가 | 블로그 재인용만 있음 |
| 2 | 수치와 날짜가 원문에 있는가 | 본문에는 없고 요약에만 있음 |
| 3 | 인용문이 실제 문장과 같은가 | 의미만 비슷하게 바뀜 |
| 4 | 모른다고 답해야 할 부분은 없는가 | 모든 항목을 채워 넣음 |

특히 사례, 수치, 인용은 따로 검증해야 합니다. 이 세 가지는 글의 신뢰를 가장 빨리 무너뜨립니다.

AI에게 “출처도 같이 줘”라고 말하는 것만으로는 부족합니다. 출처처럼 보이는 링크도 만들어질 수 있습니다.

더 나은 방법은 범위를 좁히는 것입니다. 공식 문서, IR PDF, 공시 자료, 제품 도움말처럼 원문성이 높은 자료를 먼저 줍니다.

```text
아래 자료만 사용해서 답하라.
- 공식 IR PDF
- 제품 도움말 문서
- 회사 블로그 원문

금지:
- 추정
- 재인용 블로그
- 출처 없는 숫자
- 문서에 없는 기능 설명
```

답변을 받은 뒤에는 숫자만 다시 검사합니다. 날짜, 금액, 비율, 순위, 사람 이름을 따로 뽑습니다.

```text
방금 답변에서 검증이 필요한 항목만 표로 뽑아라.
열: 항목, 값, 출처 문장, 페이지, 검증 상태
검증 상태는 확인됨 / 문서에 없음 / 애매함 중 하나로 적어라.
```

번거로운 작업입니다. 그래도 한 번의 오답이 더 비쌉니다. 잘못된 가격표는 고객 문의를 만들고, 잘못된 인용은 글 전체를 흔듭니다.

## AI 답변을 믿기 전에 봐야 할 것

AI 답변에서 가장 위험한 신호는 문장이 매끄럽다는 점입니다. 매끄러움은 정확성의 증거가 아닙니다.

저는 이제 답변을 세 부류로 나눕니다. 바로 쓰는 답, 확인 뒤 쓰는 답, 버리는 답입니다.

| 분류 | 기준 | 처리 |
|---|---|---|
| 바로 쓰는 답 | 문체 변환, 요약 구조, 초안 정리 | 사실 추가 없이 사용 |
| 확인 뒤 쓰는 답 | 수치, 날짜, 제품 기능 포함 | 원문 대조 후 사용 |
| 버리는 답 | 출처 없음, 고유명사 과다, 단정 표현 | 다시 질문하거나 폐기 |

환각은 AI가 나빠서 생기는 문제가 아닙니다. 모델의 작동 방식과 사용자의 기대가 어긋날 때 생깁니다.

AI는 초안을 빠르게 만듭니다. 검증은 사용자의 일입니다. 이 경계가 흐려지면, 틀린 문장이 업무 문서로 들어갑니다.

2026년 6월 29일 기준으로도 이 원칙은 그대로입니다. 출처가 없는 확신은 답이라기보다 검토할 후보에 가깝습니다.
