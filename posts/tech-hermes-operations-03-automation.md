## 자동화의 끝은 자동 발행이 아닙니다

콘텐츠 자동화를 만들 때 가장 위험한 착각이 있습니다.

자동화가 곧 자동 발행이라고 생각하는 것입니다.

아닙니다.

자동화는 수집, 요약, 초안, 검수, 승인, 배포를 나누는 일입니다. 발행은 마지막 권한입니다. 앞 단계가 자동이어도 마지막 권한까지 자동으로 열 필요는 없습니다.

Sutory에서는 이 선을 강하게 둡니다. Gutenberg는 `#배포-승인`이 없는 콘텐츠를 배포하지 않습니다. 초안이 완성되어도 발행이 아닙니다. 리뷰가 끝나도 발행이 아닙니다. 승인 표시가 있어야 배포합니다.

이 구조를 만들 때 Hermes의 Kanban과 Cron이 같이 쓰입니다.

---

## Kanban은 상태를 남깁니다

Hermes Kanban은 여러 프로필이 함께 쓰는 작업 보드입니다. 공식 문서 기준으로 durable SQLite-backed task board입니다. 작업은 카드가 되고, 카드는 담당자와 상태를 가집니다.

대화가 아니라 카드가 기준입니다.

| 상태 | 의미 | 운영에서 필요한 행동 |
|---|---|---|
| ready | 실행 가능한 작업 | 담당 프로필이 가져갑니다 |
| running | 작업 중 | 필요하면 heartbeat를 남깁니다 |
| blocked | 사람 결정이 필요함 | 이유를 짧게 남기고 멈춥니다 |
| done | 완료 | summary와 metadata를 남깁니다 |

이 방식의 장점은 실패했을 때 드러납니다.

채팅만 쓰면 실패가 대화 흐름 속에 묻힙니다. Kanban을 쓰면 왜 막혔는지, 누가 맡았는지, 어떤 결과를 남겼는지 기록됩니다. 다음 워커가 이어받을 수 있습니다.

멀티에이전트 운영에서 중요한 것은 누가 말을 잘했는지가 아닙니다. 일이 끊기지 않았는지입니다.

---

## Cron은 반복 실행을 맡습니다

Cron은 정해진 시간에 작업을 실행합니다. Hermes는 자연어 일정, `every 2h` 같은 표현, 일반 cron 표현식으로 예약 작업을 만들 수 있습니다. 작업에는 스킬을 붙일 수 있고, 결과를 원래 채팅이나 지정한 플랫폼 대상으로 보낼 수 있습니다. 스크립트만 실행하는 no-agent 모드도 있습니다.

예시는 이렇습니다.

```bash
hermes cron create "every 1h" "Check configured feeds and summarize new AI agent updates" --skill blogwatcher
hermes cron create "0 9 * * *" "Prepare the morning risk brief and send it to the origin chat"
```

Cron은 반복을 잘합니다. 하지만 판단과 승인까지 모두 맡기면 위험합니다.

그래서 저는 Cron을 이렇게 씁니다.

| 단계 | 담당 | 자동화 여부 |
|---|---|---|
| 자료 수집 | Cron | 자동 |
| 요약 생성 | Cron 또는 리서치 프로필 | 자동 가능 |
| 초안 작성 | 작성 프로필 | 자동 가능 |
| 검수 | 리뷰 프로필 또는 사람 | 반자동 |
| 승인 | 사람 | 수동 |
| 배포 | Gutenberg | 승인 후 실행 |

반복 실행과 최종 권한을 분리합니다.

---

## Cron이 카드를 만들고, Kanban이 상태를 맡습니다

Cron만 있으면 반복은 됩니다. 하지만 상태가 약합니다.

Kanban만 있으면 상태는 남습니다. 하지만 정해진 시간에 시작하는 힘은 약합니다.

둘을 합치면 운영 흐름이 됩니다.

예를 들어 매일 아침 콘텐츠 파이프라인을 이렇게 만들 수 있습니다.

1. 06:00 Cron이 RSS와 YouTube, X 후보를 수집합니다.
2. 요약 결과를 보고 채널에 남깁니다.
3. 의미 있는 후보가 있으면 Kanban 카드가 생깁니다.
4. Miranda가 초안을 씁니다.
5. 리뷰 담당이 사실과 문체를 봅니다.
6. 사람의 `#배포-승인`이 붙습니다.
7. Gutenberg가 채널별 포맷으로 배포하고 URL을 기록합니다.

이 흐름에서 자동화는 사람을 빼는 장치가 아닙니다. 사람이 봐야 할 지점을 줄이고, 봐야 할 지점만 남기는 장치입니다.

---

## 좋은 카드에는 완료 조건이 있습니다

Kanban을 쓴다고 자동으로 일이 좋아지지는 않습니다. 카드가 흐리면 결과도 흐립니다.

좋은 카드는 완료 조건이 분명합니다.

| 나쁜 카드 | 좋은 카드 |
|---|---|
| "Hermes 글 써줘" | "Hermes profiles 글 1편을 stories/posts에 draft md로 작성하고 build 검증" |
| "콘텐츠 자동화 해줘" | "RSS 요약 cron 후보 3개를 조사하고 schedule, deliver, failure mode 정리" |
| "발행해줘" | "#배포-승인 태그가 있는 LinkedIn 캡션을 X/LinkedIn 포맷으로 배포하고 URL 로그" |

카드에는 담당자, 산출물, 검증 방법, 차단 조건이 들어가야 합니다.

특히 발행 작업은 더 엄격해야 합니다. 승인 표시가 없으면 blocked가 맞습니다. AI가 알아서 선의로 발행하면 안 됩니다.

---

## 자동화할수록 로그가 중요해집니다

자동화가 많아지면 작업량은 줄어듭니다. 대신 확인해야 할 것은 늘어납니다.

무엇이 실행됐는지, 어디에 보냈는지, 실패했는지, 재시도했는지 남아야 합니다. 발행 작업이라면 URL과 시간이 남아야 합니다. 실패하면 backoff 후 재시도하고, 그래도 안 되면 보고해야 합니다.

이 기준은 사람 팀과 같습니다.

| 작업 | 남길 로그 |
|---|---|
| Cron 실행 | job id, schedule, 실행 시각, 결과 요약 |
| Kanban 완료 | summary, metadata, 산출물 경로 |
| 발행 | 채널, URL, 발행 시각 |
| 실패 | 오류, 재시도 횟수, 다음 조치 |

로그가 없으면 자동화는 빨라진 것이 아니라 보이지 않게 된 것입니다.

보이지 않는 자동화는 운영 자산이 아닙니다.

---

## 작게 시작하는 구성

처음부터 전체 콘텐츠 엔진을 만들 필요는 없습니다.

가장 작은 구성은 이 정도면 됩니다.

1. 매일 아침 Cron 하나가 자료를 모읍니다.
2. 결과가 좋으면 사람이 Kanban 카드 하나를 만듭니다.
3. 작성 프로필이 초안을 만듭니다.
4. 리뷰 후 승인되면 발행 프로필이 배포합니다.

이 구조만 있어도 대화창 자동화와 다릅니다.

상태가 남습니다. 권한이 나뉩니다. 승인 전 발행이 막힙니다. 실패했을 때 다시 시작할 지점이 생깁니다.

Hermes를 운영 시스템으로 쓰려면 먼저 이 정도면 충분합니다. 더 복잡한 자동화는 이 구조가 며칠 버티는지 본 뒤에 붙이면 됩니다.

---

## 기준은 하나입니다

자동화할수록 사람의 일이 사라지는 것이 아닙니다. 사람의 일이 승인과 판단으로 좁혀집니다.

Hermes Kanban은 상태를 남깁니다. Cron은 반복을 실행합니다. 프로필은 역할을 나눕니다. Gutenberg 같은 발행 담당은 승인된 것만 밖으로 보냅니다.

이 네 가지가 분리되면 작은 AI 팀이 됩니다.

카드가 없으면 상태도 없습니다. 승인선이 없으면 자동화는 배포 사고가 됩니다.
