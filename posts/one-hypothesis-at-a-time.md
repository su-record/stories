# 한 번에 한 가설만: ‘하는 김에’ 리팩토링이 일을 망치는 이유

## 리팩토링은 새로 만드는 일이 아닙니다

![리팩토링은 동작을 바꾸지 않고 내부 구조만 바꾸는 일](/images/one-hypothesis-at-a-time/01.png)
*동작 그대로*


리팩토링 PR에서 가장 위험한 문장은 “하는 김에 같이 고쳤습니다”였습니다.

겉으로는 효율적입니다. 이미 파일을 열었습니다. 테스트도 돌릴 예정입니다. 리뷰어도 어차피 봅니다. 그러다 이름 변경, 조건 수정, 폴더 이동, 타입 정리, 작은 기능 추가가 한 번에 들어갑니다.

문제는 배포 뒤에 생깁니다. 장애가 나면 어디서 잘못됐는지 찾기 어렵습니다.

WikiDocs의 「리팩토링 작은 습관」은 리팩토링을 “기존 동작을 유지한 채 내부 구조만 개선하는 작업”으로 설명합니다. 동작이 바뀌었다면 리팩토링이 아닙니다. 기능 변경입니다.

제가 겪은 실패도 여기서 시작했습니다. 주문 목록 화면의 중복 조건문을 줄이는 작업이었습니다. 원래 목표는 하나였습니다.

| 항목 | 원래 계획 | 실제 PR |
|---|---:|---:|
| 목표 | 조건문 중복 제거 | 조건문 제거, 이름 변경, 필터 기능 추가 |
| 변경 파일 | 7개 | 31개 |
| 변경 라인 | 약 140줄 | 약 860줄 |
| 리뷰 시간 | 30분 예상 | 2일 |
| 배포 결과 | 구조 개선 | 검색 필터 누락 발생 |

리뷰어는 구조를 봐야 했습니다. 그런데 필터 로직도 봐야 했고, 변수명도 바뀌어 있었습니다. 테스트 스냅샷까지 갱신돼 있었습니다.

실패 원인은 12분이면 찾을 수 있는 코드였습니다. 실제로는 4명이 3시간을 썼습니다.

---

## ‘하는 김에’는 검증 단위를 지웁니다

![한 PR에는 한 가설만 남겨야 실패 원인을 찾을 수 있다](/images/one-hypothesis-at-a-time/02.png)
*가설 하나*


소프트웨어 변경은 대부분 가설입니다. “이 중복을 함수로 빼면 읽기 쉬워진다”도 가설입니다. “이 조건을 바꾸면 버그가 사라진다”도 가설입니다.

한 PR에 가설이 5개 들어가면 실패 원인도 5개가 됩니다.

아래 코드는 리팩토링입니다. 조건의 의미를 함수 이름으로 드러냈습니다. 입력과 출력은 그대로입니다.

```ts
function canReceiveCoupon(user: User) {
  if (user.age < 19) return false;
  if (!user.email) return false;

  return true;
}
```

```ts
const isAdult = (user: User) => user.age >= 19;
const hasEmail = (user: User) => Boolean(user.email);

function canReceiveCoupon(user: User) {
  return isAdult(user) && hasEmail(user);
}
```

여기까지는 안전합니다. 테스트가 실패하면 추출 과정에서 실수한 것입니다. 확인할 범위가 좁습니다.

하지만 “하는 김에”가 붙으면 코드가 달라집니다.

```ts
const isAdult = (user: User) => user.age >= 20;
const hasVerifiedEmail = (user: User) => user.emailVerified === true;

function canReceiveCoupon(user: User) {
  return isAdult(user) && hasVerifiedEmail(user);
}
```

이제 두 가지가 바뀌었습니다. 나이 기준은 19세에서 20세가 됐습니다. 이메일은 존재 여부가 아니라 인증 여부를 봅니다.

테스트가 실패하면 질문이 갈라집니다.

- 함수 추출이 잘못됐습니까.
- 기존 정책 이해가 틀렸습니까.
- 새 정책이 맞는데 테스트가 오래됐습니까.
- 데이터 마이그레이션이 누락됐습니까.

코드는 6줄입니다. 그런데 검증해야 할 질문은 4개입니다. 이 상태에서 리뷰는 코드 읽기가 아니라 추리 게임에 가까워집니다.

AI 모델 평가도 같은 문제를 겪습니다. OpenAI의 Sol 모델은 KingBench 3에서 복잡한 수학과 에이전트 작업에 강했습니다. 반대로 웹 프론트엔드와 비주얼 구현은 Fable 5보다 약했습니다.

여기서 모델, 프롬프트, 평가 기준을 한 번에 바꾸면 결론을 낼 수 없습니다. 점수가 왜 올랐는지도 모릅니다. 왜 내려갔는지도 모릅니다. 코드 변경도 마찬가지입니다.

## 한 PR에는 한 문장만 남겨야 합니다

![‘하는 김에’ 변경을 섞으면 리뷰와 되돌리기가 어려워진다](/images/one-hypothesis-at-a-time/03.png)
*섞지 말기*


저는 리팩토링 PR 설명을 한 문장으로 줄입니다. 한 문장으로 안 되면 PR을 나눕니다.

좋은 문장은 이렇게 생겼습니다.

> 주문 필터의 중복 조건문을 `buildOrderFilter` 함수로 이동합니다.

나쁜 문장은 이렇게 생겼습니다.

> 주문 필터를 정리하면서 네이밍을 바꾸고, 만료 주문 제외 조건을 추가하고, 테스트도 최신 정책에 맞춥니다.

두 번째 문장은 PR 하나로 다루기 어렵습니다. 최소 3개입니다.

| PR | 가설 | 허용되는 변경 |
|---|---|---|
| 1 | 중복 조건문을 함수로 빼도 동작은 같다 | 함수 추출, 호출부 교체 |
| 2 | 이름을 바꾸면 의미가 더 분명하다 | 변수명, 함수명, 파일명 변경 |
| 3 | 만료 주문은 목록에서 제외해야 한다 | 조건 추가, 테스트 케이스 추가 |

순서도 중요합니다. 동작을 유지하는 리팩토링을 먼저 하고, 기능 변경은 그 다음에 합니다.

이렇게 하면 장애가 났을 때 되돌리기 쉽습니다. PR 3에서 문제가 생기면 PR 3만 되돌립니다. PR 1의 구조 개선은 남겨둘 수 있습니다.

실제 작업에서는 커밋 단위도 나눕니다.

```bash
git commit -m "refactor: extract order filter builder"
git commit -m "rename: clarify expired order flag"
git commit -m "feat: exclude expired orders from list"
```

세 커밋이 한 PR에 같이 있다면 아직 위험합니다. 리뷰어가 세 가설을 한 번에 검증해야 하기 때문입니다. 저는 보통 이런 경우 PR도 3개로 나눕니다.

작은 PR은 느려 보입니다. 실제로는 빨랐습니다. 800줄 PR 하나보다 150줄 PR 다섯 개가 더 빨리 머지됐습니다. 리뷰어가 한 번에 기억해야 할 상태가 줄었기 때문입니다.

## 리뷰 기준은 작을수록 강해집니다

리팩토링 리뷰에서 제가 보는 기준은 5개입니다. 이 기준을 통과하지 못하면 “리팩토링”이라는 이름을 붙이지 않습니다.

| 질문 | 통과 기준 | 실패 신호 |
|---|---|---|
| 동작이 그대로입니까 | 테스트 기대값이 바뀌지 않음 | 스냅샷 대량 갱신 |
| 변경 이유가 하나입니까 | PR 설명이 한 문장 | “겸사겸사”라는 표현 |
| 되돌리기 쉽습니까 | revert 범위가 작음 | 마이그레이션과 섞임 |
| 리뷰가 가능한 크기입니까 | 300줄 안팎 | 파일 20개 이상 변경 |
| 테스트가 원인을 좁힙니까 | 실패 지점이 한 기능 | 여러 도메인 동시 실패 |

300줄은 절대 규칙이 아닙니다. 기준선입니다. 타입 생성 파일이나 자동 포맷처럼 예외도 있습니다. 그래도 사람이 읽어야 하는 로직이 300줄을 넘으면 리뷰 품질은 떨어졌습니다.

가장 자주 틀렸던 부분은 테스트였습니다. 리팩토링하면서 테스트 이름을 바꾸고, 기대값도 바꾸고, 누락된 케이스도 추가했습니다. 보기에는 좋아 보였습니다.

아니었습니다. 그 순간 테스트는 안전망이 아니라 또 하나의 변경점이 됐습니다.

리팩토링 PR의 테스트는 기존 동작을 붙잡아야 합니다. 새 정책을 설명하면 안 됩니다. 새 정책 테스트는 기능 PR에서 추가합니다.

구분은 단순합니다.

- 리팩토링은 구조 변경입니다.
- 기능 추가는 동작 변경입니다.
- 이름 변경은 인지 변경입니다.
- 테스트 기대값 변경은 명세 변경입니다.

네 가지를 한 번에 넣으면 실패 원인을 잃습니다. 한 번에 한 가설만 남기면 실패해도 배울 수 있고, 되돌리기도 쉽습니다.

리팩토링할 때는 바꾸지 않을 범위를 먼저 정해두는 편이 안전했습니다.
