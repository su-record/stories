## 사용자가 “지금 무슨 일이 일어나는지” 볼 수 있게 하기

이번 `vica` 작업은 전반적으로 **실행 과정을 더 잘 보이게 만드는 것**에 집중했습니다. 기존에는 빌더가 내부적으로 열심히 움직이고 있어도, 사용자가 보기에는 결과가 툭 튀어나오는 느낌이 강했습니다. 성공하면 다행이지만, 시간이 조금 걸리거나 중간에 변경이 많을 때는 “지금 멈춘 건가?”, “어떤 파일을 건드리는 중이지?” 같은 불안함이 생길 수밖에 없었습니다.

그래서 빌더의 진행 상황을 서술형으로 보여주는 흐름을 추가했습니다. 단순히 로그를 늘어놓기보다는, 현재 어떤 판단을 하고 있고 어떤 작업 단계에 있는지 자연스럽게 전달하는 쪽에 가깝습니다. 사용자가 기다리는 동안 맥락을 잃지 않도록 하는 게 목표였습니다.

또 변경 내용을 보여주는 디프 화면에도 배경색을 더했습니다. 추가, 삭제, 변경이 시각적으로 더 잘 구분되도록 해서, 코드를 꼼꼼히 읽기 전에도 “어디가 바뀌었는지” 빠르게 감을 잡을 수 있게 했습니다. 작은 UI 변화지만, 리뷰 피로도를 꽤 줄여줄 수 있는 부분이라고 생각합니다.

## 코드 보기 경험 다듬기

두 번째 축은 코드 출력 영역의 사용성 개선입니다. 코드 색상 테마로 `one-dark`, `dracula`, `monokai` 계열을 지원하도록 확장했습니다. 개발자마다 익숙한 색감이 다르고, 특히 어두운 테마 안에서도 대비나 채도 취향이 꽤 갈리기 때문에 선택지를 늘리는 게 필요했습니다.

단순히 보기 좋은 색을 추가한 것 이상으로, 도구 출력 영역의 여백도 함께 정리했습니다. 이전에는 도구 실행 결과가 표시될 때 불필요하게 공간이 벌어지거나, 같은 내용이 중복되어 보이는 경우가 있었습니다. 이런 부분은 기능 자체를 망가뜨리지는 않지만, 계속 보다 보면 제품이 덜 정돈되어 있다는 인상을 줍니다.

이번에 여백과 중복 출력을 정리하면서 화면이 조금 더 차분해졌습니다. 특히 여러 도구 실행 결과가 연달아 나오는 상황에서 차이가 더 크게 느껴집니다. 정보는 줄이지 않되, 사용자가 읽어야 하는 노이즈는 줄이는 방향으로 손봤습니다.

## 독립 작업은 동시에, 더 빠르게

마지막으로 내부 실행 구조에 **병렬 웨이브** 개념을 넣었습니다. 한 페이즈 안에 서로 의존하지 않는 작업들이 있다면, 이제 워커들이 동시에 처리할 수 있습니다. 예전 방식은 비교적 안전하지만 순차 실행에 가까워서, 독립적인 작업까지 기다리게 되는 비효율이 있었습니다.

이번 변경으로 각 페이즈에서 병렬 처리 가능한 작업들을 묶고, 워커들이 나눠 수행하도록 개선했습니다. 물론 병렬화는 항상 조심스럽습니다. 빠르게 만드는 대신 실행 순서, 결과 병합, 상태 공유 같은 부분에서 복잡도가 올라가기 때문입니다. 그래서 모든 것을 무작정 동시에 돌리기보다는, 같은 페이즈 안에서 독립성이 확보된 작업에 한정해 적용하는 쪽으로 잡았습니다.

이 변화는 사용자가 직접 “병렬 웨이브”라는 이름을 볼 일은 많지 않을 수 있습니다. 하지만 실제로는 대기 시간이 줄고, 여러 작업이 한 번에 진행되는 느낌이 더 자연스럽게 이어질 수 있습니다. 앞에서 추가한 빌더 서술 기능과도 잘 맞습니다. 내부에서는 더 빠르게 움직이고, 외부에서는 그 움직임을 더 잘 설명해주는 형태가 되었기 때문입니다.

## 정리

이번 세 커밋은 겉으로 보기에는 UI 개선, 테마 추가, 실행 구조 개선처럼 각각 다른 작업처럼 보입니다. 하지만 공통된 방향은 꽤 분명했습니다. `vica`가 단순히 결과만 내놓는 도구가 아니라, 사용자가 과정을 이해하고 믿고 기다릴 수 있는 도구가 되도록 만드는 것입니다.

아직 더 다듬을 부분은 많습니다. 특히 서술형 진행 정보는 너무 많아도 방해가 되고, 너무 적으면 의미가 없어져서 균형을 계속 봐야 할 것 같습니다. 그래도 이번 작업으로 “돌아가고는 있는데 안 보이는” 느낌을 조금 덜어낸 것 같아 만족스럽습니다. 다음에는 이 흐름 위에서 더 안정적인 피드백과 실행 제어를 붙여가면 좋겠습니다.
