## vica 개발일지: CLI와 런타임 경계를 다시 정리했습니다

이번 작업은 겉으로 보기에는 “몇 가지 버그 수정”처럼 보이지만, 실제로는 vica가 어디까지 책임지고, 외부 CLI 실행자는 어디까지 맡을지 경계를 다시 정리한 시간이었습니다. 특히 CC/Codex CLI와 함께 동작하는 흐름, 디스코드 미러링, 채널 모델 명령 처리까지 서로 맞물려 있어서 작은 수정 하나가 전체 실행 경로에 영향을 주는 구조였습니다.

## CLI는 실행자로, ship 판정은 vica가 맡도록

가장 큰 변화는 CC/Codex CLI 상호운용 쪽입니다. 이번 단계에서는 CLI를 `sub` 실행자로 두고, 최종 ship 판정은 vica가 계속 담당하도록 정리했습니다.

처음에는 CLI가 더 많은 판단을 가져가도 될 것처럼 보였지만, 그렇게 되면 vica 내부의 실행 정책과 결과 판정 흐름이 분산될 가능성이 있었습니다. 그래서 역할을 명확히 나눴습니다. 외부 CLI는 작업을 실행하는 쪽에 집중하고, vica는 그 결과를 받아 전체 흐름 안에서 ship 가능 여부를 판단합니다.

이렇게 해두면 앞으로 다른 실행자를 붙이더라도 구조가 덜 흔들립니다. “누가 실행했는가”와 “이 결과를 통과로 볼 것인가”를 분리해두는 편이 장기적으로 유지보수하기 좋다고 판단했습니다.

## 채널 model 명령의 실행자 가드 수정

CLI 상호운용을 정리하다 보니 채널 `model` 명령에서도 경계 문제가 드러났습니다. 해당 명령은 CLI 실행자와 API 기반 실행자가 함께 얽힐 수 있는 지점인데, 여기서 가드 조건이 애매하면 원하지 않는 실행 경로가 선택될 수 있었습니다.

이번에는 `sub` 쪽만 다시 고정하고, API `main` 흐름은 유지하는 방향으로 수정했습니다. 즉, CLI 실행자에 필요한 제한은 걸되 기존 API 기반 메인 실행 흐름은 건드리지 않았습니다.

이 부분은 꽤 조심스러웠습니다. 단순히 막아버리면 당장은 안전해 보이지만, 기존 사용 경로가 깨질 수 있기 때문입니다. 그래서 “어디를 잠가야 하는가”를 실행자 단위로 좁혀서 처리했습니다.

## 소켓/웹 런의 디스코드 미러링 회귀 복구

이번에 가장 체감되는 수정은 소켓/웹에서 시작된 런이 디스코드로 제대로 미러링되지 않던 회귀를 고친 부분입니다. 원인은 READY 시점에서 채널 목적지를 충분히 일찍 해석하지 못한 데 있었습니다.

런이 시작된 뒤에 목적지를 찾으려 하면, 소켓이나 웹 발 런처럼 진입 경로가 다른 경우 미러링 대상이 흔들릴 수 있었습니다. 그래서 READY 단계에서 고정 채널 목적지를 먼저 해석하도록 바꿨습니다. 결과적으로 이후 이벤트들이 어느 채널로 흘러가야 하는지 더 안정적으로 결정됩니다.

개인적으로 이런 종류의 버그는 테스트로 잡기 전까지는 “왜 가끔만 안 가지?”처럼 보이기 쉬워서 까다롭습니다. 이번 수정으로 디스코드 미러링 흐름이 다시 예측 가능해졌습니다.

## 풀 스위트 그린으로 마무리

마지막으로 선행 실패 5건을 정리하고 전체 테스트를 다시 확인했습니다. 현재 풀 스위트는 585개 모두 통과하는 상태입니다.

이번 변경들은 각각 따로 보면 작은 수정처럼 보이지만, 공통적으로는 실행 경로의 책임을 명확히 하는 작업이었습니다. CLI는 실행자, vica는 판정자, 채널 목적지는 READY에서 선해석, 실행자 가드는 필요한 범위에만 적용. 이런 식으로 기준을 세워두니 코드도 테스트도 훨씬 설명하기 쉬워졌습니다.

다음 단계에서는 이 구조 위에서 실행자 확장을 더 자연스럽게 가져갈 수 있을 것 같습니다. 이번에는 화려한 기능 추가보다는, 앞으로 덜 흔들리기 위한 바닥 다지기에 가까운 작업이었습니다.
