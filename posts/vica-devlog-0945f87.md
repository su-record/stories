# 실행 흐름은 더 잘 보이게, 검증은 더 조용하게

이번 작업에서는 `vica`를 실행하면서 느꼈던 답답함과 검증 과정에서 반복되던 불안정성을 함께 정리했습니다. 겉으로 드러나는 기능은 실행 중인 워커를 보여주고 중간 입력을 받을 수 있게 된 것이지만, 내부적으로는 서버 수명 주기와 도구 호출 예산을 다루는 방식까지 손봤습니다.

## 실행 중인 워커를 한눈에 확인하기

여러 워커가 동시에 움직일 때 가장 불편했던 점은 “지금 누가 일하고 있는지”를 바로 알기 어렵다는 것이었습니다. 로그를 거슬러 올라가면 추측할 수는 있었지만, 실행 상태를 파악하기 위해 로그를 해석해야 한다는 것 자체가 좋지 않은 경험이었습니다.

이를 개선하기 위해 화면 하단에 라이브 워커 로스터를 추가했습니다. `cc`의 하단 상태 영역과 비슷한 방식으로, 현재 활동 중인 워커를 실행 흐름을 방해하지 않는 위치에서 확인할 수 있습니다. 덕분에 작업이 실제로 진행 중인지, 어느 워커가 남아 있는지, 병렬 실행이 기대대로 이루어지는지를 훨씬 빠르게 판단할 수 있게 됐습니다.

실행 도중 한 번만 전달할 수 있는 중간 입력도 추가했습니다. 처음 요청만으로는 부족한 맥락이 뒤늦게 떠오르거나, 작업 방향을 짧게 보정하고 싶은 경우가 있었기 때문입니다. 무제한 대화형 제어보다는 범위를 명확히 한 one-shot 방식으로 시작했습니다. 개입 가능성은 열어 두되 실행 흐름이 계속 흔들리지는 않도록 한 선택입니다.

## 검증을 위해 서버를 띄우지 않도록

이번에 가장 깊게 손본 부분은 소스 검증 방식입니다. 기존 흐름에서는 검증 과정에서 서버가 함께 올라가는 경우가 있었고, 기본 포트인 `3000`이 이미 사용 중이면 검증과 무관한 충돌이 발생했습니다. 더 까다로운 문제는 편집이 끝난 뒤에도 이전 검증 서버가 남아, 방금 수정한 코드가 아니라 오래된 코드를 계속 제공하는 현상이었습니다.

처음에는 서버 종료 처리를 보강하는 식으로 접근했지만, 비슷한 문제가 다시 나타났습니다. 결국 “저레벨 소스 검증은 서버를 띄우지 않는다”는 원칙으로 구조를 정리했습니다. 서버가 꼭 필요한 경로에는 빈 포트를 주입해 고정 포트 경쟁도 제거했습니다.

이 변경으로 stale 서버를 뒤쫓아 종료하는 두더지잡기에서 벗어났습니다. 검증은 파일과 정적 결과를 확인하고, 실제 서빙이 필요한 단계만 별도의 수명 주기를 갖도록 역할이 분명해졌습니다. 단순한 누수 수정이라기보다 검증의 책임 범위를 다시 정한 작업에 가깝습니다.

## 반복 실패로 예산을 잃지 않게

`web-fetch`가 실패 가능성이 높은 호스트를 계속 호출하며 실행 예산을 소진하는 문제도 막았습니다. 호스트 단위 회로차단을 적용해 반복 실패를 감지하면 같은 대상에 대한 무의미한 재시도를 줄이도록 했습니다. 네트워크 문제를 완전히 없앨 수는 없지만, 실패가 전체 작업을 끌고 가는 상황은 피할 수 있게 됐습니다.

또한 워커의 읽기 전용 명령에서는 세미콜론으로 연결된 입력을 허용했습니다. 여러 조회 명령을 안전하게 묶어 실행하려는 경우까지 차단되어 불필요하게 우회하던 부분을 완화한 것입니다. 쓰기 작업의 경계는 유지하면서도, 조사 단계의 생산성은 놓치지 않도록 균형을 맞췄습니다.

이번 작업은 새 기능을 더한 것만큼이나, 실행 중 보이지 않던 상태와 검증 뒤에 남던 흔적을 줄이는 데 의미가 있었습니다. 앞으로도 문제가 생길 때 개별 증상만 막기보다, 해당 단계가 정말 그 책임을 가져야 하는지부터 살펴보겠습니다.
