## 의도 분류를 다시 모델 중심으로 정리했습니다

이번 작업의 핵심은 `vica`의 요청 분류 흐름을 조금 더 믿을 만하게 만드는 일이었습니다. 이전에는 사용자의 입력이 어떤 의도인지 판단할 때 정규식 기반의 동사 목록이 꽤 앞단에서 역할을 맡고 있었습니다. “만들어줘”, “작성해줘”, “구성해줘” 같은 표현을 보면 창작이나 구축 요청으로 보는 식이었죠.

처음에는 단순하고 빠르다는 장점이 있었지만, 실제 사용 문장은 생각보다 훨씬 애매했습니다. 같은 동사라도 맥락에 따라 답변형 요청일 수도 있고, 어떤 경우에는 작업 지시일 수도 있었습니다. 그래서 이번에는 의도 분류의 주도권을 모델에게 넘기고, 정규식 동사 목록은 오프라인 폴백으로 한 단계 물러나게 했습니다.

즉, 정상적인 경로에서는 모델이 문맥을 보고 판단하고, 모델을 사용할 수 없는 상황에서만 기존 규칙이 최소한의 안전망으로 동작합니다. 규칙을 완전히 버리지는 않았지만, “가장 먼저 판단하는 기준”에서 “마지막에 기대는 백업”으로 위치를 바꾼 셈입니다. 개인적으로는 이 방향이 장기적으로 더 자연스럽다고 느꼈습니다. 대화형 서비스에서 의도는 단어 하나보다 문장 전체의 분위기와 목적에 더 많이 기대니까요.

## 창작·구축 요청이 일반 답변으로 새던 구멍을 막았습니다

분류기를 모델 중심으로 바꾸는 과정에서, 특히 신경 쓴 부분은 창작·구축 요청이 `answer`로 잘못 흘러가는 문제였습니다. 사용자가 무언가를 “써달라”, “만들어달라”, “구성해달라”고 했는데 시스템이 이를 단순 질의응답으로 처리하면 이후 단계의 응답 품질이 크게 흔들립니다.

예를 들어 블로그 글 작성, 문서 초안 생성, 기능 설계, 코드 구조 제안 같은 요청은 단순히 정보를 설명하는 것과는 다르게 다뤄야 합니다. 그런데 분류가 `answer`로 떨어지면 결과물이 지나치게 설명형이 되거나, 사용자가 기대한 산출물 형태를 놓치는 경우가 생깁니다.

이번 수정에서는 그런 경계 사례들을 다시 확인하고, 창작과 구축 성격의 요청이 올바른 흐름으로 들어가도록 보강했습니다. 거창한 기능 추가라기보다는, 실제 사용 중에 은근히 불편함을 만들던 틈을 메운 작업에 가깝습니다. 이런 부분은 티가 크게 나지는 않지만, 사용자가 “왜 이렇게 답하지?”라고 느끼는 순간을 줄여준다는 점에서 꽤 중요합니다.

## 모델 라인업은 실측 기준으로 다시 맞췄습니다

마지막으로 모델 설정도 정리했습니다. 한동안 제외했던 `gpt-5.3-codex-spark`를 다시 복원했습니다. 이유는 단순합니다. 실제로 아직 서빙되고 있었기 때문입니다. 반대로 `5.3-codex`는 더 이상 살아있지 않은 것으로 확인되어, 현재 상태에 맞게 판단했습니다.

모델 이름이나 제공 상태는 문서만 보고 있으면 애매할 때가 많습니다. 특히 내부적으로는 비슷한 이름이 이어지거나, 특정 변형만 살아남는 경우도 있어서 결국 실측이 제일 확실합니다. 이번에도 “그럴 것 같다”가 아니라 직접 확인한 결과를 기준으로 정리했습니다.

이 작업은 코드상으로는 작은 설정 변경처럼 보이지만, 실제 운영에서는 꽤 중요합니다. 죽은 모델을 후보에 남겨두면 실패 경로가 늘어나고, 반대로 살아있는 모델을 빼두면 선택지가 줄어듭니다. 그래서 모델 라인업은 가능한 한 현실과 가깝게 유지하려고 합니다.

## 마무리

이번 변경들은 모두 분류와 모델 선택이라는, 사용자의 눈에는 잘 보이지 않는 층에 집중되어 있습니다. 하지만 이런 층이 흔들리면 최종 응답은 바로 어색해집니다. 반대로 기반이 안정되면 같은 프롬프트라도 훨씬 덜 새고, 의도에 맞는 결과로 이어집니다.

정리하자면, 이번에는 규칙보다 문맥을 더 신뢰하도록 분류 구조를 조정했고, 창작·구축 요청의 누수를 막았으며, 실제 서빙 상태에 맞춰 모델 설정을 갱신했습니다. 작지만 서비스의 감각을 조금 더 자연스럽게 만드는 업데이트였습니다.
