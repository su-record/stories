## 같은 설명을 세 번 하는 순간, AI는 도구에 머뭅니다

![모델의 기억과 사용자가 관리하는 메모리는 다르다](/images/ai-with-lasting-memory/01.png)
*기억의층*


평서체로 써라. H2부터 시작해라. 이모지는 쓰지 마라.

AI에게 글을 맡길 때 제가 가장 자주 붙인 문장들입니다. 한두 번은 괜찮지만, 매일 반복하면 낭비입니다.

문제는 프롬프트 길이만이 아닙니다. AI가 매번 처음 만난 사람처럼 굴면 협업은 늘 온보딩부터 다시 시작합니다.

OpenAI는 ChatGPT Memory를 두 층으로 나눠 설명합니다. 사용자가 직접 저장한 `saved memories`가 있고, 과거 대화를 참고하는 개인화가 있습니다.

Google Gemini도 past chats를 이용해 답변을 맞출 수 있다고 설명합니다. Perplexity는 선호, 관심사, 공유 정보, 검색 기록 기반 Memory와 관리 기능을 제공합니다.

메모리는 답변을 부드럽게 다듬는 기능이 아닙니다. 다시 설명하는 비용을 줄이는 인터페이스입니다.

## 모델의 기억과 나의 기억은 다릅니다

![반복 규칙은 메모리, 일회성 자료는 컨텍스트다](/images/ai-with-lasting-memory/02.png)
*저장선별*


LLM에는 이미 많은 기억이 들어 있습니다. 위키피디아, 뉴스, 웹 문서, 서적, 코드 같은 비정형 텍스트를 사전 학습합니다.

그 과정에서 언어 구조와 세계 지식이 수십억에서 수조 개 매개변수 안에 압축됩니다. 이것이 모델의 기본 기억입니다.

3Blue1Brown의 LLM 설명 영상이 680만 회 이상 조회된 이유도 여기에 있습니다. 사람들은 AI가 어떻게 다음 단어를 맞히는지 알고 싶어 했습니다.

다만 그 기억은 나를 기억하는 방식이 아닙니다. 내가 평서체를 선호하는지, 회사 문서에서 어떤 용어를 금지하는지는 기본 모델이 알 수 없습니다.

| 구분 | 저장 위치 | 예시 | 내가 제어할 수 있는가 |
|---|---|---|---|
| 파라미터 기억 | 모델 가중치 | 문법, 일반 지식, 코드 패턴 | 거의 불가 |
| 대화 맥락 | 현재 세션 | 방금 붙여 넣은 요구사항 | 세션 종료 시 약함 |
| 제품 메모리 | 서비스 계층 | 선호 문체, 업무 환경, 금지어 | 보기·수정·삭제 가능 |
| 조직 메모리 | 워크스페이스 | 팀 규칙, 보안 정책, 템플릿 | 관리자 정책에 따름 |

반복 설명을 줄여 주는 건 세 번째 줄입니다. 사용자가 관리할 수 있는 제품 메모리입니다.

AI가 더 똑똑해지는 일과 나에게 맞춰지는 일은 다릅니다. 메모리는 후자에 가깝습니다.

## 메모리에 넣을 것과 넣지 말아야 할 것

![메모리는 많을수록 좋은 것이 아니라 통제와 삭제가 필요하다](/images/ai-with-lasting-memory/03.png)
*삭제통제*


좋은 메모리는 짧고 반복해서 쓸 수 있어야 합니다. 매번 바뀌는 사실은 메모리가 아니라 컨텍스트입니다.

이번 글의 자료, 오늘 회의록, 특정 고객사의 견적은 대화에 넣어야 합니다. 반대로 매번 같은 규칙은 저장해 둘 만합니다.

```yaml
assistant_memory:
  writing:
    tone: 평서체
    headings: H2부터 시작
    banned:
      - 이모지
      - 과한 수식어
      - 억지 마무리 문장
  dev_env:
    os: macOS
    editor: VS Code
    package_manager: pnpm
  work_rules:
    prefer_table: true
    explain_with_numbers: true
```

이 정도만 저장해도 첫 프롬프트가 짧아집니다. 예전에는 ‘내 문체 규칙’을 8줄씩 붙였습니다.

지금은 ‘이 주제로 초안 작성’만으로 시작할 수 있습니다. 남는 시간은 결과물 검수에 씁니다.

OpenAI FAQ도 한계를 분명히 둡니다. 정확한 템플릿이나 긴 원문을 그대로 보존하는 용도로 Memory를 쓰면 안 됩니다.

그런 내용은 파일, 노션, 레포지토리, 시스템 프롬프트에 두는 편이 맞습니다. 메모리는 데이터베이스가 아닙니다.

| 저장하면 좋은 것 | 저장하면 안 되는 것 |
|---|---|
| 평서체 선호 | 계약서 원문 전체 |
| Python보다 TypeScript 선호 | 일회성 API 키 |
| 블로그는 H2부터 시작 | 고객 개인정보 |
| 표로 비교하는 설명 선호 | 매번 바뀌는 실험 결과 |
| 회사 용어집의 핵심 금지어 | 법적 효력이 필요한 문구 |

틀리기 쉬운 부분이 여기입니다. 메모리는 내가 반복해서 알려 준 경향을 담는 얇은 층입니다.

## 에이전트 시대에는 기억보다 통제가 먼저입니다

빅테크가 단순 채팅 UI를 넘어 AI 에이전트에 집중하는 이유가 있습니다. 에이전트는 답만 생성하지 않습니다.

목표를 받고, 계획을 세우고, 도구를 쓰고, 중간 결과를 확인합니다. 이 과정에서 메모리가 빠지면 금방 삐걱거립니다.

| 작업 | 메모리 없음 | 메모리 있음 |
|---|---|---|
| 초안 작성 | 문체 규칙을 매번 입력 | 저장된 톤으로 바로 작성 |
| 코드 예시 | 언어 선호를 매번 질문 | TypeScript 우선 제안 |
| 배포 | 레포 구조를 매번 확인 | `~/stories` 기준으로 안내 |
| 검수 | 금지 표현을 놓침 | 금지어 목록으로 재검토 |

AI 에이전트가 업무 대리인에 가까워질수록 메모리는 부가 기능이 아닙니다. 작업의 연속성을 만드는 최소 조건입니다.

물리적 메모리도 이 흐름과 연결됩니다. 엔비디아 중심의 AI 인프라 투자는 HBM과 선단공정 수요를 밀어 올렸습니다.

2026년 7월 기준 HBM 장기공급계약은 메모리 가격 하락을 방어하는 근거로 언급됩니다. 서비스의 기억 아래에는 추론 인프라가 있습니다.

사용자가 체감하는 메모리는 HBM 용량이 아닙니다. ‘내가 지난번에 뭐라고 했는지 다시 묻지 않는가’에 가깝습니다.

그래서 먼저 볼 것은 통제입니다. OpenAI, Gemini, Perplexity 모두 보기, 수정, 삭제, 비활성화 같은 관리 기능을 설명합니다.

| 질문 | 판단 |
|---|---|
| 다음 달에도 반복될 규칙인가 | 메모리에 저장 |
| 틀리면 보안 문제가 되는가 | 저장하지 않음 |
| 원문 그대로 재현해야 하는가 | 별도 문서에 저장 |
| 내가 직접 삭제할 수 있는가 | 사용 가능 |
| 팀 전체에 적용되는가 | 개인 메모리보다 워크스페이스 규칙 |

AI 메모리의 가치는 거창하지 않습니다. 같은 말을 덜 하게 하고, 첫 응답부터 내 환경을 반영하게 합니다.

그렇다고 기억은 많을수록 좋은 기능이 아닙니다. 오래된 환경 정보는 계속 답변을 오염시킵니다.

`npm`을 쓰던 시절의 기억이 남아 있으면, `pnpm` 프로젝트에서 계속 틀립니다. 저장보다 삭제가 더 필요할 때도 있습니다.
