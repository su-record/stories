### 1. 🛠️ CI/CD 및 Flutter 환경 안정화 (6개 커밋)

이번 기간 작업의 60% 이상을 차지한 영역입니다. Flutter는 빠르게 발전하고 있어서 CI 환경을 제때 업데이트하지 않으면 알 수 없는 빌드 오류를 만나기 쉽습니다. 특히 웹(Web) 빌드를 염두에 두고 있기 때문에 환경 의존성을 낮추는 작업이 필수였습니다.

| 커밋 메시지 (요약) | 상세 설명 | 영향 영역 |
| :--- | :--- | :--- |
| `ci: Flutter 버전 고정 제거, stable 채널 자동 사용` | 특정 버전을 고정하지 않고, 안정적인 `stable` 채널의 최신 버전을 자동으로 사용하도록 변경하여 관리 부담을 줄였습니다. | CI/CD 안정성 |
| `ci: Flutter 3.29.0 / Dart 3.7.0으로 업그레이드` | 기존 3.27.0에서 한 단계 더 업그레이드하여 최신 API를 적용했습니다. | 개발 환경 |
| `fix: Flutter 3.38+ 호환 - --web-renderer 옵션 제거` | Flutter 버전이 3.38 이상으로 올라갈 경우, 웹 렌더러를 수동으로 지정하던 `--web-renderer html` 옵션이 충돌을 일으키는 것을 확인하고 해당 옵션을 삭제했습니다. 렌더링 방식이 자동 최적화되기를 기대하고 있습니다. | 웹 호환성 |

**배운 점:** 버전을 고정하는 것이 개발 환경에서는 편할 수 있지만, 장기 프로젝트에서는 특정 버전을 벗어났을 때 호환성 문제에 더 취약해질 수 있습니다. CI/CD에서 `stable` 채널을 사용하되, 주요 업그레이드 시점마다 빌드 테스트를 명확히 하는 전략으로 전환했습니다.

## 💡 작업 하이라이트: 반복된 Flutter CI/CD 작업의 이유

이번 기간의 하이라이트는 단연 CI/CD 환경 안정화입니다.

현재 Fallingo는 프론트엔드에 `Flutter`를 사용하고 있고, 웹/모바일 동시 빌드를 목표로 합니다. GitHub Actions를 이용해 매 커밋마다 자동 빌드 및 배포 테스트를 진행하고 있는데요.

최근 Flutter가 웹 렌더링 엔진 관련하여 대규모 업데이트를 진행하면서, 기존에 명시적으로 넣어두었던 `--web-renderer` 옵션이 최신 버전과 충돌하기 시작했습니다. 이 때문에 빌드가 실패했고, 이를 해결하기 위해 CI 파일에서 버전을 올리고, 기존 플래그를 삭제하는 등의 조치가 연달아 필요했습니다.

이런 사소한 환경 설정 오류 때문에 시간을 낭비하지 않도록, `stable` 채널 자동 사용 및 GitHub Actions 캐시 업그레이드(v3 -> v4)를 동시에 진행하여 CI 환경의 **보수성**을 높이는 데 집중했습니다.

현재 Fallingo는 MVP(Minimum Viable Product) 단계로 넘어가는 초입에 있습니다.

| 영역 | 진행도 (추정) | 비고 |
| :--- | :--- | :--- |
| **백엔드 (FastAPI/PostgreSQL)** | 60% | 핵심 AI 추천 로직 설계 및 데이터 모델링 진행 중 |
| **프론트엔드 (Flutter)** | 40% | 로그인 및 기본 UI 컴포넌트 개발 완료. 위치 기반 피드 UI 구현 중 |
| **인프라/CI/CD** | 85% | 기본적인 자동화 환경은 구축 완료. 부하 테스트 인프라 보강 예정 |
| **문서화/기획** | 75% | 핵심 가치 정의 완료. 세부 기능 UI/UX 문서화 진행 필요 |

개발 과정에서 항상 느끼는 거지만, 새로운 기술(FastAPI, Flutter)을 배우면서 빠르게 구현하는 것은 AI의 도움 덕분에 가능하지만, **운영 환경과 지속 가능한 코드 베이스**를 만드는 것은 결국 개발자의 몫인 것 같습니다. 특히 CI/CD와 문서화는 눈에 띄지 않지만 프로젝트의 수명을 늘리는 중요한 작업이라고 생각합니다.
