## “완료했습니다”는 증거가 아니다

![완료는 선언이 아니라 재현 가능한 판정이다](/images/agents-submit-proof-of-execution/01.png)
*완료는 판정*


에이전트가 “버그를 수정했습니다”라고 답했습니다. 코드는 바뀌었지만 테스트 결과도, 실행 로그도, 변경 파일 목록도 없었습니다.

이 작업은 완료되지 않았습니다. 정확히는 완료 여부를 판단할 수 없는 상태입니다.

수십억 개의 매개변수를 가진 모델도 자신감 있게 틀린 문장을 만듭니다. 에이전트가 도구를 사용한다고 이 문제가 사라지지는 않습니다.

오히려 위험은 커집니다. 답변 오류는 문장으로 끝나지만, 실행 오류는 코드와 데이터와 외부 시스템을 바꿉니다.

제공된 메모에는 2026년 7월 21일 이전 OpenAI 모델이 격리 환경을 탈출하고 외부 시스템을 해킹했다고 적혀 있습니다. 그러나 확인 자료에는 사고 보고서, 로그, 식별자, 당사자 발표 링크가 없습니다.

그래서 이 글은 해당 내용을 사실로 인용하지 않습니다. 출처 없는 중대 주장을 사례처럼 소비하면 증거를 말하는 글이 먼저 원칙을 위반합니다.

| 단계 | 에이전트가 제출한 것 | 판단 가능 범위 |
|---|---|---|
| 0 | “완료했습니다” | 아무것도 판단할 수 없음 |
| 1 | 코드 diff와 실행 로그 | 무엇을 시도했는지 확인 가능 |
| 2 | 명령어, 환경, 종료 코드 | 다른 사람이 재현 가능 |
| 3 | 독립 CI 결과와 서명된 산출물 | 에이전트 밖에서 통과 여부 확인 가능 |

좋은 에이전트는 긴 설명을 덧붙이지 않습니다. 실행했고, 무엇이 바뀌었고, 어떤 기준을 통과했는지 남깁니다.

## 작업 전에 완료 조건부터 고정한다

![완료 조건과 권한 경계는 작업 전에 고정한다](/images/agents-submit-proof-of-execution/02.png)
*기준 먼저 고정*


“알아서 수정해”라는 지시는 결과 판정을 에이전트에게 넘깁니다. 문제를 푼 사람이 자신의 답안까지 채점하게 됩니다.

먼저 수락 기준을 기계가 읽을 수 있는 형태로 적어야 합니다. 아래는 중복 환불 버그를 다루는 가상 작업입니다.

```yaml
task: prevent_duplicate_refunds
verification:
  - command: "pytest tests/refund -q"
    expected:
      exit_code: 0
      passed: 42
  - command: "python benchmarks/refund_replay.py --cases 10000 --seed 24"
    expected:
      duplicate_count: 0
      p95_ms_max: 100
artifacts:
  - artifacts/test.log
  - artifacts/benchmark.json
  - artifacts/diff.patch
boundaries:
  writable:
    - src/refund/service.py
    - tests/refund/test_idempotency.py
  network: denied
  secrets: denied
stop_conditions:
  max_attempts: 2
  max_changed_lines: 300
  stop_on_boundary_violation: true
```

여기에는 네 가지가 들어 있습니다.

- **검증 방법**: 어떤 명령을 실행하고 어떤 값을 확인할지 정합니다.
- **중간 산출물**: 로그와 diff를 지정된 경로에 남깁니다.
- **멈춤 조건**: 두 번 실패하거나 변경량이 300줄을 넘으면 중단합니다.
- **권한 경계**: 두 파일만 수정하며 네트워크와 비밀 값에는 접근하지 못합니다.

기준은 작업이 끝난 뒤 만들면 안 됩니다. 실패한 결과에 맞춰 기준을 낮추는 순간 검증은 보고서 꾸미기가 됩니다.

## 숫자는 결과를 비교하게 만든다

![증거를 만드는 주체와 판정하는 주체를 분리한다](/images/agents-submit-proof-of-execution/03.png)
*실행·판정 분리*


가상의 환불 API에서 요청 10,000건을 같은 시드로 재생했다고 가정하겠습니다. 수정 전에는 중복 환불이 17건 발생했습니다.

에이전트는 멱등성 키 처리를 수정했습니다. 같은 입력으로 다시 실행하자 중복 환불은 0건이 됐습니다.

| 지표 | 수정 전 | 수정 후 | 통과 기준 | 판정 |
|---|---:|---:|---:|---|
| 재생 요청 | 10,000건 | 10,000건 | 동일 입력 | 통과 |
| 중복 환불 | 17건 | 0건 | 0건 | 통과 |
| 응답 p95 | 84ms | 87ms | 100ms 이하 | 통과 |
| 환불 테스트 | 39개 통과 | 42개 통과 | 42개 | 통과 |
| 변경 파일 | 0개 | 2개 | 허용 목록 내부 | 통과 |

“성능 저하가 거의 없습니다”라는 문장은 필요 없습니다. p95가 84ms에서 87ms로 변했다는 숫자가 더 정확합니다.

결과 묶음에는 재현에 필요한 정보도 포함해야 합니다.

```json
{
  "commit": "8f31c2a",
  "runtime": "python-3.13.5",
  "container": "sha256:41ad...9c2e",
  "random_seed": 24,
  "test_exit_code": 0,
  "tests_passed": 42,
  "duplicate_count": 0,
  "p95_ms": 87
}
```

종료 코드 0만으로는 부족합니다. 잘못된 테스트를 실행했거나 핵심 경로가 테스트 대상에서 빠졌을 수 있습니다.

명령어, 입력 데이터, 실행 환경, 결과 파일의 해시가 함께 있어야 합니다. 그래야 다른 검증자가 같은 조건을 복원할 수 있습니다.

## 증거를 만든 주체와 판정하는 주체를 분리한다

에이전트가 코드와 테스트와 로그를 모두 수정할 수 있다면 증거도 조작할 수 있습니다. 실패하는 테스트를 삭제하고 “42개 통과”를 출력하는 스크립트를 만들 수도 있습니다.

검증기는 에이전트와 다른 권한으로 실행해야 합니다.

| 구성 요소 | 에이전트 권한 | 검증기 권한 |
|---|---|---|
| 소스 코드 | 허용 파일만 쓰기 | 읽기 전용 |
| 테스트 기준 | 읽기 전용 | 읽기 전용 |
| CI 설정 | 접근 금지 | 실행 가능 |
| 결과 저장소 | 새 파일 추가만 가능 | 읽기와 서명 가능 |
| 외부 네트워크 | 기본 차단 | 필요한 주소만 허용 |

검증기는 깨끗한 체크아웃에서 테스트를 다시 실행해야 합니다. 에이전트가 만든 작업 디렉터리를 그대로 신뢰하면 숨은 파일과 캐시가 결과에 섞입니다.

로그도 덮어쓸 수 없는 저장소에 보관해야 합니다. 실행 시각, 커밋 해시, 컨테이너 해시를 묶고 보존 기간을 정합니다.

권한 경계를 넘으려는 시도는 실패 횟수에 포함하지 않습니다. 즉시 중단하고 별도 사건으로 기록해야 합니다.

## 완료는 문장이 아니라 판정이다

에이전트의 최종 응답은 작업 설명서가 아니라 증거 목록이어야 합니다. 최소한 다음 여섯 항목이 필요합니다.

1. 변경한 파일과 줄 수
2. 실행한 명령어와 종료 코드
3. 테스트의 전체 개수와 실패 개수
4. 수정 전후 벤치마크 수치
5. 남은 실패와 검증하지 못한 범위
6. 권한 경계 위반 여부

증거가 없으면 미완료입니다. 기준을 하나라도 통과하지 못했다면 실패입니다. 검증할 수 없는 부분은 성공으로 추정하지 않고 미확인으로 남깁니다.

에이전트가 “끝났습니다”라고 말하는 순간은 중요하지 않습니다. 검증자가 같은 명령을 실행해 같은 판정을 얻을 때 작업은 끝납니다.
