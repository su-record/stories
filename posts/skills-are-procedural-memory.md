## 한 번 가르쳤는데 매번 다시 설명하고 있었습니다

![스킬은 사실 저장소가 아니라 실행 순서다](/images/skills-are-procedural-memory/01.png)
*절차는스킬*


모델 성능 문제가 아니었습니다. 같은 설명을 제가 매 세션 반복하고 있었습니다.

PR 리뷰 기준, 블로그 문체, 장애 리포트 형식, 고객 응대 톤을 매번 붙였습니다. 900자짜리 규칙이 8개만 돼도, 시작하기 전에 컨텍스트를 7,000자 넘게 씁니다.

LLM은 이전 토큰을 보고 다음 토큰을 예측합니다. 3Blue1Brown의 LLM 설명 영상이 680만 회 이상 조회된 이유도 여기에 있습니다. 복잡한 모델을 결국 다음 단어 예측으로 보여줬기 때문입니다.

실제 업무에서 더 자주 걸리는 지점은 예측 능력이 아니었습니다. 기억을 어디에 둘 것인가였습니다.

모델의 매개변수는 사전 학습에서 만들어집니다. 위키피디아, 뉴스, 웹 문서, 책, 코드가 들어갑니다. 이 지식은 거의 고정되어 있습니다. 내가 오늘 정한 리뷰 절차는 매개변수 안에 없습니다.

그래서 매번 프롬프트에 붙였습니다. 방식이 잘못됐습니다. 절차는 프롬프트가 아니라 스킬로 빼야 했습니다.

---

## 스킬은 지식이 아니라 실행 순서입니다

![좋은 스킬은 트리거·단계·출력물이 선명하다](/images/skills-are-procedural-memory/02.png)
*신호-순서-결과*


스킬은 사실을 저장하는 곳이 아닙니다. 절차 기억입니다. 사람으로 치면 자전거 타는 법에 가깝습니다.

서울의 인구, HTTP 429의 의미, 특정 라이브러리 문법은 선언적 지식입니다. 장애 리포트를 쓰는 순서, PR을 리뷰하는 순서, 고객 메일을 정리하는 순서는 절차입니다.

| 구분 | 저장 대상 | 실패 방식 | 적합한 위치 |
|---|---:|---|---|
| 지식 | 사실, 정의, 링크 | 오래되거나 틀림 | 문서, 검색, RAG |
| 취향 | 말투, 금지어, 형식 | 일관성 깨짐 | 프로필, 메모리 |
| 절차 | 단계, 체크리스트, 판단 기준 | 매번 다시 설명 | 스킬 |
| 작업 상태 | 오늘 한 일, 남은 일 | 세션 종료로 유실 | 세션 로그 |

Embody changelog에도 비슷한 방향이 보입니다. 상세 guidance를 on-demand skills로 옮겨 세션당 token overhead를 줄인다고 설명했습니다. 긴 설명을 늘 들고 다니는 대신, 필요할 때만 부르는 방식입니다.

스킬은 세션 시작 프롬프트를 줄입니다. 대신 호출 가능한 절차를 늘립니다. 다음 세션은 처음부터 모든 규칙을 읽지 않습니다. 작업 신호가 보이면 관련 스킬만 꺼냅니다.

## 좋은 스킬은 트리거와 출력물이 선명합니다

![모든 규칙을 들고 다니지 말고 필요할 때만 호출한다](/images/skills-are-procedural-memory/03.png)
*필요할때만*


스킬을 만들 때 자주 망하는 지점은 설명을 길게 쓰는 것입니다. 설명이 길면 다시 프롬프트 덩어리가 됩니다.

좋은 스킬에는 세 가지가 있습니다.

1. 언제 호출할지 알 수 있는 트리거
2. 어떤 순서로 처리할지 보이는 단계
3. 마지막에 어떤 산출물을 낼지 정해진 형식

예시는 이 정도로 짧아야 합니다.

```yaml
name: korean-pr-review
description: 한국어 서비스 코드 PR을 리뷰할 때 사용합니다.
triggers:
  - PR 리뷰
  - diff 검토
  - 코드 변경점 확인
steps:
  - 변경 목적을 한 문장으로 요약합니다.
  - 버그 가능성을 먼저 봅니다.
  - 성능, 보안, 유지보수 순서로 봅니다.
  - 칭찬은 구체적 코드 줄에만 붙입니다.
  - 수정 요청은 재현 조건과 대안을 함께 씁니다.
output:
  - 요약
  - 반드시 고쳐야 할 것
  - 선택적으로 고칠 것
  - 질문
rules:
  - 추측이면 추측이라고 씁니다.
  - 근거 없는 칭찬을 쓰지 않습니다.
```

이 정도면 다음 세션에서 바로 꺼낼 수 있습니다. 중요한 건 모델에게 성격을 주입하지 않는 것입니다. 역할극은 쉽게 흔들립니다. 절차는 덜 흔들립니다.

나쁜 스킬은 보통 이렇게 생겼습니다.

```text
너는 뛰어난 시니어 개발자입니다. 항상 꼼꼼하고 친절하며 비즈니스 맥락을 고려합니다.
좋은 리뷰를 해주세요. 보안도 보고 성능도 보고 가독성도 봐주세요.
```

기준이 없습니다. 좋은 리뷰가 무엇인지 매번 달라집니다. 다음 세션이 꺼내 쓸 수 있는 것은 칭찬 문장이 아니라 판단 순서입니다.

## 컨텍스트 비용은 조용히 누적됩니다

긴 시스템 프롬프트는 처음에는 편해 보입니다. 모든 규칙을 한곳에 넣으면 빠뜨릴 일이 없어 보입니다.

문제는 그다음입니다. 새 작업을 시작할 때마다 같은 규칙을 다시 읽힙니다. 쓰지 않는 규칙도 비용을 씁니다. 모델은 블로그 작성 규칙을 읽은 뒤 코드 리뷰를 하고, 고객 응대 규칙을 읽은 뒤 SQL을 봅니다.

간단히 계산해보면 차이가 보입니다.

| 방식 | 세션 시작 시 고정 토큰 | 실제 사용 규칙 | 낭비 |
|---|---:|---:|---:|
| 전체 프롬프트 | 8,000 | 1,200 | 6,800 |
| 스킬 호출 | 900 | 1,200 | 0에 가까움 |
| 검색 문서 | 400 | 상황별 | 검색 품질에 의존 |

2026년 Claude Workflow Library에도 장기 세션, persistent memory, context management, memory-system evaluation 관련 항목이 계속 추가되고 있습니다. 이유는 명확합니다. 긴 대화에서 성능이 떨어지는 원인은 지능 부족보다 컨텍스트 관리 실패인 경우가 많습니다.

스킬은 이 문제를 직접 줄입니다. 모든 것을 기억하라고 하지 않습니다. 필요한 절차를 찾게 만듭니다.

OpenClaw 같은 모듈형 구조도 같은 힌트를 줍니다. 플러그인과 확장을 필요한 만큼 설치합니다. 전체 기능을 항상 켜두지 않습니다. 스킬도 같은 방식으로 설계해야 합니다.

## 한 번 가르친다는 말의 정확한 의미

한 번 가르친다는 말은 모델이 영구히 똑똑해진다는 뜻이 아닙니다. 매개변수가 바뀌는 것도 아닙니다.

정확히는 이렇습니다.

절차를 파일이나 라이브러리로 분리합니다. 호출 조건을 붙입니다. 다음 세션에서 작업 신호가 나오면 해당 절차를 로드합니다. 그래서 다시 설명하지 않습니다.

이 구조가 생기면 프롬프트 작성 방식이 바뀝니다.

```text
이 PR 리뷰해줘. korean-pr-review 기준으로 봐줘.
```

이 한 줄이면 충분해야 합니다. 충분하지 않다면 스킬이 아직 절차가 아니라 설명문이라는 뜻입니다.

스킬은 프롬프트를 잘 쓰는 기술이 아닙니다. 반복되는 일을 절차로 접어두는 기술입니다. 세션이 바뀌어도 같은 순서로 꺼내 쓰게 하는 장치입니다.
