## vica 개발일지: 자가진단 흐름과 스킬 노출 정책 정리

이번 작업에서는 vica를 실제로 쓰는 과정에서 “문제가 생겼을 때 스스로 확인할 수 있는가”, 그리고 “사용자에게 보여야 할 것만 깔끔하게 보여주는가”에 초점을 맞췄습니다. 겉으로는 작은 변경처럼 보일 수 있지만, CLI 도구를 오래 믿고 쓰기 위해서는 이런 기본기가 꽤 중요하다고 느꼈습니다.

### `/doctor` 자가진단 기능 추가

먼저 `/doctor` 명령을 통해 현재 실행 환경을 점검할 수 있도록 했습니다. vica를 쓰다 보면 설정 경로나 런타임 상태, 세션 관련 정보처럼 사용자가 바로 눈치채기 어려운 부분에서 문제가 생길 수 있습니다. 그럴 때 무작정 로그를 뒤지기보다는, 도구 안에서 “지금 상태가 괜찮은지”를 확인할 수 있는 진단入口가 필요했습니다.

이번에 추가한 `/doctor`는 그런 용도에 맞춰 기본적인 자가진단 역할을 하도록 만들었습니다. 특히 종료 시 세션 ID를 안내하도록 해서, 사용자가 문제를 공유하거나 나중에 특정 실행 흐름을 되짚을 때 기준점으로 삼을 수 있게 했습니다. CLI에서 세션 ID 같은 정보는 평소에는 크게 신경 쓰이지 않지만, 디버깅이나 문의 상황에서는 꽤 든든한 단서가 됩니다.

개발하면서는 “정보를 얼마나 보여줄 것인가”를 고민했습니다. 너무 적게 보여주면 진단 기능의 의미가 약해지고, 너무 많이 보여주면 사용자에게 부담이 됩니다. 그래서 지금은 문제를 설명하고 추적하는 데 도움이 되는 방향으로 균형을 잡는 데 집중했습니다.

### 스킬 목록의 표면 정리

두 번째 큰 흐름은 스킬이 사용자에게 어떻게 보이는지를 정리한 부분입니다. vica에는 시스템 내부에서 사용하는 내장 스킬과 사용자가 직접 정의한 스킬이 함께 존재할 수 있습니다. 이전 구조에서는 내부 구현에 가까운 시스템 내장 스킬이 노출될 여지가 있었고, 이 경우 사용자는 “이걸 내가 써야 하나?” 하고 헷갈릴 수 있었습니다.

이번에는 시스템 내장 스킬을 완전히 비노출하도록 정리했습니다. 사용자가 다룰 수 있는 표면에는 사용자가 의미 있게 선택하고 관리할 수 있는 항목만 남기는 방향입니다. 내부 기능은 내부에 머물고, 사용자 인터페이스에는 사용자 관점의 정보만 드러나는 것이 더 자연스럽다고 판단했습니다.

또한 같은 이름의 스킬이 있을 때는 사용자 정의 스킬을 우선하도록 했습니다. 이 결정도 중요했습니다. 사용자가 명시적으로 만든 것은 그 환경에서 가장 강한 의도를 가진 설정이라고 볼 수 있기 때문입니다. 기본값이나 내장 동작보다 사용자의 선택을 존중하는 쪽이 CLI 도구답다고 생각했습니다.

### 설정 경로 변경에 맞춘 테스트 보강

마지막으로 테스트 쪽에서는 설정 홈 상수 캡처 로직을 정리했습니다. `VICA_CONFIG_DIR` 리네임이 반영되면서, 기존 테스트가 예전 이름이나 흐름에 기대고 있던 부분을 맞춰야 했습니다. 기능 변경은 아니지만, 이런 부분을 방치하면 나중에 실제 동작보다 테스트가 뒤처지거나 반대로 잘못된 안정감을 줄 수 있습니다.

특히 설정 디렉터리처럼 환경 변수와 연결되는 영역은 작은 이름 변경도 영향 범위가 은근히 큽니다. 그래서 테스트가 현재의 명명과 동작을 정확히 바라보도록 갱신했습니다. 이런 정리는 티가 많이 나지는 않지만, 이후 리팩터링이나 기능 추가 때 마음 놓고 움직일 수 있게 해주는 기반 작업입니다.

### 마무리

이번 커밋들은 모두 “사용자에게 보이는 표면을 정돈한다”는 공통점이 있었습니다. `/doctor`는 문제가 생겼을 때 확인할 수 있는 통로를 제공하고, 스킬 표면 정리는 불필요한 내부 정보를 숨기며, 테스트 보강은 변경된 설정 체계를 안정적으로 붙잡아 줍니다.

아직 더 다듬을 여지는 많습니다. 특히 자가진단 결과를 얼마나 친절하게 설명할지, 스킬 충돌 상황을 사용자에게 어느 정도까지 알려줄지는 계속 사용해 보면서 조정해야 할 것 같습니다. 그래도 이번 작업으로 vica가 조금 더 덜 헷갈리고, 조금 더 신뢰할 수 있는 도구에 가까워졌다고 느낍니다.
