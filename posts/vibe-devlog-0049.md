# vibe 개발일지 #12 - 오케스트레이션 SSOT 단일화와 신규 Z.ai(GLM) 엔진 연동 (17개 커밋)

**작업 기간**: 2026-06-27 ~ 2026-07-01

안녕하세요, vibe를 개발하고 있는 Su입니다. 

"Agent는 Model과 Harness의 결합이다"라는 vibe의 철학을 바탕으로, 요즘은 AI 코딩 하네스를 더욱 견고하고 유연하게 만드는 작업에 집중하고 있습니다. 최근 며칠 동안은 기능 추가 속도가 빨라지면서 슬금슬금 고개를 들던 **'설정 파편화(Config Drift)'** 문제를 해결하는 대대적인 리팩토링을 진행했습니다. 동시에 아시아권에서 강력한 성능을 보여주고 있는 **Z.ai(GLM) 엔진을 새롭게 지원**하게 되었습니다.

지난 5일간의 치열했던 기록을 공유합니다!

## 📝 이번 기간 작업 내용

이번 작업은 크게 **오케스트레이션 정비**, **신규 LLM 엔진(Z.ai) 도입**, 그리고 **Clone 스킬 안정화**라는 세 가지 축으로 진행되었습니다.

### 1. 오케스트레이션 및 정책 단일화 (SSOT 확보)
프로젝트 규모가 커지면서 에이전트별 프롬프트와 모델 라우팅 로직이 코드 이곳저곳에 흩어지기 시작했습니다. 이를 한곳에서 관리할 수 있도록 Single Source of Truth(SSOT)를 구축했습니다.
* **에이전트별 모델 할당 단일화** (`refactor(orchestration): consolidate per-agent...`): 에이전트마다 모델을 매핑하던 복잡한 분기문을 하나의 SSOT 맵으로 통합했습니다.
* **시스템 프롬프트 중복 제거** (`refactor(orchestration): dedupe task system prompts...`): 태스크별 중복되던 시스템 프롬프트를 하나의 SSOT 파일로 모아 일관성을 높였습니다.
* **라우팅 우선순위 단일화** (`refactor(orchestration): unify LLM routing priority...`): LLM 라우팅 우선순위를 결정하는 정책 함수(`policy function`)를 단 하나로 통일했습니다.
* **계약(Contract) 명세화** (`refactor(orchestration): codify orchestrator-skill contract...`): 오케스트레이터와 스킬 간의 통신 규격을 더 명확하게 정의하고 에이전트 네이밍을 통일했습니다.

### 2. Z.ai / GLM 모델 연동 및 환경 구축
새로운 성능 강자로 떠오른 Zhipu AI의 Z.ai (GLM) 엔진을 신규 벤더로 추가했습니다.
* **듀얼 키 지원 및 UI 라우팅 최적화** (`feat(zai): add Z.ai / GLM as an LLM vendor...`): Z.ai 엔진 연동을 마쳤으며, Z.ai 사용이 가능할 경우 모든 UI 렌더링 및 분석 작업을 GLM 최상위 모델로 우선 라우팅하도록 자동화했습니다.
* **실시간 모델 동기화 명령어 추가** (`feat(llm): add vibe llm command...`): `vibe llm` 명령어와 `vibe.llm` 스킬을 추가하여 최신 모델 목록을 로컬 SSOT에 즉시 갱신할 수 있도록 했습니다.
* **설정 마법사 및 상태창 데뷔** (`feat(zai): surface ZAI in central auth...`): 사용자가 `vibe setup`을 실행할 때 Z.ai 설정도 자연스럽게 진행할 수 있도록 위저드와 인증 상태 UI를 업데이트했습니다.

### 3. Clone 스킬 강화 및 버그 픽스
Figma나 기존 웹페이지를 템플릿 코드로 복제해 주는 `clone` 스킬의 완성도를 높였습니다.
* **상태 캡처 및 계약 모델 정립** (`feat(clone): interaction-model classification...`): 인터랙션 모델 분류와 UI 상태 캡처 기능을 강화하여 복제 스펙의 정확도를 높였습니다.
* **`page.evaluate` 호출 버그 수정** (`fix(clone): invoke PAGE_EXTRACT as call expression...`): 브라우저 컨텍스트 내에서 웹페이지를 추출할 때, `PAGE_EXTRACT` 함수를 올바르게 호출식(Call Expression)으로 평가하도록 수정하여 런타임 에러를 잡았습니다 (v2.16.2 반영).

## 💡 작업 하이라이트

### 1. "하나의 진실만 남기기" - 오케스트레이션 SSOT 리팩토링
이전에는 새로운 에이전트를 만들거나 프롬프트를 수정하려면 여러 파일(라우터, 에이전트 설정, 프롬프트 템플릿 등)을 동시에 고쳐야 했습니다. 개발자로서 가장 기분 나쁜 냄새인 **'동기화 누락으로 인한 미묘한 버그'**가 발생하기 딱 좋은 구조였죠.

```
[Before]
- agent-a.ts  --> 자체 모델 매핑 로직 & 시스템 프롬프트 수동 관리
- agent-b.ts  --> 자체 모델 매핑 로직 & 약간 다른 시스템 프롬프트
- router.ts   --> 또 다른 우선순위 판별 조건문

[After]
- orchestration/ssot/
    ├── models.ts   (모든 에이전트의 모델 할당 SSOT)
    ├── prompts.ts  (모든 태스크 프롬프트 SSOT)
    └── policy.ts   (단일화된 LLM 라우팅 우선순위 엔진)
```

이 리팩토링 덕분에 코드 베이스가 엄청나게 날씬해졌고, 새로운 LLM 엔진인 Z.ai를 추가할 때도 이 SSOT 한 곳만 건드려도 온전히 전체 시스템에 적용되는 짜릿함을 맛볼 수 있었습니다. 역시 기초 공사가 튼튼해야 확장이 편하다는 진리를 다시금 깨닫습니다.

### 2. 신형 엔진 GLM-5.2 탑재기 (feat. 철저한 리서치)
Z.ai 연동 작업을 시작할 때 처음에는 문서만 대충 훑어보고 플래그십 모델을 `glm-4.6`으로 지정해 연동했습니다. 그런데 실제 연동 후 테스트를 해보니 기대만큼의 퍼포먼스가 안 나오더라고요. 

이상하다 싶어 깊게 파고들며 리서치를 해보니, 최신 플래그십 모델 라인업의 정식 명칭은 **`glm-5.2`**였습니다! 급히 모델 라인업 문서를 보강하고 엔드포인트를 실사하여 수정했습니다(`fix(zai): GLM flagship is glm-5.2 (latest)`). 

수정 후 테스트해보니 UI 컴포넌트 분석과 명세 추출 작업에서 기대했던 놀라운 속도와 정확도를 보여주었습니다. 빠르게 최신 스펙을 파악하고 바로잡아 다행이었습니다.
