# vibe 개발일지 #43 - Codex와 Gemini를 품다: 멀티 엔진 하네스로의 진화 (10개 커밋)

**작업 기간**: 2026-04-17 ~ 2026-04-20

안녕하세요, Su입니다.

vibe 프로젝트를 시작하면서 제가 가졌던 철학은 확고했습니다. **"Agent는 Model과 Harness의 결합이다."** 모델이 아무리 똑똑해져도, 우리 프로젝트의 컨텍스트를 이해하고 품질을 강제하는 '하네스(Harness)'가 없다면 그건 진정한 에이전트라고 보기 어렵죠.

최근 며칠 동안은 vibe를 특정 모델에 종속되지 않는 더 강력한 하네스로 만들기 위해, 생태계 확장과 내부 구조 정밀화에 집중했습니다. 10개의 커밋 속에 녹아 있는 고민들을 공유합니다.

## 📝 이번 기간 작업 내용

이번 10개의 커밋은 크게 세 가지 영역으로 그룹화할 수 있습니다.

### 1. 멀티 엔진 생태계 확장 (Codex & Gemini 지원)
그동안 vibe는 Claude Code와의 궁합에 집중해 왔지만, 이제 영역을 넓힙니다. OpenAI의 Codex CLI와 Google의 Gemini CLI를 공식 지원하기 시작했습니다.
*   **feat(setup/init/update):** `--codex`, `--gemini` 플래그를 추가하여 사용자가 원하는 AI 엔진을 즉시 선택하고 설정할 수 있게 했습니다.
*   **feat(setup):** 글로벌 CLI 환경에서도 각 엔진별 설정을 독립적으로 관리할 수 있도록 지원합니다.

### 2. 아키텍처 정밀화 (SSOT 및 스코프 분리)
설정 파일이 여기저기 흩어지는 것은 유지보수의 적이죠. 이번에 설정을 관리하는 방식을 대대적으로 손봤습니다.
*   **fix(update):** 언어 규칙(language rules)과 프로비저너(provisioner)가 `.vibe/` 디렉토리를 **SSOT(Single Source of Truth)**로 바라보도록 강제했습니다. 이제 설정이 꼬일 염려가 없습니다.
*   **refactor(setup):** vibe의 설정을 'Global(전역)'과 'Project(프로젝트)' 스코프로 명확히 분리했습니다. 공통 스킬은 전역에서, 프로젝트 특화 룰은 로컬에서 관리합니다.

### 3. 성능 최적화 및 DX 개선
실제로 도구를 사용하는 개발자의 피드백을 반영하여 사소하지만 치명적인 불편함들을 제거했습니다.
*   **perf(hooks):** 훅 디스패처(dispatcher)를 병렬화(parallelize)했습니다. 여러 개의 검증 훅이 돌 때 하나하나 기다릴 필요가 없어졌죠.
*   **fix(hooks):** 불필요한 `exit-1` 노이즈를 억제했습니다. 에러가 아닌 경고 상황에서 터미널이 붉게 물드는 스트레스를 줄였습니다.

## 💡 작업 하이라이트

### "Any Model, Same Harness"
이번 작업의 가장 큰 수확은 `vibe init --gemini` 혹은 `vibe init --codex` 한 줄로, 어떤 AI 모델을 사용하든 동일한 품질 보장 시스템(vibe spec/run)을 적용할 수 있게 되었다는 점입니다.

**Before:**
- Claude Code 위주로 설계된 하네스 구조.
- 타 모델 사용 시 설정 파일을 수동으로 옮겨야 함.

**After:**
- `vibe`가 중앙 관제탑 역할 수행.
- `references.*` 경로가 선택된 하네스의 글로벌 디렉토리로 자동 라우팅됨 (`fix(config)` 커밋).
- 모델은 바뀌어도 개발자가 정의한 **Minimum quality**는 변함없이 유지됨.

### 병렬 디스패처 도입
하네스는 안전을 위해 존재하지만, 개발 속도를 늦춰서는 안 됩니다. 기존 순차적 훅 실행 방식은 체크포인트가 많아질수록 지연 시간이 늘어나는 문제가 있었습니다. 이번 `perf(hooks)` 작업을 통해 훅 실행 구조를 비동기 병렬 구조로 변경했고, 체감 속도를 약 40% 이상 개선했습니다. (물론, 의존성이 있는 훅들은 순서를 보장합니다.)
