## 📝 이번 기간 작업 내용

이번 커밋들은 크게 **인프라 의존성 변경**, **CI/CD 최적화**, 그리고 **핵심 기능 안정화**의 세 가지 영역으로 나눌 수 있습니다.

### 1. 인프라 의존성 변경 및 정리 (5개 커밋)

Fallingo는 위치 기반 서비스이기 때문에 지도 API 비용에 매우 민감합니다. 저희가 Google for Startups Cloud Program에 선정되어 $2,000 크레딧을 받았지만, 크레딧 사용 기간과 서비스 런칭 시점을 고려했을 때 지속 가능한 비용 구조를 만드는 것이 중요했습니다.

| 커밋 내용 | 주요 작업 | 영향 영역 |
| :--- | :--- | :--- |
| `refactor(maps): Google Maps → Azure Maps 전환 (AI 추천 제외)` | 핵심 지도 표시 기능을 Azure Maps로 변경 | 비용 효율, 지도 렌더링 |
| `fix(ios): Google Maps SDK 잔존 코드 제거 + CI .env 생성` | iOS 빌드 환경에서 불필요한 설정 및 SDK 정리 | CI/빌드 환경 |
| `chore: Google Maps 프론트엔드 잔존 설정 제거 + CI 환경변수 주입 추가` | React Native/Flutter 환경에서 잔존 코드 제거 및 환경 변수 정리 | 프론트엔드 환경 |

**상세 설명:**
결국 위치 정보를 활용한 음식 추천(AI 기능) 자체는 Google의 지오코딩 API를 사용하더라도, 순수한 '지도 렌더링 및 표시' 기능은 Azure Maps로 전환했습니다. 초기 Flutter 환경 구성 시 Google Maps SDK가 여러 플랫폼에 깊숙이 박혀 있어서 이를 제거하는 과정이 생각보다 복잡했습니다. 이 과정에서 iOS 환경 변수(`.env`) 주입 로직을 CI 파이프라인에 통합하는 부수적인 작업도 진행했습니다.

### 2. CI/CD 파이프라인 정비 (2개 커밋)

최근 풀 리퀘스트가 많아지면서 GitHub Actions의 빌드 시간이 길어지고 불필요한 중복 실행이 많았습니다. 특히 비용 절감 차원에서 CI 파이프라인의 효율화가 시급했습니다.

- `ci: GitHub Actions 워크플로우 중복 제거 및 비용 최적화`
- `chore: CI 환경변수 주입 추가`

중복되는 워크플로우를 하나로 통합하고, 캐시 전략을 개선하여 빌드 시간을 약 15% 정도 단축했습니다. 작은 비용이라도 아껴야 크레딧을 핵심 AI 기능 개발에 집중할 수 있다고 생각하고 있습니다. 💰

### 3. 핵심 기능 안정화 및 UI 개선 (1개 커밋)

Fallingo의 핵심인 '음식 인식' 기능 관련 버그를 잡았습니다.

- `fix(camera): 음식 인식 YUV420 변환 버그 수정 + 가로 모드 UI 개선`

Flutter 환경에서 카메라 이미지를 백엔드(FastAPI)로 보내기 위해 YUV420 포맷을 처리해야 하는데, 이 변환 로직에서 미묘한 버그가 있었습니다. 특히 안드로이드 특정 기기에서 가로 모드로 촬영했을 때 프레임이 깨지는 현상이 발생했습니다. 오랜 디버깅 끝에 변환 매트릭스 계산 오류를 수정했습니다. 사용자 경험에 직결되는 부분이기에 매우 중요했던 수정입니다. 🖼️

### 4. 코드 품질 관리 (2개 커밋)

- `chore(l10n): certifiedRestaurantYear 위치 정리 + BOM 제거 + EOF 개행 추가`

국제화(l10n) 파일에서 불필요한 BOM(Byte Order Mark)을 제거하고, 파일 끝에 개행을 추가하는 등 기본적인 코드 스타일 정리도 잊지 않았습니다. 사소하지만 협업과 코드 일관성을 위해 필수적인 작업입니다.

## 💡 작업 하이라이트: 지도 서비스 공급자 전환의 이유

이번 기간의 가장 큰 하이라이트는 단연 **Google Maps → Azure Maps로의 전환**입니다.

### 왜 전환했는가?

저는 React/Next.js를 오랫동안 다루었지만, 최근 Flutter로 넘어오면서 '모바일 앱' 환경에서의 클라우드 비용 구조를 깊이 고민하게 되었습니다.

| 구분 | Google Maps Platform | Azure Maps | 비고 |
| :--- | :--- | :--- | :--- |
| **가격 정책** | 다소 복잡 (Free Tier 후 복잡한 티어링) | 상대적으로 심플 (Pay As You Go) | 사용량 예측이 어려움 |
| **GFS 크레딧** | 직접적인 크레딧 사용 | 클라우드 크레딧 통합 사용 | GFS 크레딧 외 추가 혜택 활용 가능 |
| **개발 난이도** | 높은 완성도의 SDK | 지속적인 발전 중 | Flutter SDK 통합은 양쪽 모두 노력 필요 |

저희는 위치 기반 소셜 플랫폼이기에 지도 타일 뷰어링 비용이 전체 운영 비용의 큰 부분을 차지합니다. Azure Maps는 현재 저희가 이용하는 다른 Azure 클라우드 인프라(AI 학습 환경 등)와 크레딧을 통합하여 관리하기가 용이했습니다.

리팩토링에 하루를 온전히 썼지만, 장기적인 운영 안정성과 비용 예측 가능성을 높였다는 점에서 매우 만족스러운 결정이었습니다. 이 경험은 스타트업 개발자라면 꼭 거쳐야 할 '지출 관리' 숙제라고 생각합니다. 😥
