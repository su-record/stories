## 이번 주 요약

백엔드 Phase 1 구현을 완료하고, Flutter 프론트엔드 개발을 본격적으로 시작했다. GCP Load Balancer 경로 기반 라우팅을 설정하고, 프론트엔드-백엔드 통합 배포를 완료했다. 55개 커밋, 그중 26개가 버그 수정이었다. 배포는 역시 만만치 않았다.

## 주요 작업

### 프론트엔드 기반 구축
- Flutter 웹 프로젝트 초기화
- 주요 화면 구현 (Feed, AI 추천, 프로필)
- Mock API를 활용한 UI 개발
- Lint 에러 해결 (코드 품질 관리 시작)

- 프로덕션 환경 변수 관리
- GitHub Actions 테스트 워크플로우 구축

### AI 추천 기능 추가

- 티어별 기능 접근 제어 로직

### 데이터 & 메타데이터
- 실제적인 샘플 데이터 추가
- 파비콘 및 웹 메타데이터 개선
- SEO 기본 설정

## 배운 점

**환경변수와의 싸움**

Flutter 웹 빌드 시 환경변수 주입이 까다로웠다. `--dart-define`으로 빌드 타임에 주입해야 한다는 것을 알았지만, 프로덕션과 개발 환경을 분리하는 전략을 세우는 데 시간이 걸렸다. 로컬에서 잘 되던 것이 배포하면 안 되는 경험은 여전히 낯설다.

**경로 라우팅의 복잡함**

GCP Load Balancer로 `/api/*`는 백엔드로, `/*`는 프론트엔드로 라우팅하려고 했다. 생각보다 복잡했다. CORS 설정, preflight 요청 처리, URL 스키마 통일. 하나씩 해결하면서 배포 인프라의 복잡함을 체감했다.

**버그 수정이 절반**

55개 커밋 중 26개가 버그 수정이었다. 배포 환경에서만 발생하는 이슈들을 잡는 데 시간이 많이 걸렸다. 로컬과 프로덕션 환경의 차이를 최소화해야 한다는 것을 배웠다. 테스트 자동화의 필요성도 절실히 느꼈다.

**Claude가 작성한 스크립트**

배포 스크립트를 Claude와 함께 작성했다. GCP 설정 베스트 프랙티스를 빠르게 학습할 수 있었다. 하지만 최종 검증은 직접 해야 한다는 것을 잊지 않았다. AI는 훌륭한 조력자지만, 책임은 내가 진다.
