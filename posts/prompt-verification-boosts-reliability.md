AI에게 결과만 맡기면, 오류는 가장 조용한 단계에서 생깁니다.

## 답을 받는 요청과 믿을 수 있는 요청은 다릅니다

![검증은 산출물 조건·테스트 방법·실패 처리·증거로 쪼갠다](/images/prompt-verification-boosts-reliability/01.png)
*네칸검증*


대부분의 프롬프트는 산출물만 요구합니다. 보고서 써라. 메일 분류해라. 표로 정리해라. 여기까지는 어렵지 않습니다.

문제는 그 다음입니다. 결과가 맞는지 확인하지 않습니다. AI는 틀린 표도 그럴듯하게 만듭니다. 빠진 행도 자연스럽게 설명합니다.

자동화에서는 이 문제가 더 커집니다. Google Gemini Spark 같은 에이전트는 자연어 지시로 태스크, 스킬, 스케줄을 실행합니다. Gmail, 캘린더, 서드파티 앱과도 연결됩니다.

자는 동안에도 백그라운드에서 계속 돌아가는 구조입니다. 그래서 더 위험합니다. 한 번 잘못 만든 규칙은 밤새 반복될 수 있습니다.

요청 안에 검증을 넣어야 합니다. 답을 만들게 하는 문장과, 그 답을 의심하게 하는 문장은 달라야 합니다.

---

## 검증 문장은 네 가지로 쪼갭니다

![자동화 요청에는 실패하면 멈추는 조건을 먼저 넣는다](/images/prompt-verification-boosts-reliability/02.png)
*실패시정지*


검증은 추상적으로 쓰면 잘 작동하지 않습니다. '정확히 확인해줘'는 거의 도움이 안 됩니다. 무엇을 확인할지, 어떻게 확인할지, 실패하면 어떻게 할지까지 적어야 합니다.

제가 쓰는 기본 구조는 네 칸입니다. 산출물 조건, 테스트 방법, 실패 처리, 증거입니다.

| 항목 | 나쁜 요청 | 좋은 요청 |
|---|---|---|
| 산출물 조건 | 표로 정리해줘 | 2026년 데이터만 포함하고, 행 수를 마지막에 적어라 |
| 테스트 방법 | 검토해줘 | 날짜 형식이 YYYY-MM-DD인지 전 행 검사해라 |
| 실패 처리 | 이상하면 알려줘 | 1건이라도 실패하면 최종 요약 대신 실패 목록을 먼저 출력해라 |
| 증거 | 근거도 줘 | 사용한 출처명, 확인 날짜, 판단 기준을 별도 열로 남겨라 |

차이는 큽니다. 앞의 요청은 결과만 봅니다. 뒤의 요청은 실패를 처음부터 염두에 둡니다.

프롬프트에 붙이면 이렇게 됩니다.

```text
목표: 월간 고객 문의를 분류한다.
산출물: 카테고리별 건수와 대표 사례 3개를 표로 출력한다.
검증 1: 전체 문의 수와 카테고리별 합계가 같은지 확인한다.
검증 2: 중복 문의 ID가 있는지 검사한다.
검증 3: 날짜가 2026-07 형식에 맞는 행만 포함한다.
실패 시: 요약을 만들지 말고 실패한 행 번호와 이유를 먼저 출력한다.
근거: 분류 기준과 애매한 사례를 별도 목록으로 남긴다.
마지막 줄: 통과 또는 실패 중 하나만 적는다.
```

짧지만 역할은 분명합니다. 이렇게 쓰면 AI가 작성자 역할만 하지 않고 1차 리뷰도 하게 됩니다.

## Gemini Spark 자동화에는 실패 조건을 먼저 넣습니다

![테스트는 실행보다 결과 해석과 실패 기준이 중요하다](/images/prompt-verification-boosts-reliability/03.png)
*결과해석*


Gemini Spark는 코딩 지식 없이 자연어로 업무 자동화를 만들 수 있습니다. 태스크, 스킬, 스케줄을 조합해 반복 업무를 처리합니다.

예를 들어 매일 오전 8시에 Gmail 문의를 읽고, 캘린더 일정을 확인한 뒤, Slack이나 문서에 요약을 남길 수 있습니다.

처음 쓰는 요청은 보통 이렇게 생깁니다.

```text
매일 오전 8시에 어제 들어온 고객 문의 메일을 요약해서 팀 문서에 정리해줘.
```

틀렸습니다. 너무 많은 것을 믿고 있습니다. 메일 범위, 중복 처리, 누락 확인, 실패 알림이 없습니다.

수정한 요청은 이렇습니다.

```text
매일 오전 8시에 전날 00:00부터 23:59까지 받은 고객 문의 메일을 처리해라.

작업:
1. 제목, 발신자, 수신 시간, 문의 유형, 긴급도를 표로 정리한다.
2. 같은 Message-ID는 1건으로 계산한다.
3. 캘린더 일정과 충돌하는 긴급 문의는 별도 표시한다.

검증:
1. 검색된 메일 수와 표의 행 수가 같은지 확인한다.
2. Message-ID 중복 제거 전후 건수를 출력한다.
3. 수신 시간이 범위 밖인 메일은 제외하고 목록에 남긴다.
4. 문서 업데이트 후 행 수가 이전보다 증가했는지 확인한다.

실패 시:
검증 하나라도 실패하면 문서에 쓰지 말고, 실패 사유만 나에게 메일로 보낸다.
```

여기서 중요한 부분은 마지막 세 줄입니다. 자동화는 실패했는데도 조용히 지나가면 안 됩니다. 멈추고 흔적을 남겨야 합니다.

24시간 작동하는 에이전트일수록 이 원칙이 중요합니다. 사람이 확인하기 전까지 오류가 계속 쌓일 수 있기 때문입니다.

## 테스트 결과도 해석까지 시킵니다

테스트를 실행하라는 요청만으로는 부족합니다. 테스트 결과를 어떻게 읽을지도 같이 줘야 합니다.

Microsoft Learn의 Azure Pipelines 문서는 2025-12-17 기준으로 테스트 결과 검토를 별도 단계로 다룹니다. 실행보다 리뷰가 중요하다는 뜻입니다.

GoodNovel도 2026-02-05에 '테스트 결과를 어떻게 해석해야 하나요?'라는 문서를 냈습니다. 테스트는 통과 숫자만 보는 일이 아닙니다.

성능 테스트에서는 더 분명합니다. JMeter 계열 결과는 확장자가 jtl인 파일로 저장될 수 있습니다. 파일 존재 여부, 샘플 수, 실패율을 같이 확인해야 합니다.

| 테스트 요청 | 확인해야 할 것 | 실패 기준 |
|---|---|---|
| API 테스트 실행 | 상태 코드, 응답 시간, 실패율 | 5xx 1건 이상 또는 p95 800ms 초과 |
| 성능 테스트 실행 | jtl 파일 생성, 샘플 수, 에러율 | 샘플 0건 또는 에러율 1% 초과 |
| UI 테스트 실행 | 실패 스텝, 스크린샷, 재현 경로 | 동일 스텝 2회 이상 실패 |
| 배포 검증 | 헬스체크, 로그, 롤백 조건 | 헬스체크 3회 연속 실패 |

AI 테스트 도구도 늘고 있습니다. 2026-05-17 문서 기준으로 KaneAI 같은 도구가 언급됩니다. 도구가 바뀌어도 원칙은 같습니다.

요청은 이렇게 끝나는 편이 좋습니다.

```text
테스트를 실행한 뒤 결과를 해석해라.
단순 통과 여부만 말하지 마라.
실패한 케이스, 실패율, 재현 가능성, 로그 위치를 표로 남겨라.
결과 파일이 있으면 파일명과 확장자를 적어라.
성능 테스트라면 jtl 파일 생성 여부와 샘플 수를 확인해라.
판단이 애매하면 성공으로 처리하지 말고 보류로 표시해라.
```

'성공'이라는 말은 가볍게 쓰면 안 됩니다. 조건을 통과했을 때만 붙일 수 있는 라벨입니다.

## 요청 끝에 붙이는 8줄

매번 긴 프롬프트를 새로 만들 필요는 없습니다. 작업 요청 끝에 검증 블록만 붙여도 품질이 달라집니다.

```text
검증 기준:
1. 산출물이 요청한 형식과 일치하는지 확인한다.
2. 숫자 합계, 행 수, 날짜 범위를 검사한다.
3. 중복, 누락, 범위 밖 데이터를 별도 표시한다.
4. 사용한 출처와 확인 날짜를 남긴다.
5. 테스트 또는 확인 절차를 짧게 설명한다.
6. 실패한 항목이 있으면 최종 답변보다 먼저 보여준다.
7. 불확실한 내용은 추정하지 말고 불확실로 표시한다.
8. 마지막에 통과, 실패, 보류 중 하나를 적는다.
```

이 8줄이 AI를 더 똑똑하게 만들지는 않습니다. 대신 거짓 확신을 줄여줍니다.

좋은 요청은 답을 빨리 받기 위한 문장만은 아닙니다. 틀렸을 때 멈출 수 있게 쓰는 문장이기도 합니다.
