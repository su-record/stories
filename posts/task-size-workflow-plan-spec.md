1파일 작업을 스펙으로 시작하면 시간이 샙니다. 5파일 작업을 한 문장으로 맡기면 리뷰가 감당이 안 됩니다.

문제는 모델 성능보다 운영 방식에 가까웠습니다. 크기가 다른 일을 같은 방식으로 처리한 탓입니다.

---

## 파일 수가 기준이 되는 이유

![예상 수정 파일 수로 워크플로우를 나눈다](/images/task-size-workflow-plan-spec/01.png)
*파일수분기*


AI 코딩에서 작업 크기는 시간보다 파일 수로 보는 편이 더 정확했습니다. 시간은 예상치이고, 파일 수는 실제 영향 범위에 가깝습니다.

파일이 늘어나면 결정할 것도 같이 늘어납니다. 함수명, 타입, 테스트, 상태, 라우팅, 롤백 기준이 함께 움직입니다.

| 수정 규모 | 대표 작업 | 추천 방식 | 산출물 | 실패 신호 |
|---|---|---|---|---|
| 0파일 | 질문, 조사, 설계 검토 | 대화 | 답변 | 코드 수정을 시작함 |
| 1파일 | 버튼 문구, 조건문, 작은 컴포넌트 | 플랜모드 | 짧은 계획 | 파일 밖을 건드림 |
| 2파일 | 구현 파일과 테스트 파일 | 플랜모드 | 수정 순서 | 테스트 범위가 흐림 |
| 3파일 이상 | API, UI, 상태, 테스트 동시 변경 | 스펙 | 명시 문서 | 요구사항을 재해석함 |

공개된 2026년 Claude Code 계열 가이드에서도 비슷한 기준이 보입니다. 단순한 단일 파일 수정은 계획 없이 진행해도 되는 경우가 있습니다.

반대로 3개 이상 파일을 건드리는 작업은 실행 전에 계획을 세우는 쪽을 권합니다. 제가 덧붙인 규칙은 그 다음입니다.

3파일 이상은 계획이 아니라 스펙으로 올립니다. 도구의 공식 규칙은 아니고, 제가 쓰는 운영 규칙입니다.

---

## 1-2파일은 플랜모드로 충분합니다

![실행 중 3파일 이상으로 커지면 멈추고 스펙으로 전환한다](/images/task-size-workflow-plan-spec/02.png)
*증가시중단*


플랜모드는 읽기 전용 흐름입니다. 코드를 먼저 분석하고, 수정하기 전에 접근 방식을 제안하게 만듭니다.

핵심은 편집을 늦추는 데 있습니다. AI가 바로 고치기 시작하면 작은 작업도 방향을 잃기 쉽습니다.

제가 쓰는 프롬프트는 짧습니다. 1-2파일 작업에서는 이 정도로 충분했습니다.

```text
Plan Mode로 읽기만 합니다.
목표: 검색 입력에 300ms debounce를 추가합니다.
예상 수정 파일은 1-2개입니다.

확인할 것:
1. 현재 입력 상태가 어디서 관리되는지
2. 기존 테스트가 있는지
3. 수정 파일 목록
4. 변경 순서

아직 코드는 수정하지 않습니다.
```

예를 들어 `SearchBox.tsx` 하나만 고치는 작업이 있습니다. 이런 경우 스펙 문서는 과합니다.

테스트까지 같이 고치면 2파일입니다. `SearchBox.tsx`, `SearchBox.test.tsx` 정도입니다.

이때 필요한 것은 계약서가 아닙니다. 수정 순서와 멈춤 기준이면 됩니다.

```text
승인 후 실행합니다.
수정 파일은 SearchBox.tsx와 SearchBox.test.tsx로 제한합니다.
다른 파일이 필요하면 먼저 이유를 설명합니다.
```

이 문장 하나가 꽤 중요했습니다. AI가 유틸 파일을 새로 만들고, 훅을 분리하고, 테스트 헬퍼까지 건드리는 일을 막아줍니다.

README 오타나 문구 한 줄은 예외입니다. 그런 작업에는 플랜모드도 비용입니다.

다만 AI에게 판단을 맡기는 순간부터는 달라집니다. 1파일이어도 조건 분기가 있으면 먼저 읽게 합니다.

---

## 3+파일은 스펙으로 계약합니다

![인증·결제·DB 같은 고위험 작업은 파일 수와 무관하게 스펙이 먼저다](/images/task-size-workflow-plan-spec/03.png)
*위험은스펙*


3파일부터는 작업이 단순한 코드 수정이 아니라 조율에 가까워집니다. UI와 API가 만나고, 테스트가 따라오고, 타입이 흔들립니다.

로그인에 `remember me`를 넣는다고 해보겠습니다. 말로는 작아 보이지만 파일은 금방 늘어납니다.

| 파일 | 변경 내용 | 숨은 결정 |
|---|---|---|
| `LoginForm.tsx` | 체크박스 추가 | 기본값, 문구, 접근성 |
| `auth.schema.ts` | 요청 타입 변경 | optional 여부 |
| `auth.api.ts` | payload 전달 | 기존 호출 호환성 |
| `session.ts` | 만료 정책 변경 | 보안 기준 |
| `LoginForm.test.tsx` | 테스트 추가 | 성공, 실패 케이스 |

이 작업을 한 문장으로 맡기면 AI는 빈칸을 알아서 채웁니다. 문제는 그 빈칸이 제품 결정이라는 점입니다.

그래서 스펙을 씁니다. 길게 쓰지는 않습니다. 한 화면 안에 들어오게 씁니다.

```md
# Spec: remember me 로그인 옵션

## 목표
사용자가 로그인 유지 여부를 선택할 수 있게 한다.
체크하지 않으면 기존 세션 정책을 유지한다.

## 수정 범위
- LoginForm.tsx
- auth.schema.ts
- auth.api.ts
- session.ts
- LoginForm.test.tsx

## 승인 기준
- 체크박스 기본값은 false다.
- true일 때만 long-lived session을 요청한다.
- 기존 로그인 테스트는 통과해야 한다.
- 새 테스트는 true, false 케이스를 모두 포함한다.

## 제외 범위
- 소셜 로그인은 수정하지 않는다.
- UI 디자인 시스템 컴포넌트는 새로 만들지 않는다.
- 세션 저장소 구조는 바꾸지 않는다.

## 롤백
문제가 생기면 payload 필드와 UI 체크박스를 제거한다.
기존 로그인 API 시그니처는 유지한다.
```

스펙은 AI를 느리게 만드는 문서가 아닙니다. 리뷰를 짧게 만드는 기준에 가깝습니다.

제가 보는 핵심은 결과물보다 루프입니다. AI 빌더가 만들고, 사람은 판별합니다.

그 판별 기준이 머릿속에만 있으면 매번 말이 달라집니다. 스펙은 그 기준을 밖으로 꺼내는 역할을 합니다.

---

## 실제 운영 규칙

저는 작업을 시작하기 전에 예상 수정 파일 수를 먼저 묻습니다. 이 답으로 워크플로우를 정합니다.

```text
먼저 작업을 분석합니다.
예상 수정 파일을 목록으로 제시합니다.
1-2파일이면 Plan Mode로 진행합니다.
3파일 이상이면 spec.md 초안을 먼저 작성합니다.
아직 코드는 수정하지 않습니다.
```

AI가 처음에는 2파일이라고 했는데, 실행하다 보니 4파일이 필요해지는 경우가 있습니다. 그때는 중단합니다.

멈추지 않으면 작업은 계속 커집니다. 나중에는 리뷰어도 원래 요청이 무엇이었는지 놓칩니다.

제가 쓰는 전환 기준은 단순합니다.

| 상황 | 처리 |
|---|---|
| 1파일에서 끝남 | 플랜 승인 후 바로 수정 |
| 2파일, 테스트 포함 | 플랜 승인 후 수정 |
| 3파일 예상 | 스펙 작성 후 승인 |
| 실행 중 3파일로 증가 | 중단 후 스펙 전환 |
| DB, 인증, 결제 포함 | 파일 수와 무관하게 스펙 |

파일 수만 보는 규칙은 아닙니다. 위험도가 높은 영역은 더 빨리 스펙으로 올립니다.

인증, 결제, 데이터 삭제, 마이그레이션은 1파일이어도 스펙이 낫습니다. 실패 비용이 다릅니다.

기준은 단순합니다. AI가 추측해도 되는 작업인지, 추측하면 안 되는 작업인지입니다.

1-2파일은 플랜모드로 추측을 줄입니다. 3+파일은 스펙으로 추측을 막습니다.

작업 크기를 나누면 프롬프트가 짧아집니다. 리뷰 기준도 고정됩니다. 실패했을 때 되돌릴 위치도 보입니다.
