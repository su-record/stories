## 문제는 답변 품질이 아니었습니다

AI에게 일을 시키면 처음에는 답변 품질만 봅니다.

좋은 모델을 고릅니다. 프롬프트를 길게 씁니다. 예시를 붙입니다. 결과가 조금 좋아집니다.

그런데 며칠 지나면 다른 문제가 보입니다.

일이 남지 않습니다.

어제 시킨 일이 어느 상태인지 모릅니다. 누가 맡았는지 모릅니다. 막혔는지 끝났는지도 대화창을 다시 읽어야 압니다. 같은 지시를 매번 반복합니다. 승인 전에 발행되면 안 되는 작업도 채팅 흐름 안에서는 쉽게 섞입니다.

저는 이 문제를 Hermes Agent로 풀고 있습니다. 핵심은 Hermes를 채팅봇으로 보지 않는 것입니다. 작업 시스템으로 봅니다.

---

## Hermes를 작업 시스템으로 본다는 뜻

Hermes Agent는 Nous Research가 만든 터미널 기반 AI 에이전트 프레임워크입니다. 로컬, Docker, SSH 같은 실행 환경에서 터미널과 파일 도구를 쓰고, 프로필과 스킬과 메모리를 통해 상태를 나눕니다. 게이트웨이를 붙이면 Telegram, Discord, Slack 같은 메시징 플랫폼에서도 같은 에이전트를 호출할 수 있습니다.

여기서 중요한 것은 기능 목록이 아닙니다.

중요한 것은 일을 구성하는 단위입니다.

| 단위 | 역할 | 운영에서 생기는 효과 |
|---|---|---|
| 프로필 | 에이전트 한 명의 집 | 역할, 기억, 세션, 도구가 섞이지 않습니다 |
| SOUL | 정체성과 금지선 | 해도 되는 일과 하면 안 되는 일이 분명해집니다 |
| 스킬 | 반복 업무 절차 | 매번 설명하지 않아도 같은 품질로 반복합니다 |
| 도구 | 실제 실행 권한 | 검색, 파일 수정, 터미널 실행이 가능합니다 |
| 게이트웨이 | 외부 채널 연결 | CLI 밖에서 팀처럼 요청하고 보고받습니다 |
| 칸반 | 작업 상태 저장 | 담당, 차단, 완료, 의존성이 기록됩니다 |
| 크론 | 반복 실행 | 정해진 시간에 수집, 요약, 점검을 돌립니다 |

이 조합이 만들어지면 AI는 답변자가 아니라 작업자가 됩니다.

## 대화창은 상태 저장소가 아닙니다

대화창은 일을 시작하기 좋습니다. 그러나 운영 상태를 맡기기에는 약합니다.

예를 들어 콘텐츠 발행을 생각해봅니다.

1. 리서치 봇이 자료를 모읍니다.
2. 작성 봇이 초안을 씁니다.
3. 검수 봇이 문체와 사실을 봅니다.
4. 발행 봇이 승인된 글만 배포합니다.

이 흐름을 한 채팅방에서만 굴리면 금방 엉킵니다. 누가 지금 무엇을 해야 하는지 알기 어렵습니다. 검수 전 초안이 발행될 수도 있습니다. 실패 로그가 흩어집니다.

그래서 Sutory에서는 작업을 카드로 남깁니다. 카드는 담당자를 가집니다. 막히면 blocked가 됩니다. 끝나면 summary와 metadata가 남습니다.

대화는 인터페이스입니다. 보드가 상태입니다.

## 서버 한 대에서 팀처럼 굴리는 구조

이전 글에서는 Oracle Cloud 무료 ARM 서버 위에 Hermes를 올리고, 디스코드 봇 8명을 직원처럼 나눈 과정을 적었습니다.

그 글은 전체 설치기였습니다. 이번 시리즈는 운영기입니다.

저는 처음부터 거창한 자동화를 만든 것이 아닙니다. 먼저 역할을 나눴습니다.

| 봇 | 맡은 일 |
|---|---|
| Pepper | 요청을 받아 작업을 나누는 오케스트레이터 |
| Tony | 구현 담당 |
| Linus | 코드 리뷰와 QA |
| Miranda | 콘텐츠 초안과 마케팅 |
| Gutenberg | 승인된 콘텐츠 배포 |

이 구조에서 Gutenberg는 아무 글이나 발행하지 않습니다. `#배포-승인`을 통과한 것만 배포합니다. 이 규칙은 말투가 아니라 권한의 경계입니다.

AI 직원 운영에서 제일 중요한 것은 똑똑한 답변이 아닙니다. 경계입니다.

---

## 처음부터 전부 열 필요는 없습니다

Hermes에는 많은 도구가 있습니다. 웹 검색, 파일 편집, 터미널 실행, 브라우저, 이미지 분석, 크론, 메모리, 세션 검색, delegation 같은 도구를 켤 수 있습니다.

하지만 도구는 권한입니다.

처음부터 모든 권한을 열면 운영이 아니라 사고 실험이 됩니다. 작은 단위로 시작하는 편이 낫습니다.

| 단계 | 열어볼 것 | 이유 |
|---|---|---|
| 1단계 | CLI + web/file | 자료 확인과 문서 정리에 충분합니다 |
| 2단계 | terminal | 빌드, 테스트, git 상태 확인이 필요할 때 엽니다 |
| 3단계 | gateway | 요청과 보고를 팀 채널로 옮깁니다 |
| 4단계 | kanban | 여러 에이전트가 끊기지 않게 작업을 나눕니다 |
| 5단계 | cron | 반복 점검과 브리핑을 자동화합니다 |

순서는 절대 규칙이 아닙니다. 다만 권한을 열기 전에 실패했을 때 되돌릴 방법을 먼저 정해야 합니다.

## 이 시리즈에서 다룰 것

이 시리즈는 설치법을 반복하지 않습니다. 운영법을 씁니다.

다룰 내용은 세 가지입니다.

1. AI 직원을 프로필로 나누는 법.
2. SOUL, 스킬, 메모리로 반복 업무를 남기는 법.
3. 칸반과 크론으로 작업과 자동화를 끊기지 않게 만드는 법.

Hermes의 장점은 기능이 많다는 데 있지 않습니다. 기능을 묶어 작은 운영체계로 만들 수 있다는 데 있습니다.

서버 한 대와 봇 몇 명으로 시작해도 됩니다. 중요한 것은 첫날부터 상태와 권한과 승인선을 분리하는 것입니다.
