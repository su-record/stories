## 최근 작업 일지: “겉으로는 작은 버그, 안쪽으로는 신뢰도 정리”

이번 작업은 `vica`에서 사용자가 직접 마주치는 어색함을 줄이는 데 집중했습니다. 기능을 크게 늘렸다기보다는, 이미 있는 흐름이 더 정확하게 동작하도록 다듬은 쪽에 가깝습니다. 특히 터미널 환경 감지, 오퍼레이터 응답 품질, 리뷰/취소 흐름처럼 “한 번 삐끗하면 신뢰가 떨어지는” 부분들을 손봤습니다.

## 터미널 로고 표시 문제: 추측보다 확인

먼저 터미널에서 로고가 빈 화면처럼 보이던 문제를 고쳤습니다. 기존에는 `LC_TERMINAL` 같은 환경 변수를 보고 “아마 이런 터미널이겠지” 하고 추측하는 방식이었는데요. 이 방식은 생각보다 허술했습니다. 환경 변수는 비어 있을 수도 있고, 터미널/셸/실행 환경에 따라 기대와 다르게 들어오는 경우도 많았습니다.

그래서 이번에는 단순 추측 대신 터미널에 능동적으로 질의하는 방식으로 바꿨습니다. 말하자면 “네가 누구니?”를 직접 물어보고, 응답을 기반으로 처리하도록 한 셈입니다. 덕분에 메뉴 쪽에서 로고가 제대로 표시되지 않던 상황을 줄일 수 있었습니다.

이런 문제는 겉으로 보면 사소해 보이지만, CLI 도구에서는 첫 화면이 곧 첫인상이라 꽤 중요합니다. 실행하자마자 빈 화면처럼 보이면 사용자는 “이거 멈춘 건가?” 하고 느끼기 쉽거든요.

## 오퍼레이터 응답 제어: 챗봇 말투를 막기

두 번째는 오퍼레이터 쪽에서 플랜 모델이 엉뚱하게 “파일 업로드해주세요” 같은 챗봇식 응답을 내던 문제를 막았습니다.

`vica`의 오퍼레이터는 일반 상담 챗봇처럼 동작하면 안 됩니다. 사용자의 요청을 실행 가능한 작업 단위로 해석하고, 현재 컨텍스트 안에서 판단해야 합니다. 그런데 모델이 익숙한 챗봇 패턴으로 빠지면, 실제 개발 도구로서는 굉장히 어색한 결과가 나옵니다. 특히 이미 로컬 파일과 코드 컨텍스트를 보고 있어야 하는 상황에서 “업로드해주세요”라고 말하면 사용자는 바로 맥이 빠지죠.

이번 수정에서는 그런 류의 응답이 나오지 않도록 제어를 강화했습니다. 모델이 할 수 있는 말과 하면 안 되는 말을 더 분명히 나누고, 도구의 역할에 맞는 응답만 통과시키는 방향입니다. 결국 중요한 건 “똑똑해 보이는 말”이 아니라 “현재 작업에 맞는 말”이라는 걸 다시 확인한 작업이었습니다.

## 리뷰와 로컬라이즈 흐름: 실제 코드를 보게 하고, 취소는 존중하기

마지막으로 재검수 과정에서 나온 P3 항목들을 정리했습니다. 핵심은 두 가지였습니다.

첫째, 그린필드 리뷰가 실제 코드를 제대로 보도록 했습니다. 새로 만드는 프로젝트나 아직 맥락이 적은 작업에서는 모델이 구조를 상상으로 채우기 쉽습니다. 하지만 리뷰는 추측이 아니라 관찰에 기반해야 합니다. 그래서 리뷰 단계에서 코드 확인이 더 잘 이뤄지도록 흐름을 보강했습니다.

둘째, `localize` 작업에서 취소를 더 확실히 존중하도록 했습니다. 사용자가 취소했는데 내부 흐름이 계속 이어지거나, 이미 멈췄어야 할 작업이 뒤늦게 영향을 주면 굉장히 불쾌한 경험이 됩니다. 자동화 도구일수록 “멈춰”라는 신호를 정확히 받아들이는 게 중요합니다. 이번에는 그 부분을 더 안전하게 정리했습니다.

## 마무리

이번 세 커밋은 화려한 기능 추가보다는 제품의 기본기를 다지는 작업이었습니다. 터미널은 더 정확히 감지하고, 오퍼레이터는 자기 역할을 벗어나지 않게 하고, 리뷰와 취소 흐름은 사용자의 의도에 더 충실하게 만들었습니다.

요즘 느끼는 건, 개발 도구의 품질은 큰 기능보다 이런 작은 순간에서 더 자주 드러난다는 점입니다. 화면이 제대로 뜨는지, 말이 어색하지 않은지, 취소했을 때 정말 멈추는지. 이런 부분들이 쌓여야 사용자가 안심하고 도구를 맡길 수 있다고 생각합니다.
