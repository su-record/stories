# tory 개발일지 #21 - 채팅 리뉴얼 + dead code 청소 (10개 커밋)

**작업 기간**: 2026-03-14

## 📝 이번 기간 작업 내용

### 채팅 UI 전면 재작성 (2개 커밋)

채팅 UI를 SSE 스트리밍과 마크다운 렌더링이 동작하는 구조로 전면 재작성했다. Claude 스타일의 UI를 참고해 메시지 버블 레이아웃, 코드 블록 하이라이팅, 인라인 마크다운을 적용했다. 기존 폴링 방식을 제거하고 SSE로 실시간 스트리밍 응답을 받는다.

CLI 스트리밍도 함께 개선했다. 실시간 출력이 제대로 흘러가도록 수정하고, 타이핑 인디케이터가 중복 표시되는 버그를 제거했다.

| 커밋 | 내용 |
|------|------|
| `feat: 채팅 전체 리뉴얼 — SSE 스트리밍, 마크다운 렌더링, 클로드 스타일 UI` | 채팅 UI 전면 재작성 |
| `fix: CLI 스트리밍 실시간 출력 + 타이핑 인디케이터 중복 제거` | CLI 스트리밍 수정 |

### dead code 제거 (1개 커밋)

데스크톱 앱에서 1,249줄의 dead code를 제거하고 중복 로직을 통합했다. 사용하지 않는 컴포넌트, 미사용 타입, 중복된 상태 관리 코드가 대상이었다. 코드베이스가 가벼워지고 후속 작업이 빨라졌다.

| 커밋 | 내용 |
|------|------|
| `refactor: 데스크톱 앱 dead code 제거 + 중복 로직 통합 (-1,249줄)` | 1,249줄 제거 |

### 모델 선택 영속화 + 회의 시스템 수정 (2개 커밋)

모델 선택이 실제 LLM 호출에 반영되지 않는 버그를 수정했다. 데몬 DB에 선택된 모델을 영속화해 재시작 후에도 유지되도록 했다. 회의 시스템에서는 토큰 추적, 투표 편향 수정, 타임아웃 처리, 프로필 모델명 처리를 개선했다.

| 커밋 | 내용 |
|------|------|
| `fix: 모델 선택이 실제 LLM 호출에 반영되도록 데몬 DB 영속화` | 모델 선택 영속화 |
| `fix: 회의 시스템 개선 — 토큰 추적, 투표 편향 수정, 타임아웃, 프로필 모델명` | 회의 시스템 버그 수정 |

### 온톨로지 메인 파이프라인 연결 + RAG 버그 수정 (2개 커밋)

온톨로지 시스템을 메인 파이프라인에 연결했다. 회의, 토론, 첨부파일이 처리될 때 자동으로 온톨로지 태그가 추출되어 RAG 인덱스에 반영된다. 별도 설정 없이 대화 맥락이 자동으로 구조화된다.

RAG 버그도 수정했다. 온톨로지 태그 추출 오류와 세션 recencyScore 계산의 안전성 문제를 해결했다.

| 커밋 | 내용 |
|------|------|
| `fix: RAG 버그 수정 — 온톨로지 태그 추출 오류 + 세션 recencyScore 안전성` | RAG 버그 수정 |
| `feat: 온톨로지 시스템 메인 파이프라인 연결 — 회의/토론/첨부파일 자동 태깅` | 온톨로지 자동 태깅 |

### 채팅-Secretary 파이프라인 연결 + 린트 (2개 커밋)

채팅 입력이 Secretary를 통해 회의 파이프라인으로 흐르도록 연결했다. SSE 스트리밍 안정화도 함께 진행했다. Biome 자동 정렬과 미사용 변수 정리로 린트와 타입체크를 전체 통과시켰다.

| 커밋 | 내용 |
|------|------|
| `chore: 린트 + 타입체크 전체 수정 — biome 자동 정렬 + 미사용 변수 정리` | 린트 및 타입체크 통과 |
| `feat: 채팅 → Secretary 회의 파이프라인 연결 + SSE 스트리밍 안정화` | 채팅-회의 파이프라인 연결 |

## 💡 작업 하이라이트

**하루에 1,249줄을 지우다**

추가만 하는 날도 있지만 이 날은 지우는 것도 함께 했다. 1,249줄 제거는 단순히 파일 크기를 줄이는 것이 아니다. 무엇이 실제로 사용되는지, 어느 로직이 중복인지를 파악해야 가능한 작업이다. 채팅 UI를 전면 재작성하는 과정에서 기존 코드의 구조를 다시 이해하고, 불필요한 것을 과감하게 걷어냈다.

**온톨로지가 파이프라인에 자리를 잡다**

온톨로지 태그 자동 추출이 메인 파이프라인에 연결됐다. 사용자가 회의를 시작하거나 파일을 첨부하면 내용이 자동으로 구조화된다. RAG 검색이 이 태그를 활용해 더 정확한 문서를 찾아낸다. 각 컴포넌트가 독립적으로 존재하다가 연결되는 순간이다.
