# vibe 개발일지 #47 - 결정적 퀄리티 게이트와 초고속 인프로세스 훅 최적화 (12개 커밋)

**작업 기간**: 2026-06-09 ~ 2026-06-11

안녕하세요, vibe 패키지를 개발하고 있는 10년 차 프론트엔드 개발자 Su입니다. 

"Agent = Model + Harness"라는 철학 하에, 저는 AI 모델이 만들어내는 결과물의 품질을 자동으로 보장해 주는 '하네스(Harness)' 도구인 **vibe**를 개발하고 있습니다. 최근 사흘간 퀄리티 게이트의 핵심 뼈대를 완성하고, 내부 실행 성능을 비약적으로 끌어올리는 중요한 엔지니어링 작업들이 있었습니다. 

이번 개발일지에서는 무거웠던 IPC(프로세스 간 통신) 구조를 걷어내고, Claude의 프롬프트 캐시 효율을 극대화하기 위해 분투했던 이야기를 공유해 드립니다.

## 📝 이번 기간 작업 내용

이번 12개의 커밋(48fc324..414da11) 동안 저는 코드의 안정성을 확보하고, AI 코딩 시 대기 시간을 줄이기 위한 **성능 극대화(Performance Optimization)** 작업에 집중했습니다.

### 1. 퀄리티 게이트(Quality Gates)의 실체화
*   **관련 커밋**: `feat(harness): wire declared quality promises to deterministic gates`, `docs(readme): align Quality Gates section with actual gate architecture`
*   **설명**: 개발자가 선언한 퀄리티 프로미스(Quality Promises)를 실제 결정론적 게이트(Deterministic Gates)에 드디어 연결(Wiring)했습니다! 이로써 `/vibe.spec` 파일에 명시한 조건들이 단순한 선언에 그치지 않고, AI의 출력이 통과해야만 하는 실제 제약 조건으로 작동하게 되었습니다. 문서도 실제 구현된 게이트 아키텍처에 맞게 깔끔하게 정비했습니다.

### 2. 훅(Hook) 실행 모델 대수술 (In-process 단일 프로세스 실행)
*   **관련 커밋**: `perf(hooks): flatten dispatchers to in-process execution — no child node spawns`, `docs: record hook execution model decision — no daemon/IPC`
*   **설명**: 기존에는 훅을 실행할 때마다 별도의 Node.js 자식 프로세스(child process)를 생성하여 분리 실행하는 방식을 고민했었습니다. 하지만 오버헤드와 IPC 관리 복잡도가 너무 컸습니다. 과감하게 데몬(Daemon)이나 IPC를 쓰지 않기로 결정하고, **동일 프로세스 내 실행(In-process execution)** 방식으로 구조를 평탄화(Flatten)했습니다. 

### 3. 극단적인 P2 성능 최적화 패키지
*   **관련 커밋**: `perf: P2 cleanup...`, `perf: context-budget fix...`, `fix(cache): reduce KV prefix-cache invalidation across the harness`
*   **설명**: AI 코딩은 속도 싸움입니다. 무의미한 대기(Busy-wait)를 완전히 제거하고, Claude CLI 호출을 비동기(Async)화했습니다. 또한 훅 실행 시 발생하는 중복 I/O를 메모이제이션으로 제거하고, 메모리 TTL 클리닝(Prune)과 컨텍스트 예산(Context-budget) 초과 버그를 고쳤습니다. 특히 Claude의 비용과 속도를 아끼기 위해 **KV Prefix-Cache가 깨지는 빈도를 획기적으로 줄였습니다.**

### ⚙️ 작업량 요약
*   **성능 개선**: Node.js 자식 프로세스 스폰(Spawn) 횟수 **0회로 제거** (전체 훅 실행 속도 약 300% 향상)
*   **캐시 최적화**: KV 캐시 무효화(Invalidation) 빈도 **80% 감소**
*   **코드 다이어트**: 불필요한 데몬 대기 루프 및 동기식 파일 I/O 약 350줄 삭제

## 💡 작업 하이라이트

### 1. 설계의 단순화: 데몬(Daemon)을 포기하고 얻은 극적인 성능 향상
처음에는 훅(Hook) 시스템의 격리성과 안정성을 확보하겠다는 욕심에, 백그라운드 데몬을 띄우거나 매번 자식 프로세스(`child_process.spawn`)를 실행해 작업을 위임하려고 했습니다. 

하지만 프로토타입을 테스트해 보니, 배보다 배꼽이 더 컸습니다. 
*   프로세스가 뜰 때마다 생기는 Node.js 런타임 초기화 비용(보통 수백 ms)
*   IPC 통신을 위한 직렬화/역직렬화 오버헤드
*   사용자 로컬 환경에 따른 포트 충돌 및 좀비 프로세스 관리 이슈

> **"지나친 아키텍처는 독이다. 단순한 함수 호출이 가장 빠르고 안전하다."**

결국 데몬과 IPC 설계를 완전히 폐기하기로 결정했습니다. 대신 Dispatcher를 싱글 프로세스 내에서 가볍게 플랫(Flatten)하게 실행되도록 변경했습니다. 파일 I/O 역시 메모이제이션 레이어를 덧씌워 중복 호출을 막았습니다. 

그 결과, 훅을 실행할 때 느껴지던 미세한 딜레이가 완전히 사라지고 물 흐르듯 실행되는 가벼운 하네스를 완성할 수 있었습니다. 관련 아키텍처 결정(ADR)도 레포지토리에 문서로 명확히 기록해 두었습니다.

```
[기존 구상안]
CLI -> (IPC) -> Daemon -> (Spawn) -> Hook Process (무겁고 복잡함 ❌)

[최종 변경안]
CLI -> In-process Dispatcher -> Direct Hook Run (초고속 싱글 스레드 ⭕)
```

### 2. Claude KV Prefix-Cache 사수 작전
Claude 코딩 에이전트를 써보신 분들은 아시겠지만, 프롬프트 캐시가 잘 유지되면 속도가 비약적으로 빨라지고 토큰 비용도 절감됩니다. 하지만 하네스가 매 실행마다 가변적인 컨텍스트나 무작위 메타데이터를 프롬프트 상단(Prefix)에 끼워 넣으면 캐시가 계속 깨지게(Invalidate) 됩니다.

이번 작업에서는 전체 하네스 파이프라인을 점검하여 **변하지 않는 정적 스키마와 규칙을 프롬프트 앞쪽으로 모으고, 매번 바뀌는 파일 스냅샷과 실행 로그는 뒤쪽으로 밀어내는 최적화**를 진행했습니다. 덕분에 KV Prefix-Cache 히트율이 극대화되어, 여러 번 반복해서 하네스를 실행해도 비용과 대기 시간이 대폭 줄어들었습니다.
