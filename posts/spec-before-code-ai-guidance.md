AI가 코드를 못 짜서 문제가 생긴 건 아니었습니다. 빈칸을 너무 많이 남긴 채 맡겼습니다.

---

## SPEC 없이 만든 코드는 빨리 보이고, 나중에 느려집니다

![SPEC은 긴 문서가 아니라 AI가 넘지 않을 구현 경계선이다](/images/spec-before-code-ai-guidance/01.png)
*경계 먼저*


6월에 작은 온보딩 화면을 만들었습니다. 처음에는 Claude Code에 바로 지시했습니다.

`신규 사용자가 가입 후 체크리스트를 볼 수 있게 만들어줘.`

결과물은 얼핏 괜찮았습니다. 컴포넌트가 생겼고, API도 붙었습니다. 문제는 그 다음에 나왔습니다.

AI는 9개 파일을 수정했습니다. 기존 상태 관리 규칙은 무시됐습니다. `completed_at`을 쓰던 프로젝트에 `completedAt`이 새로 들어갔습니다. 빈 상태 문구도 디자인 시스템 밖에서 만들어졌습니다.

리뷰 코멘트는 14개였습니다. 되돌리는 데 2시간 40분이 걸렸습니다. 기능을 만드는 시간보다 정리하는 시간이 더 길었습니다.

돌아보니 프롬프트가 짧아서 생산성이 오른 게 아니었습니다. 내가 결정해야 할 일을 AI에게 넘긴 셈이었습니다.

| 구분 | SPEC 없이 지시 | SPEC 먼저 작성 |
|---|---:|---:|
| 첫 지시 시간 | 1분 | 18분 |
| 수정 파일 수 | 9개 | 4개 |
| 리뷰 코멘트 | 14개 | 3개 |
| 되돌린 코드 | 5개 파일 | 0개 파일 |
| 총 소요 시간 | 3시간 30분 | 1시간 25분 |

작은 숫자입니다. 그래도 같은 일이 반복됐습니다. 병목은 코드가 아니라 결정 비용이었습니다.

---

## SPEC은 긴 문서가 아닙니다. AI가 흔들리지 않을 경계선입니다

![Non-goals는 AI가 빈칸을 새 기능으로 채우는 흔한 실수를 막는다](/images/spec-before-code-ai-guidance/02.png)
*빈칸 막기*


제가 쓰는 SPEC은 보통 40~80줄입니다. 기획서처럼 꾸미지 않습니다. AI가 헷갈릴 만한 지점만 먼저 고정합니다.

핵심은 6개입니다. 목표, 제외 범위, 데이터 형태, 화면 상태, 수용 기준, 프로젝트 규칙입니다.

```md
# SPEC: 온보딩 체크리스트

## Goal

![안전한 AI 코딩 흐름은 SPEC, 구현 계획, 코드 생성, 수용 기준 점검 순서다](/images/spec-before-code-ai-guidance/03.png)
*점검 순서*

신규 사용자가 가입 후 3개 필수 작업을 확인한다.
체크리스트는 대시보드 상단에만 노출한다.

## Non-goals
이 SPEC에서는 알림 메일을 보내지 않는다.
체크리스트 항목 편집 기능을 만들지 않는다.

## Data
- source: GET /api/onboarding/status
- field: completed_at, snake_case 유지
- null이면 미완료로 처리한다.

## UI states
- loading: skeleton 3줄
- empty: 체크리스트 숨김
- error: 기존 ToastError 컴포넌트 사용

## Acceptance criteria
- 완료 항목은 다시 클릭할 수 없다.
- 3개 항목 완료 시 카드가 사라진다.
- 모바일 360px에서 줄바꿈이 깨지지 않는다.

## Project rules
- 상태 관리는 기존 useOnboardingStore만 사용한다.
- 새 패키지를 추가하지 않는다.
- API 타입은 src/types/api.ts에만 정의한다.
```

이 정도만 적어도 AI가 하는 일이 달라집니다. 새 구조를 만들기보다 기존 규칙 안에서 답을 찾습니다.

SPEC은 AI에게 친절하게 설명해주는 문서가 아닙니다. 사람이 먼저 결정한 내용을 적어두는 문서입니다. 그 차이가 큽니다.

---

## 프로젝트 규칙까지 붙이면 AI는 덜 창의적이 됩니다

AI 코딩에서 가장 위험한 순간은 AI가 창의적으로 움직일 때입니다. 함수 하나면 되는 곳에 새 추상화를 만들고, CSS 한 줄이면 되는 곳에 유틸 클래스를 늘립니다.

그래서 SPEC 옆에 프로젝트 규칙을 둡니다. 파일명은 `docs/specs/2026-07-05-onboarding-checklist.md`처럼 날짜와 기능명을 붙입니다.

그 다음 AI에게 이렇게 지시합니다.

```md
작업 전 docs/specs/2026-07-05-onboarding-checklist.md를 먼저 읽어라.
코드를 쓰기 전에 구현 계획을 5단계로 요약해라.
SPEC 밖의 기능은 만들지 마라.
새 패키지가 필요하면 설치하지 말고 이유만 적어라.
완료 후 acceptance criteria 기준으로 자체 점검해라.
```

Claude Code Superpowers 같은 흐름도 같은 문제를 다룹니다. 코드 작성 전에 기획서와 요구사항 명세를 먼저 만들게 합니다. Claude Code의 스킬, 훅, MCP, 서브에이전트도 결국 맥락을 고정하는 장치입니다.

중요한 건 도구명이 아닙니다. 순서입니다.

Google Gemini Spark 같은 노코드 에이전트도 비슷합니다. 자연어로 태스크, 스킬, 스케줄을 만들 수 있습니다. 하지만 `매일 아침 메일 정리해줘`만으로는 부족합니다.

`Gmail에서 보낸 사람이 고객 도메인이고, 제목에 계약 또는 invoice가 있으면, 오전 8시에 요약하고, 캘린더 일정과 충돌하면 Slack에만 보낸다.`

이 지시가 SPEC입니다. 코드가 없어도 명세는 필요합니다. 24시간 백그라운드에서 도는 에이전트라면 더 그렇습니다.

## 좋은 SPEC은 AI의 답을 줄이지 않고, 실수를 줄입니다

SPEC을 쓰면 AI가 답답해질 것처럼 느껴질 수 있습니다. 실제로는 반대였습니다. 선택지가 줄어들수록 결과가 더 빨리 나왔습니다.

제가 확인하는 기준은 세 가지입니다.

| 기준 | 나쁜 SPEC | 좋은 SPEC |
|---|---|---|
| 목표 | 온보딩 개선 | 가입 후 3개 작업 완료 여부 표시 |
| 제외 범위 | 없음 | 메일, 관리자 편집, 통계 제외 |
| 완료 조건 | 잘 동작 | 360px, 에러, 완료 후 숨김까지 명시 |
| 규칙 | 알아서 | 새 패키지 금지, 기존 store 사용 |

특히 제외 범위가 중요합니다. AI는 빈칸을 기능으로 채우려 합니다. 사람은 빈칸을 나중에 결정할 일로 남겨둡니다. 여기서 충돌이 생깁니다.

그래서 저는 SPEC에 `Non-goals`를 꼭 씁니다. 만들지 않을 것을 적으면 코드가 줄어듭니다. 리뷰도 줄어듭니다.

마지막 점검은 단순합니다.

```md
- 이 SPEC만 읽고 구현 범위를 알 수 있는가?
- 데이터 필드 이름이 실제 코드와 같은가?
- 실패 상태와 빈 상태가 적혀 있는가?
- 만들지 않을 기능이 적혀 있는가?
- 완료 여부를 테스트할 문장이 있는가?
```

다섯 개 중 하나라도 비어 있으면 AI가 추측합니다. 추측이 들어가면 재작업이 생깁니다.

코드 생성은 이제 첫 단계가 아니었습니다. SPEC 작성, 구현 계획, 코드 생성, 수용 기준 점검 순서가 더 안전했습니다.

AI에게 일을 맡긴다는 말은 결정을 넘긴다는 뜻이 아닙니다. 결정은 사람이 하고, 반복 작업을 AI가 맡습니다.
