## vica 개발일지: 팀 모드 감시 루프와 작업 상태 가시성 다듬기

이번 작업은 크게 보면 **팀 모드의 자동 감시 흐름을 실제로 굴러가게 만드는 일**과, 그 과정에서 드러난 **작업 상태 표시·안정성 문제를 정리하는 일**에 가까웠습니다. 기능을 새로 붙이는 작업도 있었지만, 그보다 “이제 사람이 계속 붙잡고 있지 않아도 vica가 상황을 더 잘 이해하고 움직이게 만들자”는 방향성이 더 컸습니다.

### 팀 모드 감시 루프: 모델이 판단하고 코드가 움직이게 하기

가장 큰 변화는 팀 모드 단계 3에 해당하는 **감시 루프**입니다. 기존에는 워커가 자연어로 개입해야 할 상황이 생겨도, 그 판단과 라우팅이 비교적 코드 중심으로 고정되어 있었습니다. 이번에는 모델이 현재 상황을 보고 “워커 개입이 필요한가?”를 판정하고, 코드는 그 결과를 받아 적절한 흐름으로 라우팅하는 방식으로 바꿨습니다.

개인적으로 이 지점이 vica의 성격을 꽤 많이 바꾼다고 느꼈습니다. 단순히 명령을 실행하는 도구라기보다, 여러 작업자와 흐름 사이에서 중간 판단을 맡는 쪽에 가까워졌기 때문입니다. 물론 모델 판단이 들어가면 항상 고민이 생깁니다. 너무 많은 권한을 주면 예측하기 어려워지고, 너무 적게 주면 자동화의 의미가 줄어듭니다. 그래서 이번에는 “판단은 모델이 하되, 실제 이동과 처리 경로는 코드가 책임진다”는 선을 유지하려고 했습니다.

### 현재 브랜치 표시: 지금 어디서 작업 중인지 바로 보기

사용성 측면에서는 인포라인에 **현재 브랜치 표시**를 추가했습니다. 폴더명 옆에 `⎇ 브랜치` 형태로 표시되도록 해서, 지금 어떤 저장소의 어떤 브랜치에서 작업 중인지 바로 볼 수 있게 했습니다.

작은 기능처럼 보이지만, 여러 작업을 동시에 오가다 보면 이런 정보가 정말 중요합니다. 특히 팀 모드처럼 워커가 개입하고, 여러 흐름이 오가는 구조에서는 “내가 지금 보고 있는 컨텍스트가 맞나?”를 빠르게 확인할 수 있어야 합니다. 브랜치 정보는 실시간으로 반영되도록 했기 때문에, 브랜치를 바꾼 뒤 상태가 뒤늦게 따라오는 어색함도 줄였습니다.

이번에 느낀 건, 개발 도구의 신뢰감은 거창한 기능보다 이런 기본 정보의 정확도에서 많이 온다는 점입니다. 지금 위치, 지금 상태, 지금 작업 대상이 분명하면 사용자가 도구를 덜 의심하게 됩니다.

### 코드리뷰 반영: 워커 스코프와 중단 처리 정리

마지막으로 3자 코드리뷰에서 나온 P1/P2 피드백을 반영했습니다. 여기에는 워커 스코프, halt 처리, 고아 상태, 마일스톤, 지문, CLI 실패 처리 같은 항목들이 포함됐습니다.

이 작업은 화려하진 않지만 꽤 중요했습니다. 감시 루프처럼 자동으로 판단하고 움직이는 구조에서는 예외 상황이 훨씬 더 무섭습니다. 워커의 스코프가 애매하면 엉뚱한 작업에 개입할 수 있고, halt가 명확하지 않으면 멈춰야 할 흐름이 계속 이어질 수 있습니다. 고아 상태나 CLI 실패도 마찬가지로, 한 번 방치되면 사용자는 “지금 이게 살아 있는 건가, 죽은 건가?”를 알기 어려워집니다.

그래서 이번 수정은 자동화의 뒷정리에 가깝습니다. 모델 기반 판단을 더 믿고 맡기려면, 실패했을 때의 경계와 정리 방식이 먼저 탄탄해야 한다고 봤습니다. 기능을 앞으로 밀어붙이는 것도 중요하지만, 멈춰야 할 때 제대로 멈추고, 실패했을 때 사용자가 이해할 수 있게 남기는 것도 같은 무게로 다뤄야 한다는 생각이 들었습니다.

### 마무리

이번 변경들을 묶어보면 vica는 조금 더 “팀 작업을 지켜보고 조율하는 도구”에 가까워졌습니다. 모델은 자연어 상황 판단을 맡고, 코드는 실행 경로와 안전장치를 관리합니다. 여기에 브랜치 표시처럼 현재 컨텍스트를 보여주는 요소를 더해, 사용자가 흐름을 놓치지 않도록 했습니다.

아직 다듬을 부분은 많습니다. 특히 감시 루프는 실제 사용 시나리오가 쌓여야 더 안정적으로 튜닝할 수 있을 것 같습니다. 그래도 이번 작업으로 자동 개입, 상태 표시, 실패 처리의 기본 골격이 한 단계 단단해졌다는 점에서 꽤 만족스러운 진전이었습니다.
