## vica 개발일지: 스킬을 더 잘 보이게, 더 안전하게 쓰게 만들기

이번 작업은 크게 보면 “vica가 할 수 있는 일을 사용자에게 더 명확히 보여주고, 사용자가 허용한 범위 안에서만 실행하게 만들자”는 방향이었습니다. 기존에는 내장 스킬이 있어도 사용자가 그 존재를 자연스럽게 발견하기 어렵고, 어떤 스킬이 실제로 활성화되어 있는지도 흐릿한 편이었습니다. 여기에 업데이트 안내도 “새 버전이 있어요” 수준에 머물러 있어서, 실제 설치까지 이어지는 흐름이 조금 끊겨 있었습니다.

### 내장 스킬을 명령어처럼 발견할 수 있게 했습니다

가장 눈에 띄는 변화는 내장 스킬을 명령어로 승격한 부분입니다. 이제 활성화된 스킬은 `/` 자동완성과 `/help`에서 함께 노출됩니다. vica를 쓰는 입장에서는 “무엇을 할 수 있는지”를 따로 문서를 뒤지지 않고도 대화 중에 바로 확인할 수 있게 된 셈입니다.

이전에는 기능이 있어도 사용자가 그 기능명을 알고 있어야 호출할 수 있는 느낌이 강했습니다. 이번 변경으로 스킬이 좀 더 CLI의 하위 명령처럼 보이게 되었고, 사용 가능한 기능 목록도 현재 상태에 맞춰 보여줄 수 있게 됐습니다. 특히 `/help`가 단순 설명서가 아니라 “지금 내 vica가 실제로 쓸 수 있는 도구 목록”에 가까워진 점이 마음에 듭니다.

물론 모든 스킬을 무조건 보여주면 오히려 혼란스러울 수 있어서, 여기서는 활성화 상태가 중요한 기준이 됩니다. 이 부분이 다음 작업과 자연스럽게 이어졌습니다.

### 체크한 스킬만 vica가 사용하도록 정리했습니다

스킬 활성화 체크리스트도 추가했습니다. 이제 사용자는 vica가 사용할 스킬을 직접 고를 수 있고, 체크한 스킬만 실제 실행 대상이 됩니다. 이건 단순한 UI 옵션 추가라기보다, vica의 동작 범위를 사용자에게 돌려주는 작업에 가깝습니다.

AI 도구는 기능이 많아질수록 편리해지지만, 동시에 “얘가 어디까지 할 수 있는 거지?”라는 불안도 커집니다. 그래서 스킬을 명시적으로 켜고 끄는 구조가 필요했습니다. 특히 파일을 다루거나 외부 명령과 연결되는 기능이 늘어날수록, 기본값과 권한 경계는 더 분명해야 한다고 느꼈습니다.

이번 체크리스트 덕분에 자동완성, 도움말, 실제 실행 가능 스킬이 같은 기준을 바라보게 되었습니다. 사용자가 체크하지 않은 스킬은 vica도 사용하지 않고, 따라서 도움말이나 명령어 후보에서도 불필요하게 튀어나오지 않습니다. 작은 정리처럼 보이지만, 앞으로 스킬이 늘어날 때 꽤 중요한 기반이 될 것 같습니다.

### 업데이트 안내를 실제 설치 흐름으로 연결했습니다

마지막으로 실행 시 새 버전을 확인하는 흐름을 개선했습니다. 기존에는 업데이트가 있어도 안내문을 보여주는 데 그쳤습니다. 사용자는 메시지를 보고, 직접 설치 명령을 찾아 실행해야 했습니다. 솔직히 이런 흐름은 알림만 친절하고 행동은 불친절한 상태였습니다.

이제는 새 버전이 감지되면 인터랙티브하게 설치를 선택할 수 있습니다. 실행 시점에 업데이트 여부를 알려주고, 사용자가 원하면 바로 새 버전을 설치하는 흐름으로 이어집니다. CLI 도구에서는 이런 작은 마찰이 누적되면 업데이트가 계속 미뤄지기 쉬운데, 이번 변경으로 최신 버전을 따라가는 부담이 줄어들었습니다.

다만 자동으로 덮어쓰는 방식은 피했습니다. 업데이트는 편해야 하지만, 사용자의 실행 환경을 바꾸는 일이기도 해서 최종 선택은 사용자에게 남겨두는 편이 맞다고 봤습니다. 그래서 “알아서 설치”가 아니라 “물어보고 설치”하는 쪽으로 정리했습니다.

### 마무리

이번 세 커밋은 각각 명령어 노출, 스킬 활성화 제어, 업데이트 설치라는 다른 영역처럼 보이지만, 공통적으로는 vica의 사용 경험을 더 예측 가능하게 만드는 작업이었습니다. 무엇을 할 수 있는지 잘 보이고, 무엇을 해도 되는지 사용자가 정하고, 새 버전으로 넘어가는 길도 끊기지 않게 만들었습니다.

다음 단계에서는 이 스킬 구조 위에서 더 많은 기능을 얹더라도 복잡해 보이지 않게 정리하는 일이 중요할 것 같습니다. 기능을 추가하는 것만큼이나, 사용자가 안심하고 이해할 수 있는 형태로 보여주는 일이 계속 과제가 될 것 같습니다.
