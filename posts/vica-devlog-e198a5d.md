## 검증은 “제품마다 다르다”는 전제에서 다시 보기

이번 작업에서 가장 크게 방향을 잡은 부분은 검증 로직입니다. 기존에는 검증 과정을 비교적 하나의 흐름으로 바라보는 성격이 강했는데, 실제로 제품 단위로 내려가 보면 검증의 모양이 꽤 다릅니다. 어떤 제품은 특정 태스크가 중요하고, 어떤 제품은 단계별 비중이나 확인 방식이 다르게 잡혀야 합니다.

그래서 이번에는 검증을 제품별로 더 유연하게 적용할 수 있도록 “태스크-shaped” 구조로 다듬었습니다. 말 그대로 검증 기준을 하나의 고정된 체크리스트처럼 두기보다는, 각 제품의 작업 형태에 맞춰 비례적으로 검증할 수 있게 만든 셈입니다.

이 작업을 하면서 느낀 점은, 검증 로직은 단순히 “맞다/틀리다”를 판단하는 코드가 아니라는 점입니다. 제품의 맥락을 얼마나 잘 반영하느냐가 품질에 직접 영향을 주더라고요. 앞으로 제품별 검증 공정이 늘어나도 억지로 하나의 틀에 끼워 맞추지 않고, 필요한 만큼만 자연스럽게 확장할 수 있는 기반을 마련한 것이 이번 변경의 핵심입니다.

## 프로덕션 실행 흐름 분리하기

두 번째로는 서버 실행 쪽을 정리했습니다. 프로덕션 러너 역할을 별도 파일로 분리하면서, 기존에 한 파일에 몰려 있던 책임을 조금 덜어냈습니다. 줄 수로만 보면 1,254줄에서 1,141줄로 줄어든 정도라 아주 극적인 숫자는 아니지만, 체감상으로는 꽤 의미 있는 분리였습니다.

특히 운영 환경에서 실행되는 흐름은 실수했을 때 영향 범위가 크기 때문에, 관련 코드가 어디에 있고 어떤 책임을 갖는지가 분명해야 합니다. 이번에는 `channels/run.ts` 쪽으로 프로덕션 러너를 옮기면서, “실행을 준비하는 코드”와 “실제로 채널을 구동하는 코드”의 경계를 조금 더 명확히 했습니다.

솔직히 이런 리팩터링은 겉으로 보이는 기능 변화가 크지 않아서 티가 덜 납니다. 하지만 나중에 장애를 보거나 실행 옵션을 손볼 때, 이런 분리가 되어 있느냐 아니냐가 유지보수 난이도를 크게 가릅니다. 지금 당장 눈에 띄는 성과보다, 다음 작업자가 덜 헤매게 만드는 쪽에 가까운 작업이었습니다.

## 세션 god object 쪼개기

마지막으로 가장 속이 시원했던 작업은 세션 관련 객체 정리입니다. 기존 세션 코드는 1,356줄짜리 거대한 객체에 가까웠습니다. 여러 책임이 한곳에 모여 있다 보니, 작은 수정 하나를 하더라도 주변 맥락을 계속 추적해야 했고, 읽는 사람 입장에서도 부담이 컸습니다.

이번에는 이 덩어리를 책임별로 나누면서 919줄 규모로 줄였습니다. 단순히 줄 수를 줄이는 것이 목표는 아니었고, “이 코드가 무엇을 책임지는가”를 기준으로 경계를 다시 잡는 데 초점을 뒀습니다. 세션 상태 관리, 실행 흐름, 외부와의 연결 지점처럼 성격이 다른 부분들이 한 클래스나 객체 안에서 뒤섞이지 않도록 정리했습니다.

god object를 분해할 때는 항상 조심스럽습니다. 너무 잘게 나누면 오히려 흐름이 흩어지고, 너무 적게 나누면 리팩터링의 효과가 약해집니다. 이번에는 일단 가장 통증이 큰 지점부터 걷어내는 방식으로 접근했습니다. 완벽하게 끝났다고 보기는 어렵지만, 이제는 세션 코드를 열었을 때 “어디부터 봐야 하지?”라는 막막함이 꽤 줄었습니다.

## 정리

이번 작업들은 화려한 기능 추가라기보다는, 앞으로의 변경을 견디기 위한 구조 정리에 가까웠습니다. 제품별 검증을 더 현실적으로 다루고, 프로덕션 실행 흐름을 분리하고, 세션의 거대한 책임을 나누면서 코드베이스가 조금 더 숨 쉴 공간을 얻었습니다.

당장 사용자에게 보이는 화면이 바뀌는 작업은 아니지만, 이런 기반 작업이 쌓여야 다음 기능을 더 안전하게 올릴 수 있다고 생각합니다. 특히 `vica`처럼 실행 흐름과 검증 로직이 함께 얽히는 프로젝트에서는, 구조를 미리 정리해두는 일이 결국 기능 개발 속도에도 영향을 주는 것 같습니다.
