## 개발일지: 캐시를 “잘 타는” 구조로 다듬기

이번 작업은 눈에 확 띄는 기능 추가보다는, `vica`가 반복 작업을 할 때 불필요하게 같은 내용을 다시 읽고 다시 말하지 않도록 만드는 성능 개선에 가까웠습니다. 최근 흐름을 정리해보면 핵심은 하나였습니다. **변하지 않은 것은 다시 처리하지 말고, 바뀐 것만 모델 앞단에서 더 영리하게 다루자**는 방향입니다.

## 파일 읽기 비용 줄이기

먼저 손본 부분은 `read` 쪽 메모 전략입니다. 기존에는 파일이 실제로 바뀌지 않았더라도 흐름상 다시 읽는 일이 생길 수 있었는데요. 이번에는 안 바뀐 파일에 대해서는 재읽기 대신 스텁을 활용하는 방향으로 정리했습니다.

이건 단순히 “파일 I/O를 조금 아꼈다” 정도의 의미만은 아닙니다. `vica`처럼 모델 앞단에서 컨텍스트를 구성하는 도구에서는, 같은 파일 내용을 반복해서 밀어 넣는 순간 비용이 꽤 빠르게 쌓입니다. 그래서 변경 여부를 기준으로 읽기 결과를 재활용할 수 있게 만들면, 이후 단계에서 모델이 받아보는 입력도 더 안정적으로 관리할 수 있습니다.

물론 캐시는 언제나 조심스럽습니다. 잘못하면 오래된 내용을 붙잡고 있는 문제가 생기니까요. 그래서 이번 작업은 “무조건 캐시”가 아니라, 변경되지 않았다는 판단이 가능한 경우에 한해 재읽기를 생략하는 쪽에 가깝습니다. 성능은 챙기되, 결과의 신뢰도를 해치지 않는 선을 찾으려 했습니다.

## 워커 입력 구조 재배치

두 번째로는 워커에 전달되는 프리픽스 레이아웃을 손봤습니다. 공통 규칙은 앞쪽에 두고, 각 워커마다 달라지는 고유 태스크는 뒤쪽에 놓는 방식입니다.

이 변경은 KV 캐시 관점에서 중요했습니다. 여러 워커가 비슷한 규칙과 지침을 공유한다면, 그 공통 부분은 가능한 한 같은 prefix로 유지되는 편이 좋습니다. 그래야 모델 입장에서 앞부분을 재사용하기 쉬워지고, 뒤쪽의 태스크 차이만 새로 처리하면 되는 구조가 됩니다.

사실 이런 류의 최적화는 코드만 보면 조금 심심해 보일 수 있습니다. 하지만 실제 운영에서는 꽤 체감이 납니다. 특히 여러 작업을 나눠 돌리거나, 비슷한 지시를 반복해서 처리하는 상황에서는 입력의 배치만 바꿔도 캐시 효율이 달라집니다. 이번에는 그 전제를 코드 구조에 더 명확하게 반영했습니다.

## “암송”을 줄이고, 원칙을 문서화하기

마지막으로 감사 과정에서 나온 KV 캐시 친화성 관련 피드백을 반영했습니다. 대표적으로 TODO 같은 반복 정보는 매번 길게 다시 말하지 않고, 변경이 있을 때만 언급하도록 방향을 잡았습니다.

개발 도구가 친절해지려다 보면, 이미 알고 있는 내용을 계속 반복해서 설명하는 경우가 생깁니다. 처음에는 도움이 되지만, 반복 작업에서는 오히려 노이즈가 됩니다. 특히 모델 컨텍스트 안에서는 그런 반복이 그대로 토큰 비용이 되고, 캐시 효율도 떨어뜨립니다.

그래서 이번에는 “무엇을 매번 말하고, 무엇은 바뀔 때만 말할지”에 대한 원칙을 더 분명히 했습니다. 단순한 TODO 정리 규칙을 넘어서, 앞으로 `vica`에서 프롬프트와 작업 컨텍스트를 설계할 때 지켜야 할 캐시 독트린을 명문화한 셈입니다.

## 마무리

이번 세 커밋은 모두 같은 방향을 보고 있었습니다. 파일을 다시 읽는 비용을 줄이고, 워커들이 공유하는 입력을 캐시하기 좋은 형태로 맞추고, 반복 설명을 변경 중심으로 바꾸는 작업이었습니다.

아주 화려한 변화는 아니지만, 이런 정리는 프로젝트가 커질수록 중요해진다고 느낍니다. 기능이 많아질수록 결국 발목을 잡는 건 “한 번 더 해도 되겠지” 하고 지나친 반복들이니까요. 이번 작업으로 `vica`는 조금 더 덜 말하고, 덜 읽고, 필요한 순간에만 제대로 반응하는 쪽에 가까워졌습니다.
