버그를 못 고친 게 아니었습니다. 애초에 버그를 제대로 본 적이 없었습니다.

예전에 알림이 중복 발송되는 문제를 로그만 보고 고친 적이 있습니다. 원인은 debounce라고 봤습니다.

코드 4줄을 바꾸고 배포까지 했습니다. 이틀 뒤 같은 버그가 다시 올라왔습니다.

문제는 debounce가 아니었습니다. 재시도 큐와 타임존 경계가 겹치는 케이스였습니다.

---

## 재현 없이 고치면 수정이 아니라 추측입니다

![재현 없이 고치면 수정이 아니라 추측이다](/images/reproduce-before-fixing-bugs/01.png)
*먼저재현*


버그는 단순한 증상이 아닙니다. 특정 조건에서 기대값과 실제값이 달라지는 실패입니다.

그 조건을 모르면 고칠 수 없습니다. 증상이 우연히 사라졌다고 해서 원인을 제거했다고 말하기도 어렵습니다.

Paul Butcher의 『Debug It!』에는 ‘Reproduce First, Ask Questions Later’라는 원칙이 나옵니다. 일단 재현부터 하라는 뜻입니다.

Stanford CS107 디버깅 가이드도 비슷한 순서를 말합니다. 먼저 관찰하고, 가장 작고 단순한 실패 입력을 만들라고 합니다.

| 접근 | 첫 행동 | 결과 |
|---|---|---|
| 추측 수정 | 로그에서 의심 코드부터 변경 | 같은 증상이 다른 조건에서 재발 |
| 재현 우선 | 실패 입력과 환경을 고정 | 수정 전 실패, 수정 후 성공을 확인 |
| 최소 사례 | 관련 없는 설정 제거 | 원인 후보가 10개에서 2개로 감소 |
| 회귀 테스트 | 실패를 테스트로 저장 | 같은 버그가 CI에서 바로 차단 |

재현은 번거로운 부가 작업이 아닙니다. 디버깅을 시작할 기준점에 가깝습니다.

기준점이 없으면 대화도 흐려집니다. “안 됩니다”와 “이 입력에서 100번 중 17번 실패합니다”는 완전히 다른 말입니다.

---

## 재현은 실패를 작게 만드는 일입니다

![실패 입력과 환경을 고정해 최소 사례로 줄인다](/images/reproduce-before-fixing-bugs/02.png)
*작게고정*


처음부터 원인을 찾으려 하면 범위가 너무 넓어집니다. 서버, 브라우저, 캐시, 네트워크가 한꺼번에 의심됩니다.

먼저 실패를 고정해야 합니다. 고정된 실패는 작은 실험으로 나눌 수 있습니다.

제가 보통 남기는 기록은 네 가지입니다. 증상, 입력, 환경, 기대값입니다.

| 항목 | 기록 예시 | 목적 |
|---|---|---|
| 증상 | 결제 완료 알림이 2번 발송됨 | 실제 문제가 무엇인지 고정 |
| 입력 | orderId 48122, retryCount 1 | 같은 요청을 다시 실행 |
| 환경 | Asia/Seoul, 2026-07-12 00:01 | 시간과 지역 조건 고정 |
| 기대값 | 알림은 주문당 1번만 발송 | 성공 기준을 숫자로 표현 |

여기서 꼭 봐야 할 값은 재현률입니다. 한 번 성공한 재현만으로는 아직 부족합니다.

100번 실행해서 17번 실패한다면 비결정성 버그입니다. 타이머, 병렬 처리, 외부 API를 의심해야 합니다.

100번 실행해서 100번 실패한다면 좋은 출발점입니다. 이제 수정 전후를 비교할 수 있습니다.

---

## 최소 재현 사례는 디버깅 시간을 줄입니다

![수정 전 실패와 수정 후 성공을 같은 테스트로 확인한다](/images/reproduce-before-fixing-bugs/03.png)
*빨강초록*


KERN-IT 디버깅 가이드도 첫 단계를 재현 가능한 최소 사례로 둡니다. 범위를 줄이라는 뜻입니다.

아래는 날짜 경계 버그를 테스트로 고정한 예시입니다. 실제 서비스에서는 자정 직후에만 실패했습니다.

```ts
import { expect, test, vi } from 'vitest';
import { dueLabel } from './dueLabel';

test('서울 자정 직후 마감일을 지난 상태로 표시한다', () => {
  vi.setSystemTime(new Date('2026-07-11T15:01:00.000Z'));

  expect(dueLabel('2026-07-12')).toBe('overdue');
});
```

이 테스트는 한국 시간 2026년 7월 12일 00시 01분을 고정합니다. UTC로는 전날 15시 01분입니다.

버그는 여기서 나왔습니다. 서버는 UTC 날짜를 잘라 썼고, 화면은 한국 날짜를 기준으로 판단했습니다.

수정 자체는 작았습니다. 하지만 재현 코드가 없었다면 이 1줄이 맞는지 확인하기 어려웠을 겁니다.

```diff
- const today = new Date().toISOString().slice(0, 10);
+ const today = formatInTimeZone(new Date(), 'Asia/Seoul', 'yyyy-MM-dd');
```

중요한 건 패치 크기가 아닙니다. 수정 전에는 실패하고, 수정 후에는 통과하는 입력이 있는지입니다.

그 입력이 없으면 리뷰도 감으로 흘러갑니다. “괜찮아 보입니다”는 검증이 아닙니다.

## 고친 뒤에는 실패를 테스트 이름으로 남깁니다

Developer Toolkit과 Codex류 디버깅 절차도 같은 흐름을 씁니다. 로컬 재현, 원인 조사, 회귀 테스트 추가입니다.

저는 버그 티켓을 닫기 전에 아래 순서로 확인합니다. 하나라도 빠뜨리면 같은 문제가 다시 돌아온 적이 많았습니다.

| 순서 | 질문 | 통과 기준 |
|---|---|---|
| 1 | 실패를 로컬에서 재현했는가 | 명령어 1개로 실패 확인 |
| 2 | 입력을 최소화했는가 | 계정, 네트워크, UI 의존성 제거 |
| 3 | 수정 전 테스트가 실패했는가 | 빨간 테스트를 먼저 확인 |
| 4 | 수정 후 테스트가 통과했는가 | 같은 입력으로 초록 테스트 확인 |
| 5 | 재발 위치가 막혔는가 | CI 또는 배포 전 검증에 포함 |

명령어도 남겨야 합니다. 사람 기억보다 실행 가능한 기록이 더 믿을 만합니다.

```bash
npm test -- dueLabel.test.ts
for i in {1..100}; do npm test -- dueLabel.test.ts || break; done
```

첫 줄은 기능 검증입니다. 둘째 줄은 흔들리는 실패를 찾기 위한 반복 실행입니다.

버그 수정 PR에는 설명보다 증거가 먼저 있어야 합니다. 실패 입력, 실패 로그, 통과 테스트가 같이 있어야 합니다.

재현하지 않은 수정이 우연히 맞을 때도 있습니다. 다만 우연은 회귀 테스트로 남지 않습니다.

버그 보고서는 닫히고, 재현 사례는 테스트 이름에 남습니다.
