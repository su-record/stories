# SpecKit으로 완성하다

> 이전 편: "AI 개발 방법론 2 - 일관성 문제를 해결하다"

새로운 문서 작성 방식을 적용하고, 백엔드 개발이 어느 정도 진행되던 중이었습니다.

AI와의 대화 중에 우연히 SpecKit이라는 것을 알게 됐습니다.

## SpecKit을 발견하다

> "기획 문서를 좀 더 체계적으로 관리할 방법이 없을까?"

이런 고민을 AI에게 던졌더니, SpecKit을 추천받았다.

**SpecKit이란?**
- AI-First 개발을 위한 문서 프레임워크
- 기획서, 엔티티, API, 비즈니스 로직을 체계적으로 관리
- AI가 읽기 쉬운 구조화된 템플릿 제공

처음엔 "또 새로운 걸 배워야 하나?" 싶었다.

하지만 SpecKit 문서를 읽어보니, 내가 지금까지 해왔던 방식과 거의 비슷했다.

다만, 훨씬 더 체계적이고 완성도가 높았다.

## 기존 문서를 SpecKit에 마이그레이션

기존에 작성했던 문서들을 SpecKit 템플릿에 맞춰 다시 정리했다.

그런데 놀라운 발견을 했다.

### 유실된 내용이 있었다

기획 문서에 **빠진 부분**이 있었다.

분명 수시로 업데이트했다고 생각했는데, 막상 SpecKit 템플릿을 채우려니 빈칸이 보였다.

**예를 들면:**
- 특정 API의 에러 처리 방식
- 엔티티 간 관계에서 예외 케이스
- 비즈니스 로직의 경계 조건

"아, 이래서 일관성이 무너졌구나."

문서에 빠진 부분이 있으니, AI는 세션마다 다르게 추론할 수밖에 없었다.

### 유실된 내용을 채우다

SpecKit 템플릿을 하나하나 채워나가면서, 기획 방향도 다시 점검했다.

"이 기능, 정말 필요한가?"
"이 API 설계, 확장 가능한가?"
"이 비즈니스 로직, 일관성 있는가?"

문서를 다시 정의하고, 이미 작성된 소스코드도 수정했다.

**그렇게 백엔드가 완성됐다.**

## 프론트엔드는 수월하게

백엔드 API와 문서의 완성도가 높아지니, 프론트엔드 개발이 정말 빨라졌다.

AI에게 이렇게 요청하면 됐다:

> "API 문서 읽고, 이 화면 구현해줘."

그러면 AI는:
1. API 문서 읽기
2. 엔티티 구조 파악
3. UI 컴포넌트 생성
4. API 연동 코드 작성

**내 역할은?**
- 틀린 부분 지적
- UI/UX 피드백
- "이건 이렇게 하는 게 더 나을 것 같은데?" 제안

### 기술 지식이 없어도

나는 FastAPI, Flutter를 직접 짤 줄 모른다.

하지만 소스코드 품질은 확인할 수 있다.

**이 정도는 본다:**
- 한 파일에 너무 많은 걸 담고 있지 않은가?
- 중복된 코드가 있지 않은가?
- 재사용 가능한 컴포넌트로 분리했는가?

그리고 AI에게 물어본다:

> "이 코드, 더 나은 방법 있어?"

AI는 항상 개선안을 제시한다. 그걸 검토하고 결정한다.

## 폴링고는 어떻게 개발되는가

### 나의 역할

1. **기획자 (PO/PM)**
   - 기획에 맞는가?
   - 사용자 경험이 좋은가?
   - 이 기능이 정말 필요한가?

2. **검수자 (Reviewer)**
   - AI가 작성한 소스코드를 끊임없이 의심
   - 품질 검토 요청
   - 더 나은 방법 제안 요청

3. **의사결정자**
   - UI/UX 최종 결정
   - 기술 스택 선택
   - 우선순위 결정

### 진정한 AI-First, AI-Driven

**팀 구성:**
- 기획자 = 나
- 개발자 = AI (Claude, Gemini)
- QA = 나 + 와이프
- 디자이너 = AI + 와이프

**개발 프로세스:**
1. 내가 기획 문서 작성 (SpecKit 템플릿)
2. AI가 코드 작성
3. 내가 검수 및 피드백
4. AI가 개선
5. 와이프가 UX 테스트
6. 반복

이게 진정한 AI-First, AI-Driven 개발 아닐까?

## 핵심 원칙 정리

### 1. 문서 중심 개발

**순차적 문서 작성:**
기획서 → 엔티티 → ERD → 데이터모델 → API → 비즈니스 로직

**AI에게 명확한 순서 지시:**
- "기획서부터 읽어"
- "엔티티 정의 확인해"
- "API 설계 기반으로 구현해"

### 2. 일관성 유지

**세션이 바뀌어도 일관성 유지:**
- Claude Projects에 문서 등록
- Gemini Gems에 가이드 등록
- hi-ai MCP로 정보 제공

**교차 검증:**
- DB 스키마 vs 엔티티 정의
- API 응답 vs 데이터모델
- 비즈니스 로직 vs 기획서

### 3. 과감한 리셋

**일관성 없으면 전체 제거하고 다시 시작**

아까워도 독은 제거해야 한다.

처음부터 다시 하는 게 결국 더 빠르다.

### 4. 백엔드 우선

**탄탄한 백엔드가 먼저**

API 완성도가 높으면 프론트엔드는 수월하다.

데이터 모델이 명확하면 UI도 명확해진다.

### 5. 끊임없는 의심과 검증

**AI 결과물을 맹신하지 않기**

- "이게 최선이야?" 질문하기
- 품질 검토 요청
- 더 나은 방법 제안 요청

AI는 틀릴 수 있다. 하지만 질문하면 개선한다.

## SpecKit 덕분에 달라진 것들

### Before SpecKit

- 문서가 흩어져 있음
- 일관성 체크가 어려움
- AI가 문서 일부만 읽음
- 세션마다 다른 결과

### After SpecKit

- 문서가 체계적으로 정리됨
- 일관성 체크가 쉬워짐
- AI가 전체 맥락 파악
- 세션마다 일관된 결과

**가장 큰 변화:**
> "리셋 횟수가 확 줄었다."

문서 품질이 높아지니, AI가 실수할 확률이 낮아졌다.

## 결론: 방법론 완성

기술을 몰라도 AI와 함께라면 앱을 만들 수 있습니다.

**하지만 필요한 것들:**
- AI를 효과적으로 활용하는 방법론
- 문서 중심 개발
- 일관성 유지

이 과정에서 수십 번 문서를 갈아엎었습니다.

포기하려던 순간도 있었습니다.

**하지만 결국 찾았습니다.**

나만의 AI-First 개발 방법론을.

---

**AI-First 개발 핵심 원칙 요약:**

0. **시작**: 기술을 몰라도 시도하라 (AI가 있다)
1. **문서 작성**: 순차적으로, 체계적으로 (PRD → 엔티티 → API)
2. **일관성**: SpecKit 같은 도구로 관리하라
3. **검증**: AI를 맹신하지 말고 끊임없이 질문하라
4. **리셋**: 일관성이 깨지면 과감히 다시 시작하라

**참고:** SpecKit 외에도 [OpenSpec](https://github.com/Fission-AI/OpenSpec) 같은 대안이 있습니다. 중요한 건 도구가 아니라 **일관성**입니다.
