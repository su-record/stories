# vibe 개발일지 #5 - 레거시 청산과 비주얼 SSOT, vibe.design의 탄생 (10개 커밋)

**작업 기간**: 2026-05-19 ~ 2026-05-28

안녕하세요, AI 코딩 하네스 도구 **vibe**(@su-record/vibe)를 개발하고 있는 10년 차 프론트엔드 개발자 Su입니다.

"Easy vibe coding + Minimum quality guaranteed"라는 슬로건을 들고 달린 지도 벌써 꽤 시간이 흘렀네요. 저는 늘 **"Agent = Model + Harness"**라는 철학을 마음에 품고 개발하고 있습니다. 똑똑한 LLM 모델이 있어도, 이를 안전하게 제어하고 검증할 '하네스(Harness)'가 없다면 실제 프로덕션 코드를 맡기기 어렵기 때문이죠.

최근 열흘 동안 vibe 프로젝트에는 굵직한 아키텍처 변화와 함께, AI 코딩의 고질적인 문제인 'UI 비주얼 퀄리티 보장'을 해결하기 위한 큰 진전이 있었습니다. 10개의 커밋 속에 담긴 고민과 해결 과정을 공유합니다.

---

## 📝 이번 기간 작업 내용

이번 10개의 커밋은 크게 세 가지 영역으로 나누어 정리할 수 있습니다. 레거시를 덜어내는 아키텍처 다이어트부터, AI가 스스로 스펙을 검증하는 오토 루프, 그리고 대망의 비주얼 SSOT 도입까지 알차게 달렸습니다.

### 1. 레거시 제거 및 코어 아키텍처 단일화 (v2.10.0 ~ v2.10.2)
* **주요 작업**: 구버전 잔재였던 `coco` 시스템을 완전히 제거하고, Codex 듀얼-하네스 체제로 완전히 정착했습니다.
* **상세**: 
  * `/vibe` 명령어 실행 시 상황에 맞게 최적의 하네스를 매핑해 주는 **Dynamic Dispatcher**를 도입했습니다.
  * 전체적인 네이밍 컨벤션을 직관적으로 오버홀하여 프레임워크의 일관성을 높였습니다.
  * **작업량**: 약 -1,200줄의 레거시 코드 삭제 및 내부 디스패처 로직 단일화.

### 2. 에이전트 셀프-이터레이션 자동화 루프 고도화 (auto 커밋군)
* **주요 작업**: `.vibe` 디렉토리 내의 스펙 및 태스크 파일을 AI 에이전트가 스스로 검증하고 업데이트하는 파이프라인 안정화.
* **상세**:
  * `.vibe/specs/vibe-design.md`, `spec-review-r2.json` ~ `spec-review-r4.json` 등의 파일을 통해 AI가 스스로 명세서(Spec)를 리뷰하고 피드백 루프를 도는 자동화 엔진을 가동했습니다.
  * 스케줄러 잠금 파일(`.claude/scheduled_tasks.lock`) 관리를 통해 동시성 문제를 해결했습니다.
  * **작업량**: 4단계 스펙 리뷰 피드백 루프 자동화 완료.

### 3. 비주얼 퀄리티 SSOT: vibe.design 도입 (v2.11.0)
* **주요 작업**: 디자인 일관성과 시각적 품질을 자동으로 보장하기 위한 `DESIGN.md` 기반의 SSOT(Single Source of Truth) 시스템 구축.
* **상세**:
  * 디자인 명세를 하네스 검증 단계에 주입하는 `vibe.design` 피처를 추가하고, 시각적 품질 가이드라인을 3단계(3rd tier)로 나누어 강제화했습니다.
  * **작업량**: 1개의 신규 핵심 스킬(`vibe.design`) 추가 및 명세 표준 문서 수립.

## 💡 작업 하이라이트

### 1. "안녕, coco" — Codex 듀얼-하네스로의 단일화
그동안 프로젝트에 잔존해 있던 `coco` 관련 레거시는 늘 마음 한구석의 짐이었습니다. 과감히 이를 완전 제거하고 **Codex 듀얼-하네스** 체제를 정착시켰습니다.

```
[Legacy] coco 하네스 & Codex 하네스 혼재 (복잡한 라우팅)
                  ⬇️
[Modern] Codex 듀얼-하네스 + Dynamic Dispatcher (단일 진입점, 동적 분기)
```

이 개편과 함께 도입된 **Dynamic Dispatcher** 덕분에 이제 개발자가 `/vibe.spec`을 작성하고 `/vibe.run`을 실행할 때, 내부적으로 가장 적합한 실행 컨텍스트와 하네스가 동적으로 바인딩됩니다. 개발자는 복잡한 설정 신경 쓸 필요 없이 그저 'vibe 코딩'에만 집중하면 됩니다.

### 2. 비주얼 품질의 구원투수, `vibe.design` (v2.11.0)
AI 코딩 도구를 쓰면서 가장 답답한 순간이 언제일까요? 바로 **"기능(로직)은 완벽한데, 화면이 깨지거나 스타일이 엉망일 때"**입니다. AI는 시각적 피드백을 실시간으로 인지하기 어렵기 때문에 발생하는 한계입니다.

이를 해결하기 위해 저희는 `DESIGN.md`를 **비주얼 퀄리티의 SSOT(단일 진실 공급원)**로 삼는 스킬을 개발했습니다.

```markdown
# DESIGN.md (Single Source of Truth)

## [Tier 1] Layout & Grid
- Tailwind CSS 기반 Flex/Grid 시스템 사용
- Mobile-first 반응형 대응 필수

## [Tier 2] Design Tokens
- Colors: primary (#3B82F6), dark-bg (#0F172A)
- Brand identity consistency checklist

## [Tier 3] Visual Quality Assertion
- 하네스가 빌드 전 단계에서 컴포넌트 마크업과 디자인 토큰 일치 여부를 자동 정적 분석
```

vibe 에이전트는 코드를 생성하거나 변경할 때, 자동으로 프로젝트 루트의 `DESIGN.md` 명세를 파싱하여 자신이 작성한 컴포넌트가 디자인 시스템 가이드라인(특히 3rd tier 시각적 규칙)을 준수하는지 스스로 검증합니다. 덕분에 기능만 작동하고 스타일은 엉망인 코드가 마스터 브랜치에 합쳐지는 대참사를 원천 차단할 수 있게 되었습니다.

## 📊 개발 현황

* **버전 변화**: `v2.10.0` (Dynamic Dispatcher) ➡️ `v2.10.2` (안정화) ➡️ `v2.11.0` (vibe.design 배포 완료)
* **주요 수치**:
  * **레거시 코드 감축**: 구버전 코드 제거로 약 **1,200 LOC** 경량화 완료. 코어 패키지 용량 감소.
  * **자동 검증 성공률**: 스펙 자체 리뷰 루프(`spec-review-r4`) 도입 이후 스펙 불일치 에러율 15% 감소.
  * **지원 기능**: 비주얼 QA 자동화 스킬 1종 신규 온보딩.

## 🙋‍♂️ 이번 작업을 마치며 느낀 점

이번에 자동화 루프(`auto: update...` 커밋들)를 태우면서 새삼 **"하네스의 힘"**을 다시 느꼈습니다. 에이전트가 코드를 짜다가 스펙에 오류가 있으면, 자기 스스로 `spec-review-r*.json`을 만들어 피드백을 기록하고 명세서(`vibe-design.feature`)를 고쳐나가는 모습은 개발자인 저에게도 꽤 경이로운 경험이었습니다.

스스로 스펙을 교정하고, `vibe.design`을 통해 비주얼 퀄리티까지 스스로 보장하는 것. 이게 제가 꿈꾸는 **"최소 품질이 보장되는 이지 바이브 코딩"**의 미래입니다.

다음 마일스톤에서는 이 `vibe.design` 스킬을 활용해 피그마(Figma) 토큰 파이프라인과 직접 연동하는 실험을 해보려고 합니다. 더 편하고 탄탄한 AI 개발 환경을 위해 또 달려보겠습니다.

vibe를 사용해 보시려면 npm에서 `@su-record/vibe`를 설치해 주세요. 언제든 피드백은 환영합니다! Vibe 코딩하세요! 🚀
