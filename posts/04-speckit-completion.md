# SpecKit으로 완성하기

백엔드 개발이 어느 정도 진행되던 중, SpecKit을 알게 됐다.

## 기존 문서를 SpecKit에 마이그레이션

기존 문서를 SpecKit에 맞춰 마이그레이션했다.

놀라운 발견.

기획 문서에 유실된 내용이 있었다.

항상 수시로 변경사항이 발생할 때마다 업데이트했지만, 부족했다.

### 유실된 내용을 채우다

유실된 내용을 채우면서, 기존 기획의 방향도 수정하게 됐다.

문서를 다시 재정의하고, 현재 소스를 수정 개발했다.

그렇게 지금의 백엔드가 완성됐다.

## 프론트엔드는 수월하게

API와 관련 문서의 완성도가 높으니까, 프론트엔드는 수월하게 진행되는 중이다.

나는 틀린 점을 지적해서 수정을 요청하는 것뿐이다.

### 기술 지식이 없어도

기술에 대한 지식이 없어서 수시로 소스코드 품질에 대해 검토 요청한다. 개발 가이드는 반드시 지켜야 할 사항들을 개발 규칙으로 정의했다.

기술 지식이 없어도, 지금까지의 경험으로 이 정도는 본다:

- 한 파일에 너무 많은 걸 담는다
- 중복된 소스코드나 재사용이 안 된다

## 폴링고는 어떻게 개발되는가

### 나의 역할

1. 기획에 맞는가?
2. AI가 작성한 소스코드를 끊임없이 의심하며 요청
3. UI/UX에 대해 고민하고 결정

물론 항상 AI에게 더 나은 방법을 제안받아서 결정하고 있다.

### 진정한 AI-First, AI-Driven

이게 진정한 AI-First, AI-Driven 개발 아닐까?

- 기획자 = 나
- 개발자 = AI
- QA = 나 + 와이프
- 디자이너 = AI + 와이프

## 핵심 원칙 정리

### 1. 문서 중심 개발

순차적 문서 작성: 기획서 → 엔티티 → ERD → API → 비즈니스 로직
AI에게 명확한 순서 지시

### 2. 일관성 유지

세션이 바뀌어도 일관성 유지 (Projects, Gems, MCP)
교차 검증으로 차이점 발견

### 3. 과감한 리셋

일관성 없으면 전체 제거하고 다시 시작
아까워도 독은 제거

### 4. 백엔드 우선

탄탄한 백엔드가 먼저
API 완성도가 높으면 프론트는 수월

### 5. 끊임없는 의심과 검증

AI 결과물을 맹신하지 않기
품질 검토 요청
더 나은 방법 제안 요청

## 결론

기술을 몰라도 AI와 함께라면 앱을 만들 수 있다.

하지만 AI를 효과적으로 활용하는 방법론이 필요하다.

이 과정에서 수십 번 문서를 갈아엎었다. 포기하려던 순간도 있었다.

하지만 결국 찾았다. 나만의 AI-First 개발 방법론을.

이제 여러분 차례다.
