## Vica로 이름을 정리하고, 스킬 경험을 다듬었습니다

이번 작업에서는 레포 전반의 방향성을 조금 더 분명하게 만드는 데 집중했습니다. 가장 눈에 띄는 변화는 제품명을 **Vica**로 정리한 부분입니다. 기존에는 내부적으로 `product`라는 표현이 남아 있는 곳들이 있었는데요. 기능 자체와는 직접 관련이 없어 보여도, 이런 이름은 문서와 코드, 사용자에게 노출되는 메시지 전반에 영향을 줍니다.

그래서 이번에 관련 표현을 Vica 기준으로 맞추면서 “이 도구가 무엇인지”를 더 일관되게 보여줄 수 있도록 정리했습니다. 작은 리팩터링처럼 보이지만, 앞으로 기능이 늘어날수록 이름이 흔들리지 않는 게 중요하다고 생각했습니다.

## 외부 스킬을 더 잘 찾아오도록 했습니다

이번 업데이트의 핵심은 스킬 시스템입니다. Vica가 자체적으로 가진 스킬뿐 아니라, 외부 하니스 환경에서 관리되는 스킬도 인식할 수 있도록 확장했습니다.

이제 `~/.claude/skills`와 `~/.codex/skills` 경로를 발견해서 사용할 수 있습니다. 사용자가 이미 다른 도구나 환경에서 만들어둔 스킬이 있다면, Vica 안에서도 그 자산을 이어서 활용할 수 있게 된 셈입니다.

개인적으로는 이 부분이 꽤 중요하다고 봅니다. 개발 도구가 좋아지려면 사용자를 자기 생태계 안에만 가두기보다, 이미 있는 작업 흐름을 존중해야 한다고 생각합니다. 스킬을 다시 만들라고 강요하는 대신, 있는 것을 발견하고 자연스럽게 연결하는 쪽이 더 좋은 경험에 가깝습니다.

## 이름이 같은 스킬은 더 솔직하게 알려줍니다

외부 스킬을 읽어오기 시작하면 바로 생기는 문제가 있습니다. 바로 **동명 충돌**입니다. 같은 이름의 스킬이 여러 위치에 있을 수 있고, 내용이 완전히 같을 수도 있지만 다를 수도 있습니다.

이전 방식이었다면 사용자는 어떤 스킬이 실제로 선택됐는지 알기 어려웠을 겁니다. 그래서 이번에는 동명 스킬이 발견됐을 때, 특히 내용이 서로 다른 경우에는 어느 쪽이 최종적으로 적용됐는지 명시적으로 노트로 남기도록 했습니다.

이 기능은 화려하진 않지만 디버깅할 때 정말 유용합니다. “왜 내가 생각한 스킬이 안 먹히지?” 같은 상황을 줄여주기 때문입니다. 충돌 자체를 없애는 것도 중요하지만, 충돌이 발생했을 때 시스템이 조용히 넘어가지 않고 사용자가 판단할 수 있는 정보를 주는 것도 그만큼 중요하다고 느꼈습니다.

## 스킬 생성 진입점을 추가했습니다

스킬을 쓰는 것뿐 아니라 만드는 흐름도 추가했습니다. 이제 `/skill` 명령을 통해 내장 스킬 생성을 시작할 수 있고, 꼭 명령어를 외우지 않아도 “스킬 만들고 싶다”처럼 자연어로 말했을 때 생성 흐름으로 이어질 수 있게 했습니다.

명령어 기반 인터페이스는 빠르고 명확하지만, 처음 쓰는 사람에게는 약간 딱딱하게 느껴질 수 있습니다. 반대로 자연어 트리거는 사용자가 의도를 편하게 표현할 수 있다는 장점이 있습니다. 이번 작업에서는 두 방식을 함께 열어두는 쪽을 선택했습니다.

아직 스킬 생성 경험은 더 다듬을 여지가 많습니다. 어떤 정보를 먼저 물어봐야 하는지, 생성된 스킬을 어떻게 검토하고 수정하게 할지 같은 부분은 앞으로 계속 손봐야 할 것 같습니다. 그래도 이번 변경으로 “스킬은 고급 사용자만 직접 파일을 만져서 만드는 것”이라는 느낌은 조금 줄어들었다고 생각합니다.

## 마무리

이번 커밋들은 크게 보면 Vica의 정체성을 정리하고, 스킬을 발견하고 만들고 충돌을 이해하는 흐름을 개선한 작업이었습니다. 특히 외부 스킬 인식과 충돌 고지는 앞으로 스킬 생태계가 커질수록 기반 역할을 해줄 것 같습니다.

다음에는 스킬 생성 후 편집, 검증, 미리보기 같은 부분까지 이어서 다듬어보면 좋겠습니다. 사용자가 안심하고 스킬을 만들고, 왜 그렇게 동작하는지 이해할 수 있는 방향으로 계속 개선해가겠습니다.
