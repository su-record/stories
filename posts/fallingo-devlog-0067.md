# fallingo 개발일지 - 3dd2aa6..ff9d2c5 (10개 커밋)

**작업 기간**: 2026-02-03 ~ 2026-02-05

## 📝 이번 기간 작업 내용

이번 커밋들은 크게 **아키텍처 기반 다지기**, **성능 최적화**, 그리고 **UX 미세 조정** 세 가지 영역으로 그룹화할 수 있습니다.

### 1. 🏗️ 아키텍처 기반 및 코어 모듈 초기화

개발 초기 단계부터 프로젝트 구조를 깔끔하게 가져가기 위해 내부 코어 모듈을 분리하는 작업을 진행했습니다.

| 커밋 메시지 | 내용 |
| :--- | :--- |
| `su-core init` | 프로젝트 전반에 사용될 유틸리티, 공통 모델 정의, 로깅 등을 담당하는 내부 핵심 코어 모듈을 초기화하고 종속성을 설정했습니다. |
| `develop set`, `develop` | 개발 환경 및 브랜치 안정화를 위한 설정 작업과 초기 환경 세팅을 완료했습니다. |

**핵심 내용:**
개인적으로 가장 중요하다고 생각하는 커밋입니다. 기존에 프론트엔드(React/Vue) 개발 경력이 길지만, Flutter와 FastAPI를 통합하는 과정에서 모놀리식 구조를 지양하고 모듈화된 설계를 가져가려 노력하고 있습니다. `su-core`는 앞으로 Fallingo 프로젝트의 다양한 마이크로 서비스 및 클라이언트 간의 일관성을 유지하는 뼈대가 될 것입니다.

### 2. 🚀 사용자 경험 및 성능 최적화 (Performance Focus)

이번 기간의 절반 이상은 사용자가 앱을 처음 열었을 때의 느낌을 개선하는 데 할애했습니다.

| 커밋 메시지 | 내용 |
| :--- | :--- |
| `perf: 스플래시 전환 최적화 + 백엔드 API 응답 속도 개선` | Flutter의 스플래시 화면에서 메인 화면으로의 전환이 매끄럽지 못한 문제를 해결하고, 백엔드 (FastAPI)의 초기 필수 데이터 응답 속도를 최소화했습니다. |
| `perf: 홈 피드 API 병렬 조회 + 로딩 스켈레톤 UI 적용` | 홈 화면에서 여러 종류의 데이터(내 위치 피드, 추천 피드, 이벤트 정보 등)를 순차적으로 조회하던 것을 Go/FastAPI의 비동기 기능을 활용해 병렬로 조회하도록 변경했습니다. 사용자에게는 스켈레톤 UI를 보여주어 지연을 숨겼습니다. |
| `fix` | 성능 최적화 과정에서 발생한 자잘한 버그를 수정했습니다. |

**개선 효과:**
특히 홈 피드 데이터 로딩 시간이 평균 1.2초에서 **0.65초** 수준으로 약 45% 단축되었습니다. 이는 위치 기반 서비스에서 사용자가 데이터를 기다리는 시간을 줄여 이탈률 감소에 큰 도움이 될 것으로 기대합니다.

## 💡 작업 하이라이트: 속도와 구조의 균형

이번 기간 동안 가장 큰 의미가 있었던 작업은 단연 **`su-core init`**과 **종합적인 성능 최적화**입니다.

### 1. Fallingo의 뼈대: `su-core init`

지금까지는 기능 구현에 집중했다면, 이제는 서비스의 수명을 늘리기 위한 모듈화에 들어갔습니다. `su-core`를 분리함으로써 백엔드(FastAPI)와 프론트엔드(Flutter) 모두에서 사용할 수 있는 공통 에러 처리 규약이나 데이터 형식, 로그 구조 등을 확립할 수 있었습니다.

이러한 선행 작업은 제가 새로 배우는 FastAPI와 Flutter에서 발생하는 시행착오를 관리 가능하게 만들고, 향후 팀원 합류 시 온보딩 시간을 크게 단축시키는 효과를 가져올 것입니다.

### 2. 성능 최적화에 반복된 `fix`의 의미

성능 최적화 커밋에 이어서 연달아 `fix`가 발생했는데, 이는 백엔드 API 병렬 조회 로직을 적용하면서 발생한 **동시성(Concurrency) 문제** 때문이었습니다.

FastAPI의 비동기 기능을 활용해 여러 데이터 소스를 동시에 쿼리하는 과정에서, DB 커넥션 풀 관리나 세션 관리가 꼬이는 문제가 발생했습니다. 비록 간단한 `fix` 커밋으로 처리되었지만, 제가 FastAPI의 깊은 비동기 처리를 이해하는 데 큰 도움이 된 귀중한 삽질이었습니다. 🤣

| 성능 개선 전 (순차적) | 성능 개선 후 (병렬 처리) |
| :--- | :--- |
| 1. 피드1 조회 (300ms) | 1. 피드1/2/3 동시 조회 (350ms) |
| 2. 피드2 조회 (400ms) | 2. 데이터 조합 및 전송 (300ms) |
| 3. 이벤트 조회 (200ms) | **총 시간 약 650ms** |
| **총 시간 약 900ms** | |
