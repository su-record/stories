## 최근 작업 일지: CLI 안전장치와 응답 언어 정리

이번 작업은 겉으로 보기에는 작은 수정들이지만, 실제로는 `vica`를 쓰는 흐름에서 꽤 거슬리던 지점들을 정리하는 데 집중했습니다. 크게 보면 **검증 루프 안정화**, **resume 명령 개선**, **응답 언어 일관성** 세 영역입니다.

### 셸 검증 단계의 무한 반복 방지

먼저 가장 중요하게 본 부분은 셸이 필요한 검증 체크가 특정 상황에서 계속 다시 작성되는 문제였습니다. 검증 로직은 원래 “필요하면 한 번 더 확인한다”는 의도로 들어간 장치인데, 조건이 애매하게 맞물리면 같은 검증을 반복해서 갈아엎는 상태가 될 수 있었습니다.

이런 문제는 로그만 보면 단순히 “좀 오래 걸리네?” 정도로 보일 수 있어서 더 위험합니다. 실제로는 내부 상태가 앞으로 나아가지 못하고 같은 판단을 되풀이하는 형태라, 사용자 입장에서는 멈춘 것처럼 느껴질 수밖에 없습니다.

이번에는 임시로 반복 횟수만 제한하는 방식이 아니라, 왜 재작성 판단이 계속 발생하는지를 기준 상태 쪽에서 정리했습니다. 즉, 이미 셸 검증이 필요하다고 판정된 흐름에서는 동일한 조건으로 다시 grind에 빠지지 않도록 막았습니다. 이런 류의 수정은 티가 많이 나지는 않지만, 장시간 작업이나 자동화 환경에서는 체감 안정성에 직접 영향을 줍니다.

### `resume` CLI의 사용성 보강

두 번째는 `vica resume [id]` 형태의 서브커맨드 추가와 관련된 정리입니다. 이전에는 사용자가 의도한 인자가 예상 경로를 벗어나면, 그것이 엉뚱하게 빌드 목표처럼 해석될 수 있는 여지가 있었습니다. 말하자면 사용자는 “이 작업을 이어서 진행해줘”라고 말했는데, CLI는 “아, 이걸 빌드 타깃으로 보라는 뜻인가?” 하고 받아들일 수 있었던 셈입니다.

이런 footgun은 문서로 조심하라고 적어둘 수도 있지만, 가능하면 도구가 안전하게 막아주는 편이 낫다고 생각합니다. 특히 CLI는 한 번 습관이 붙으면 빠르게 입력하게 되기 때문에, 잘못된 인자가 조용히 다른 의미로 처리되는 건 꽤 불친절합니다.

그래서 resume 흐름을 명확한 서브커맨드로 분리하고, 알 수 없는 인자가 빌드 목표로 추락하지 않도록 정리했습니다. 결과적으로 사용자는 더 예측 가능한 방식으로 이전 작업을 이어갈 수 있고, 잘못 입력했을 때도 더 일찍 문제를 알아차릴 수 있습니다.

### 요청 언어에 맞춘 응답 정리

마지막으로 `clarify`, `plan`, `SPEC` 관련 응답의 언어 선택을 손봤습니다. 기존에는 사용자가 어떤 언어로 요청했는지와 별개로 응답 언어가 기대와 다르게 나올 수 있었습니다. 특히 한국어로 작업 맥락을 주고받는 상황에서 중간 산출물이 영어로 튀어나오면 흐름이 살짝 끊깁니다.

이번 수정에서는 요청에 언어 신호가 있으면 그 언어를 따르도록 하고, 별다른 신호가 없을 때는 한국어를 기본값으로 두었습니다. `vica`를 주로 한국어 환경에서 사용하는 입장에서는 이 기본값이 더 자연스럽다고 판단했습니다.

언어 처리는 기능 자체의 핵심 알고리즘처럼 보이지는 않지만, 실제 사용 경험에서는 꽤 큰 부분을 차지합니다. 특히 clarify나 plan처럼 사용자가 다음 결정을 내리기 위해 읽는 문서는, 내용만큼이나 읽는 리듬이 중요합니다.

### 마무리

이번 세 커밋은 모두 “도구가 사용자의 의도를 덜 배신하게 만들기”에 가까운 작업이었습니다. 무한 반복을 막고, CLI 인자 해석을 더 안전하게 만들고, 응답 언어를 요청 맥락에 맞추는 일은 각각 작아 보여도 결국 같은 방향을 보고 있습니다.

아직 더 다듬을 부분은 많지만, 이번 정리로 `vica`가 조금 더 안정적이고 예측 가능한 도구에 가까워졌다고 느낍니다. 다음에는 이 안정성을 바탕으로 실제 작업 흐름을 더 매끄럽게 만드는 쪽을 이어서 살펴볼 예정입니다.
