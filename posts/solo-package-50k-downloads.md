별은 0개였습니다. 그런데 다운로드는 51,647건이었습니다.

혼자 쓰려고 만든 npm 패키지였습니다. 어느 날 통계를 열어보니 매일 수십에서 수백 명이 설치하고 있었습니다. 지난 한 달만 3,141건. 봇과 CI를 감안해도 실사용자가 분명히 있는 숫자였습니다.

그런데 그 사용자들이 저에게 말을 걸 방법이 없었습니다. 이슈 탭은 닫혀 있었고, 릴리스는 제 노트북에서 수동으로 나갔습니다. 사용자는 있는데 접점이 없는 상태였습니다.

그래서 하루를 들여 운영 체제를 갖췄습니다. 이 글은 그 기록입니다.

## 별이 없는 건 정상입니다

먼저 이상해 보이는 숫자부터 짚고 가겠습니다. 다운로드 5만에 별 0개.

npm 사용자는 GitHub에 가지 않습니다. `npm install` 하고 끝입니다. 저장소까지 찾아가서 별을 누르는 사람은 극소수입니다. 다운로드 대비 스타 비율이 1000:1이어도 이상하지 않습니다.

별은 사용자가 저장소에 올 이유가 있어야 생깁니다. 그리고 사용자가 저장소에 오는 가장 흔한 이유는 버그 신고입니다. 즉, 이슈 탭이 닫혀 있으면 별이 생길 경로도 막혀 있는 셈입니다.

## 1. 이슈 탭을 열었습니다

![이슈 탭을 연 직후의 모습](/images/solo-package-50k-downloads/01-issue-templates.png)

이슈 탭을 열고 템플릿 세 개를 준비했습니다. 버그 리포트, 기능 제안, 그리고 메이저 버전 마이그레이션 문의. 템플릿이 있으면 들어오는 이슈가 구조화됩니다. 재현 방법과 환경 정보를 처음부터 받으니 왕복 질문이 줄어듭니다.

`package.json`에 `bugs` 필드도 추가했습니다. 이 한 줄로 npm 패키지 페이지에 "Report a bug" 링크가 생깁니다. 사용자 입장에서 신고 경로가 한 클릭 줄어듭니다.

## 2. PR은 기계가 먼저 검증합니다

이슈를 열면 PR도 들어옵니다. 공개 저장소는 포크 후 PR을 막을 수 없습니다. 막을 필요도 없습니다. 검증만 자동이면 됩니다.

- 모든 PR에서 빌드와 테스트가 자동으로 돌고
- 브랜치 보호로 체크가 초록불이어야만 머지되고
- 처음 기여하는 사람의 CI는 승인 후 실행됩니다 (악성 코드가 CI에서 실행되는 것을 막는 GitHub의 안전장치입니다)

여기까지는 설정 몇 개입니다. 코드 한 줄 없이 끝났습니다.

## 3. 릴리스는 태그 하나로 끝납니다

원래 릴리스는 제 노트북에서 `npm publish`를 치는 방식이었습니다. 이걸 태그 기반 파이프라인으로 바꿨습니다.

```
pnpm release 한 번
  → 버전 올리고 태그 push
  → CI: 빌드 → 테스트 → npm 배포 → GitHub Release 노트 자동 생성
```

![릴리스 파이프라인이 완주한 모습](/images/solo-package-50k-downloads/02-release-run.png)

릴리스 노트도 자동입니다. GitHub이 직전 태그 이후 머지된 PR 목록으로 "What's Changed"를 만들어줍니다. PR을 모아서 배포하는 흐름과 정확히 맞아떨어집니다.

## 삽질: CI에는 인증앱이 없습니다

물론 한 번에 되지는 않았습니다. 첫 배포는 `EOTP` 에러로 죽었습니다.

npm 계정에 2FA를 걸어두면 배포할 때마다 일회용 비밀번호를 요구합니다. 문제는 CI에는 인증앱이 없다는 겁니다. 선택지는 세 개였습니다.

1. 2FA를 끈다 — 5만 다운로드 패키지 계정으로는 무리입니다
2. 2FA를 우회하는 토큰을 만든다 — 우회라는 단어부터 마음에 들지 않습니다
3. **토큰 자체를 없앤다** — 이게 정답이었습니다

npm의 Trusted Publishing(OIDC)을 썼습니다. npm 패키지 설정에서 "이 GitHub 저장소의 이 워크플로만 배포할 수 있다"고 등록하면, CI가 토큰 없이 자기 신원을 암호학적으로 증명하고 배포합니다.

패키지 설정은 가장 엄격한 옵션으로 잠갔습니다. **토큰 배포 자체를 차단**하고, 배포 주체는 오직 2FA를 통과한 본인 또는 등록된 CI 워크플로뿐입니다. 토큰이 없으니 유출될 토큰도, 챙길 만료일도 없습니다.

![자동 생성된 GitHub Release](/images/solo-package-50k-downloads/03-github-release.png)

배포된 패키지에는 provenance 배지가 붙습니다. "이 패키지는 이 저장소의 이 커밋에서 빌드됐다"는 증명입니다. 사용자 입장에서는 공급망을 신뢰할 근거가 하나 생깁니다.

## 4. 이슈에는 봇이 먼저 답합니다

여기까지가 보통의 오픈소스 운영 설정이라면, 마지막 하나는 조금 다릅니다.

혼자 운영하는 프로젝트에서 가장 부담스러운 건 이슈 응대입니다. 잠든 사이에 버그 리포트가 쌓이는 상상만으로도 열기가 싫어집니다. 그래서 1차 응대를 자동화했습니다.

마침 집에 놀고 있지 않은 서버가 한 대 있었습니다. 이미 에이전트가 상주하며 다른 일을 하고 있어서, 거기에 워크플로 하나를 결합했습니다.

```
새 이슈/PR/댓글 (GitHub 알림)
  → 5분마다 서버가 캐치
  → 디스코드 #monitoring에 기록
  → AI가 저장소 클론을 읽고 답변 댓글 작성
  → 디스코드 #report에 작업 결과 보고
```

동작 원리는 단순합니다. `gh api notifications`로 읽지 않은 알림을 가져오고, 이슈·PR 스레드 전체와 코드를 AI(Claude)에게 넘깁니다. AI는 유용한 답변이 가능할 때만 댓글을 정확히 1개 남깁니다. 이미 충분한 답이 달려 있으면 아무것도 하지 않습니다.

안전장치가 핵심입니다.

- **댓글 권한만** 줬습니다. 커밋, 푸시, 이슈 닫기, 라벨 변경은 도구 권한 자체가 없습니다
- 스레드의 마지막 댓글이 자기 자신이면 스킵합니다. 혼자 무한 답글을 다는 사고를 막습니다
- 실패하면 재시도하지 않고 디스코드로 사람을 호출합니다. 5분마다 실패를 반복하며 비용을 태우는 것보다 낫습니다

사용자가 봇 댓글에 다시 답하면 새 알림이 생기고, 봇이 스레드 전체를 다시 읽고 맥락을 이어갑니다. 대화가 됩니다.

## 하루의 결과

정리하면 이렇게 됐습니다.

```
이슈  → 봇이 5분 내 캐치 → 1차 답변 → 디스코드 보고
PR   → CI 검증 → 봇이 1차 리뷰 → 머지 판단만 사람이
릴리스 → 태그 하나 → 빌드·테스트·배포·릴리스 노트 전부 자동
```

사람이 하는 일은 두 가지만 남았습니다. 머지 판단과 릴리스 결정. 나머지는 기계가 합니다.

비용은 서버 한 대(이미 있던 것)와 하루치 설정이었습니다. 새로 산 것도, 새로 배운 프레임워크도 없습니다. GitHub Actions, npm Trusted Publishing, gh CLI, cron. 전부 이미 있는 도구였습니다.

## 마치며

별 0개는 실패의 숫자가 아니었습니다. 통로가 없었다는 신호였습니다.

혼자 쓰려고 만든 것이 5만 번 설치됐다면, 이미 혼자만의 것이 아닙니다. 사용자가 말을 걸 수 있는 통로를 열고, 응대와 배포를 자동화하는 데 하루면 충분했습니다.

이 글을 읽는 분 중에 "내 패키지도 다운로드가 좀 있던데"싶은 분이 있다면, 통계부터 열어보시기 바랍니다. 생각보다 많은 사람이 이미 쓰고 있을지도 모릅니다.
