## `build.ts` 다이어트: 빌드 흐름을 역할별로 나눴습니다

이번 작업은 새로운 기능을 추가했다기보다는, 점점 비대해지던 `build.ts`를 다시 읽을 수 있는 상태로 되돌리는 리팩터링에 가까웠습니다. 한 파일 안에 프롬프트 구성, transcript 정리, tool 스펙, 출력 포맷팅까지 섞여 있다 보니 “어디를 고치면 되는지” 찾는 데 드는 비용이 꽤 커졌습니다.

결과적으로 `build.ts`는 1217줄에서 983줄까지 줄었고, 일단 목표로 잡았던 1000줄 미만에 들어왔습니다. 숫자 자체가 전부는 아니지만, 파일을 열었을 때 느껴지는 압박감이 확실히 줄어든 건 꽤 만족스럽습니다.

## 프롬프트와 포맷팅 분리

먼저 leaf 단계에서 사용하는 프롬프트와 결과 포맷터를 `build-prompt.ts`로 분리했습니다. 기존에는 빌드 흐름을 따라가다가 갑자기 긴 프롬프트 문자열과 포맷팅 로직이 등장해서 맥락이 자주 끊겼습니다.

이제 `build.ts`는 “무엇을 실행하는지”에 더 집중하고, 프롬프트를 어떻게 만들고 결과를 어떻게 정리하는지는 별도 파일에서 다룹니다. 특히 프롬프트는 조금만 수정해도 전체 동작에 영향을 줄 수 있어서, 앞으로 실험하거나 문구를 조정할 때 더 부담 없이 건드릴 수 있을 것 같습니다.

## Transcript 정리 로직 독립

다음으로 transcript를 줄이고 정리하는 컨덴서와 프루너 로직을 `build-transcript.ts`로 옮겼습니다. 이 영역은 입력 맥락을 얼마나 보존할지, 어떤 내용을 줄일지와 관련되어 있어서 빌드 파이프라인 안에서도 꽤 중요한 책임을 갖고 있습니다.

그런데 기존에는 이 로직이 메인 빌드 흐름 중간에 들어가 있어, 전체 파일의 밀도를 높이는 원인이 됐습니다. 분리하고 나니 transcript 처리만 따로 읽을 수 있게 되었고, 나중에 압축 기준을 조정하거나 pruning 전략을 바꾸기도 쉬워졌습니다. 개인적으로는 이번 리팩터링 중에서 체감상 가장 정리가 잘 된 부분이었습니다.

## Tool 스펙과 실행 옵션 정리

마지막으로 tool 스펙, effort 설정, narrate 관련 로직을 `build-tools.ts`로 분리했습니다. 이 부분은 모델 호출이나 도구 사용 방식과 맞물려 있어 변경 가능성이 높은 영역입니다. 그래서 빌드 본체에 그대로 두기보다는, “도구 실행에 필요한 설정 묶음”으로 따로 관리하는 편이 더 자연스럽다고 판단했습니다.

특히 `effort`나 `narrate` 같은 옵션은 나중에 정책이 바뀌거나 모드가 늘어날 가능성이 있습니다. 이번에 경계를 나눠둔 덕분에, 이후에는 빌드 흐름을 건드리지 않고도 실행 옵션 쪽만 비교적 안전하게 수정할 수 있을 것 같습니다.

## 마무리

이번 리팩터링의 핵심은 단순히 줄 수를 줄이는 게 아니라, `build.ts`가 맡아야 할 일을 다시 좁히는 것이었습니다. 아직 완벽하게 작고 아름다운 구조라고 말하긴 어렵지만, 최소한 이제는 한 파일이 모든 책임을 떠안고 있다는 느낌은 많이 줄었습니다.

앞으로는 이번에 나눈 파일들을 기준으로 테스트하기 쉬운 단위도 더 명확히 만들고 싶습니다. 특히 프롬프트, transcript, tool 설정은 각각 변화 속도가 다른 영역이라서, 이번 분리가 이후 작업의 발판이 되어줄 것 같습니다. 코드가 커지는 건 어쩔 수 없지만, 커진 코드를 계속 다룰 수 있게 만드는 일도 꾸준히 챙겨가야겠네요.
