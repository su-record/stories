"AI에게 일을 시켰다"는 이제 흔한 문장이다. 나는 다른 걸 해봤다. 서버 한 대를 빌려 AI 직원 8명을 채용했다.

비용은 거의 0원이다. Oracle Cloud의 무료 ARM 서버 위에 Hermes Agent를 올렸고, 그 위에서 디스코드 봇 8명이 각자 역할을 가지고 일한다. 이 글은 계정 생성부터 봇 운영, 그리고 오픈소스 기여까지의 전 과정을 그대로 기록한 것이다. 순서대로 따라오면 같은 걸 만들 수 있다.

---

## 1. Oracle Cloud 무료 서버 — 가장 큰 관문

전체 과정에서 제일 까다로운 단계가 첫 단계다. 서버를 받는 것.

Oracle Cloud는 Always Free로 ARM(Ampere A1) 인스턴스를 준다. 최대 4 OCPU / 24GB까지 무료다. 문제는 인기가 많아 용량이 부족하다는 점이다. 인스턴스를 만들려고 하면 "Out of host capacity" 에러가 자주 뜬다.

여기서 중요한 팁이 있다.

- **무료 계정 그대로면 ARM 할당이 어렵다.** "Out of capacity"를 며칠씩 만날 수 있다.
- **유료(Pay As You Go)로 업그레이드하면 할당 우선순위가 올라간다.** 카드를 등록해 계정을 Pay As You Go로 바꾸면 ARM 용량을 훨씬 빨리 받는다.
- 업그레이드해도 **Always Free 한도(4 OCPU / 24GB) 안에서 쓰면 과금은 거의 없다.** 할당만 빠르게 받고 무료 범위로 운영하는 전략이다.

인스턴스 생성 설정은 이렇게 한다.

| 항목 | 값 |
|------|-----|
| 이미지 | Ubuntu (LTS) |
| Shape | VM.Standard.A1.Flex (ARM) |
| OCPU / RAM | 무료 한도 내 (예: 2 OCPU / 12GB) |
| SSH 키 | 로컬에서 생성한 공개키 등록 |

SSH 키는 미리 만들어둔다. 로컬에서 `ssh-keygen -t ed25519 -f ~/.ssh/oracle_hermes`로 키 쌍을 만들고, 인스턴스 생성 시 공개키(.pub)를 등록한다. 개인키는 절대 공유하지 않는다.

---

## 2. SSH 접속

인스턴스가 뜨면 공인 IP가 나온다. 개인키로 접속한다.

```bash
ssh -i ~/.ssh/oracle_hermes ubuntu@<공인_IP>
```

"permissions too open" 에러가 나면 키 권한을 조인다.

```bash
chmod 600 ~/.ssh/oracle_hermes
```

접속되면 서버는 빈 Ubuntu다. 여기에 직원들을 들인다.

## 3. Hermes Agent 설치

Hermes Agent는 Nous Research의 에이전트 프레임워크다. 핵심 특징 두 가지가 이 프로젝트를 가능하게 했다.

- **멀티 프로필**: 프로필마다 다른 두뇌(모델)·도구·역할·디스코드 봇 토큰을 가진다. 즉 한 서버에서 여러 명의 독립적인 에이전트를 돌린다.
- **칸반 보드**: 호스트 전체가 공유하는 SQLite 작업 보드. 오케스트레이터가 작업을 쪼개 각 봇에 배정하고, 봇은 병렬로 일하며, 막히면 넘긴다. 크래시가 나도 보드가 견딘다.

설치는 공식 문서(hermes-agent.nousresearch.com)를 따른다. 설치 후 `hermes setup --portal`로 모델 제공자와 도구(웹검색·이미지·TTS·브라우저)를 한 번에 연결한다. 무인 운영에는 `--portal`이 OAuth 자동 갱신을 해줘서 안정적이다.

게이트웨이는 디스코드·텔레그램·슬랙 등 여러 플랫폼을 한 프로세스에서 받는다. 우리는 디스코드를 팀 워크스페이스로 썼다.

---

## 4. 봇(직원) 구성 — 프로필 + 디스코드 + 칸반

직원 한 명 = 프로필 하나다. 각 프로필에 이렇게 설정한다.

1. **SOUL.md**: 그 봇의 정체성·역할·말투. (예: 비서실장은 "오케스트레이션만, 실무는 위임".)
2. **모델**: 판단·품질이 중요한 봇은 GPT, 내부 잡일은 더 싼 모델로 분담.
3. **디스코드 봇 토큰**: 디스코드 개발자 포털에서 봇을 만들어 토큰을 프로필 .env에 넣는다. MESSAGE CONTENT INTENT를 켜야 메시지를 읽는다.
4. **게이트웨이 기동**: `hermes -p <프로필> gateway install`로 systemd 서비스로 띄운다. `loginctl enable-linger`로 로그아웃 후에도 살아있게 한다.

우리 8명은 이렇게 나눴다.

| 봇 | 역할 |
|----|------|
| Pepper | 비서실장 / 오케스트레이터 |
| Tony | 수석 개발 (codex 코딩) |
| Linus | 코드리뷰 / QA |
| Steven | 기획 / 제품 |
| Tesla | R&D |
| Ive | 디자인 |
| Miranda | 마케팅 / 콘텐츠 |
| Gutenberg | 발행 / 배포 |

요청 채널에 "이거 해줘"라고 적으면 Pepper가 읽고, 칸반에 작업을 만들어 맞는 봇에 배정한다. 배정된 봇이 일하고, 결과가 보고 채널로 온다. 사람은 던지고 받기만 한다.

한 가지 배운 점. 봇이 메시지에 응답하지 않으면 대개 `config.yaml`의 `free_response_channels`가 비어 있어서다. 환경변수보다 config가 우선하니, 채널 ID를 config에 직접 넣어야 한다.

## 5. 결과 — 매일 굴러가는 콘텐츠 엔진

며칠 만에 이 팀이 스스로 돌기 시작했다.

- **데일리 브리핑**: 매일 06시, RSS 크롤러가 모은 뉴스와 유튜브, X 화제를 한 채널에 정리한다.
- **카드뉴스**: 그날 뉴스에서 중요한 것을 골라 카드 8장(커버·뉴스·영상·X·한 줄 정리)으로 자동 렌더링한다. 발행 데스크 봇이 승인 버튼과 함께 올리고, 승인하면 LinkedIn에 게시된다.
- **주간 회의**: 매주 월요일, 전 직원이 그 주를 리뷰하고 회의록을 GitHub에 남긴다.
- **장기 메모리**: Hindsight를 붙여 봇들이 대화할수록 기억이 쌓이고 회상된다.

검색 한 번이 막혔던 적이 있다. 클라우드 IP가 무료 검색엔진에 차단당해 결과가 0건이었다. 진짜 검색 API로 바꿔 해결했다. 클라우드에서 무료 스크래핑 검색은 막힌다는 걸 그때 배웠다.

## 6. 그리고 — 커뮤니티에 기여

작업 중에 발견한 게 있다. OpenAI Codex CLI에는 웹 검색 기능이 있고, ChatGPT Pro 구독으로 인증하면 별도 API 키나 콜당 과금 없이 쓸 수 있다.

이 노하우를 Hermes Agent에 기여했다. `codex exec -c tools.web_search=true`로 무료 웹 리서치를 하는 검색 스킬을 만들어 Pull Request를 보냈다. (NousResearch/hermes-agent #50526)

맨바닥에서 시작한 프로젝트가, 그 프레임워크 자체에 한 줄을 보태는 데까지 왔다.
