# vibe 개발일지 #38 - 멀티 LLM 오케스트레이션과 엔지니어링 규율의 결합 (14개 커밋)

**작업 기간**: 2026-04-12 ~ 2026-04-13

안녕하세요, vibe를 개발하고 있는 Su입니다. 

주말 동안 정말 숨 가쁘게 14개의 커밋을 쌓았습니다. 이번 업데이트의 핵심은 **"vibe가 어떻게 다양한 LLM(Claude Code, Codex, Gemini 등)을 하나의 하네스(Harness) 안에서 통제할 것인가"**에 대한 해답을 찾는 과정이었습니다. 단순히 AI 기능을 추가하는 것을 넘어, 도구가 도구를 제어할 때 발생하는 기술적 난제들을 해결하는 데 집중했습니다.

## 📝 이번 기간 작업 내용

이번 14개의 커밋은 크게 세 가지 영역으로 나뉩니다.

### 1. 멀티 LLM 오케스트레이션 및 Claude 통합
*   **Claude Code CLI 감지 및 연동**: Anthropic의 Claude Code를 vibe의 에이전트 시스템에 정식 편입했습니다. (`feat: cli-detector 추가`)
*   **Coco 환경 대응**: 내부적으로 사용하는 Coco 환경을 감지하고, 보조 LLM 분기를 확장하여 Claude와의 시너지를 극대화했습니다.
*   **설치 로직 리팩터링**: 개별 LLM(Codex/Gemini)의 직접적인 설치 로직을 제거하고, 대신 중앙 집중형 에셋 관리 방식으로 전환하여 패키지 무게를 줄였습니다.

### 2. 훅(Hooks) 시스템의 안정성 강화 (품질 보장)
*   **프로세스 증식 방지 (재귀 가드)**: 훅이 훅을 부르고, 다시 노드 프로세스가 실행되는 무한 루프를 방지하기 위해 '재귀 가드'와 '디스패처 통합'을 구현했습니다. (`fix(hooks): node 프로세스 증식 방지`)
*   **데이터 정제**: 훅 객체 내에 불필요한 주석 키(`_comment_*`)를 제거하여 런타임 오버헤드를 줄였습니다.

### 3. 엔지니어링 규율 및 문서화
*   **Rob Pike의 원칙 통합**: AI가 코드를 작성할 때 "Systematic Debugging"과 Rob Pike의 프로그래밍 철학을 따르도록 시스템 프롬프트와 규율을 통합했습니다.
*   **역할 역전 설명 추가**: LLM이 주도하는 것이 아니라, vibe(Harness)가 LLM을 도구로 사용하는 '역할 역전'에 대한 철학을 README에 명시했습니다.

## 💡 작업 하이라이트

### "좀비 프로세스는 이제 그만" — 훅 재귀 가드 (Recursion Guard)
vibe의 철학은 "Minimum quality guaranteed"입니다. 이를 위해 코드가 실행되기 전후에 다양한 하네스(훅)가 동작하는데, 이 과정에서 훅이 다시 vibe를 호출하면서 Node.js 프로세스가 기하급수적으로 증식하는 문제가 있었습니다.

**Before:**
*   `vibe run` → `pre-hook` 실행 → 내부적으로 다시 `vibe` 관련 로직 호출 → 프로세스 폭발 → 시스템 다운.

**After:**
*   `VIBE_RECURSION_GUARD` 환경 변수와 중앙 디스패처를 도입했습니다. 이미 실행 중인 vibe 컨텍스트가 있다면 새로운 프로세스를 띄우는 대신 기존 컨텍스트를 재사용하거나 실행을 차단합니다. 덕분에 이제 복잡한 훅 체인에서도 시스템이 안정적으로 유지됩니다.

### "Agent = Model + Harness" 철학의 구체화
최근 많은 AI 도구들이 나오지만, 저는 항상 **"모델은 교체 가능해야 하며, 본질은 이를 통제하는 하네스(Harness)에 있다"**고 믿습니다. 

이번 업데이트에서 Claude Code CLI를 감지하고 연동하는 기능을 넣으면서, 단순히 기능을 붙이는 게 아니라 `vibe.spec`에 정의된 규율을 Claude에게 주입하는 방식을 택했습니다. Claude가 아무리 똑똑해도 vibe의 엔지니어링 규율(Rob Pike 스타일의 간결함, 체계적 디버깅) 안에서 움직이도록 강제한 것이죠.
