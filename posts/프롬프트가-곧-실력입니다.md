같은 AI에 같은 일을 맡겼습니다. 그런데 결과는 달랐습니다.

모델이 달라서가 아니었습니다. 질문에 필요한 내용이 빠져 있었습니다.

2026년 4월 5일 기사에 이런 문장이 나왔습니다. “질문의 품질이 답변의 품질을 결정해요.” 맞는 말입니다. 여기서 말하는 품질은 문장을 예쁘게 꾸미는 능력이 아닙니다. 맥락, 제약, 예시, 출력 형식입니다.

## 답이 나쁜 게 아니라 질문이 비어 있었습니다

![프롬프트가 곧 실력입니다](/images/프롬프트가-곧-실력입니다/fig1.png)


많은 프롬프트는 이렇게 시작합니다.

```text
Gemini Spark 설명해줘.
```

짧은 질문입니다. 그래서 AI가 나머지를 알아서 추측합니다. 누구에게 설명해야 하는지, 어느 정도 길이로 써야 하는지, 어떤 관점이 필요한지 알 수 없습니다. 결과는 대체로 무난한 요약문이 됩니다.

| 빠진 요소 | AI가 추측하는 것 | 결과 |
|---|---|---|
| 독자 | 개발자, 기획자, 일반 사용자 중 하나 | 초점이 흐려짐 |
| 목적 | 소개, 비교, 설득, 사용법 중 하나 | 문장 구조가 애매해짐 |
| 맥락 | 어떤 업무에서 쓰는지 | 기능 나열로 끝남 |
| 제약 | 금지어, 분량, 톤 | 흔한 홍보문이 됨 |
| 출력 형식 | 표, 목록, 문단 | 다시 편집해야 함 |

AI는 빈칸이 있어도 멈추지 않습니다. 그냥 채웁니다. 문제는 그 빈칸이 내 의도와 다르게 채워진다는 점입니다.

“블로그 글 써줘”라고만 묻고 좋은 글을 기대하기는 어렵습니다. “누구에게, 왜, 어떤 자료로, 어떤 형식으로”가 빠져 있기 때문입니다.

## 좋은 프롬프트는 명령이 아니라 업무 인수인계입니다

프롬프트를 명령으로 보면 짧아집니다. 인수인계로 보면 내용이 달라집니다.

나는 아래 7가지를 기준으로 씁니다. 하나라도 빠지면 결과가 흔들립니다.

| 항목 | 질문 |
|---|---|
| 역할 | AI가 어떤 관점으로 답해야 하는가 |
| 상황 | 지금 어떤 배경에서 이 작업을 하는가 |
| 목표 | 결과물이 무엇을 달성해야 하는가 |
| 자료 | 반드시 반영할 사실은 무엇인가 |
| 제약 | 쓰면 안 되는 표현, 분량, 톤은 무엇인가 |
| 형식 | 표, 코드, 목록, 문단 중 무엇인가 |
| 예시 | 좋은 답의 기준을 보여주는 샘플이 있는가 |

실제로는 이렇게 씁니다.

```text
역할: B2B SaaS 콘텐츠 에디터처럼 답해라.
상황: 비개발 운영팀장이 AI 자동화 도구를 처음 검토한다.
목표: Gemini Spark가 어떤 업무에 쓰이는지 900자 안에 설명한다.
자료: 태스크, 스킬, 스케줄 기능을 반드시 포함한다.
제약: 과장 표현을 쓰지 마라. 코딩 지식 없이 쓸 수 있다는 점은 넣어라.
형식: 3문단으로 쓰고, 마지막에 활용 예시 3개를 표로 정리해라.
예시 톤: 기능보다 업무 장면을 먼저 보여준다.
```

길어 보이지만 편집 시간을 줄여줍니다. 처음부터 방향을 잡아주기 때문입니다.

짧은 질문은 입력하는 시간이 적게 듭니다. 대신 나중에 고치는 시간이 늘어납니다. 긴 질문은 처음에 시간이 조금 더 듭니다. 그만큼 바로 쓸 수 있는 결과가 나올 가능성이 높아집니다.

## 같은 질문도 세 번 바꾸면 답 품질이 달라집니다

Google Gemini Spark를 설명한다고 가정해보겠습니다. 제공된 정보는 명확합니다. 코딩 지식 없이 자연어로 업무를 자동화합니다. 태스크, 스킬, 스케줄을 통해 Gmail, 캘린더, 서드파티 앱과 연결됩니다. 사용자가 자는 동안에도 백그라운드에서 작업합니다.

질문 방식에 따라 결과는 이렇게 달라집니다.

| 단계 | 프롬프트 | 예상 결과 |
|---|---|---|
| 1단계 | Gemini Spark 설명해줘 | 일반 소개문 |
| 2단계 | 비개발 운영팀장에게 Gemini Spark를 설명해줘 | 독자에 맞춘 설명 |
| 3단계 | 업무 자동화 도입 검토 문서로 써줘. 기능, 사용 장면, 한계를 표로 나눠줘 | 의사결정용 초안 |

3단계 프롬프트는 단순히 긴 문장이 아닙니다. 목적이 있고, 독자가 있고, 출력 형식이 있습니다.

조금 더 구체적으로 쓰면 이렇습니다.

```text
비개발 운영팀장을 독자로 삼아라.
Gemini Spark를 도입 검토 문서 형식으로 설명해라.

반드시 포함할 사실:
- 자연어 지시로 업무 자동화를 설정한다.
- 태스크, 스킬, 스케줄이 핵심 기능이다.
- Gmail, 캘린더, 서드파티 앱과 연동한다.
- 사용자가 자는 동안에도 백그라운드에서 작업한다.

출력 형식:
1. 한 문장 요약
2. 핵심 기능 표
3. 실제 업무 예시 3개
4. 도입 전 확인할 질문 3개

금지:
- 획기적, 혁신적 같은 추상 표현
- 근거 없는 생산성 수치
- 기능만 나열하는 문장
```

여기서 중요한 부분은 “금지”입니다. AI는 좋은 문장을 쓰려다가 흔한 홍보 문구를 붙일 때가 많습니다. 금지어를 주면 그 방향으로 가지 않게 막을 수 있습니다.

출력 형식도 중요합니다. 표를 요구하면 비교가 쉬워집니다. 질문 3개를 요구하면 검토 문서에 가까워집니다. 같은 정보라도 어디에 쓰느냐에 따라 형태가 달라집니다.

## 예시는 답의 방향을 고정합니다

AI에게 가장 강한 힌트는 예시입니다. 길게 설명하는 것보다 예시 하나가 빠를 때가 많습니다.

“간결하게 써줘”는 모호합니다. 사람마다 생각하는 간결함이 다릅니다. 좋은 예시 1개를 주면 기준이 생깁니다.

```text
좋은 문장 예시:
나쁜 질문은 AI에게 추측을 맡깁니다. 좋은 질문은 추측할 공간을 줄입니다.

이 톤을 유지해라.
문장은 짧게 써라.
한 문단은 2~3문장으로 제한해라.
기능을 말할 때는 반드시 업무 장면을 붙여라.
```

이 방식은 글쓰기에서 특히 잘 통합니다. 톤, 길이, 문장 밀도가 한 번에 전달됩니다. “전문적으로”, “친근하게” 같은 말보다 기준이 분명합니다.

예시는 코드 리뷰에도 쓸 수 있습니다.

```text
나쁜 리뷰:
이 함수 개선해줘.

좋은 리뷰:
아래 함수에서 중복 조건문을 줄이고 싶다.
동작은 바꾸지 마라.
시간 복잡도가 달라지면 설명해라.
수정 전후 코드를 나란히 보여줘.
테스트 케이스 3개를 추가해라.
```

AI는 요구사항이 좁을수록 안정적으로 답합니다. 반대로 “알아서 잘”이라는 표현이 들어가면 결과가 흔들립니다.

## 프롬프트를 쓰기 전에 확인할 것

프롬프트는 글솜씨가 아닙니다. 업무를 맡기는 방식입니다.

보내기 전에 아래 질문을 확인합니다.

| 체크 | 질문 |
|---|---|
| 독자 | 이 답을 읽는 사람은 누구인가 |
| 목적 | 요약, 설득, 비교, 실행 중 무엇인가 |
| 자료 | 반드시 반영할 사실을 줬는가 |
| 기준 | 좋은 답의 예시를 줬는가 |
| 제약 | 금지어와 분량을 정했는가 |
| 형식 | 표, 목록, 코드, 문단을 지정했는가 |
| 검수 | 틀릴 수 있는 부분을 확인하게 했는가 |

좋은 프롬프트는 AI를 똑똑하게 만드는 문장이 아닙니다. AI가 추측하지 않게 만드는 문장입니다.

질문이 비어 있으면 답은 그 빈칸을 채워서 옵니다. 그 빈칸을 누가 채우느냐에 따라 결과가 달라집니다.
