# AI와 협업하는 새로운 방법론

포기하려던 순간, AI와 대화를 통해 새로운 방법을 찾았다.

지금까지의 실패를 AI에게 설명했다. 함께 더 나은 방법을 찾았다.

## 새로운 문서 작성 방식

기획서가 준비되어 있었다. 몇 개월 진행하며 업데이트된 아이디어가 잘 반영됐는지 확인부터 했다.

### 순차적 문서 작성 프로세스

1. 기획서 업데이트
2. 엔티티 추출 → 엔티티 문서 작성
3. ERD 설계 → 데이터 모델 문서 작성
4. API 정의 → API 문서 작성
5. 비즈니스 로직 → 비즈니스 로직 문서 작성

핵심: 순서대로, 단계별로.

### 과감한 결정: 모든 것을 제거하다

새로운 구성으로 시작할 때마다, 기획 문서를 제외한 나머지 모든 문서와 개발 소스코드를 모두 제거했다.

세 번째 문서를 삭제할 때, 손이 떨렸다. 일주일의 작업이었다.

하지만 일관성 없는 코드는 독이었다.

## 여전히 남은 문제: 일관성

문서 작성 방식을 바꿨지만, 여전히 일관성 문제가 발생했다.

교차 검증하듯 새롭게 문서를 작성해보면, 항상 이전 문서와 차이점이 발생했다.

어떤 때는 API가 180개였던 적도 있었다.

## 돌파구: 공통된 내용에 집중하다

차이가 발생하는 원인을 AI에게 분석 요청했다.

리포트를 받았다:

> 작업하는 AI마다 문서 제목이나 요약 정보를 통해 먼저 해야 할 일을 정한다.
> 이 순서가 핵심 기능 위주였다.

공통된 내용이 거의 다 AI가 핵심 기능이라고 판단한 것들이었다.

## 해결책: AI에게 명확한 순서 지시

다시 요청했다.

### 핵심 지시사항

1. 기획서를 최대한 자세하게 읽어라
2. 반드시 문서 번호대로 순차적으로 읽어라

### 개발 문서 작성 순서 강제

순서대로:

```
기획서 → 엔티티 → ERD → 데이터모델 → API, 비즈니스로직 → 데이터베이스
```

몇 번의 복수 검증을 토대로 최종 문서 작성을 완료했다.

## 개발 방식도 변경: 백엔드 먼저

보통의 개발:

- 기획 리뷰 후 인터페이스 정의
- 백엔드와 프론트가 각각 동시 개발
- 디자인 완성되면 프론트가 반영

나도 AI에게 동시 개발을 주문했었다. 개발 가이드 문서가 있으니까.

### 하지만 방법을 바꿨다

> 백엔드가 탄탄한 게 먼저다.

단번에 잘 되진 못했다. 다시 시작하는 걸 반복했다.

그래도 어느 정도 백엔드 개발이 진행되던 중, SpecKit을 알게 됐다.
