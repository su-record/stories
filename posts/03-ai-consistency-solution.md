# AI와 협업하는 새로운 방법론 - 일관성 문제를 해결하다

> 이전 편: "AI를 위한 문서 작성법 - 왜 일관성이 무너졌을까?"

## 새로운 방법론

포기하려던 순간, AI와 대화를 통해 새로운 방법을 찾았습니다.

지금까지의 실패를 AI에게 설명했습니다. 함께 더 나은 방법을 찾았습니다.

AI와 함께 찾아낸 새로운 방법은 이렇습니다:

1. **기획서 업데이트**: 몇 개월 진행하며 업데이트된 아이디어가 잘 반영됐는지 확인
2. **엔티티 추출**: 기획서를 바탕으로 엔티티를 추출해서 문서 작성
3. **ERD & 데이터 모델**: 엔티티 문서를 바탕으로 ERD와 데이터 모델 구성
4. **API 정의**: API를 정의하는 문서 작성
5. **비즈니스 로직**: 비즈니스 로직 문서 작성

항상 뭔가 새로운 구성으로 시작할 땐 **기획 문서를 제외한 나머지 모든 문서와 개발 소스 코드를 모두 제거**하는 방식으로 작업해왔습니다.

어떨 땐 너무 아까울 때도 있었지만요. ㅎㅎㅎ

## 여전히 일관성 문제

그렇게 문서 작성 방식을 바꿨지만, 그래도 여전히 일관성은 문제가 발생했습니다.

교차 검증하듯 새롭게 문서를 작성해보면 항상 이전 문서와 차이점이 발생했습니다.

어떤 때는 **API가 180개**였던 적도 있었습니다. ㅋ

## 공통된 내용에 집중하다

여기서 저는 **공통된 내용에 집중**했습니다.

그리고 왜 차이가 발생하는지 분석을 요청해서 리포트를 받아보았습니다.

그랬더니 작업을 하는 AI마다 **문서 제목이나 요약 정보를 통해 먼저 해야 할 일을 정하는데 이 순서가 핵심 기능 위주**더라고요.

공통된 내용이 거의 다 AI가 핵심 기능이라고 판단한 것들이었습니다!

## 순차적으로 읽어라

나는 다시 요청했습니다.

"기획서를 최대한 자세하게 읽어라. **반드시 문서 번호대로 순차적으로 읽어라.**"

그리고 개발 문서 작성도 순서대로:

**기획서 > 엔티티 > ERD > 데이터 모델 > API, 비즈니스 로직 > 데이터베이스**

이렇게 해라를 요청했습니다.

몇 번의 복수 검증을 토대로 최종 문서 작성을 완료한 후 개발을 시작했습니다.

## 개발 방법도 바꿨습니다

이때 개발 방법도 바꿨습니다.

보통의 개발은 기획 리뷰 후 인터페이스가 정의되면 **백엔드와 프론트가 각각 개발을 시작**하고 디자인이 완성되면 프론트가 반영하는 방식이었습니다.

저도 일해온 경험대로 AI에게도 동시 개발을 주문했습니다. 개발 가이드 문서가 있으니까요!

하지만, 그 방법을 바꿨습니다.

**"백엔드가 탄탄한 게 먼저다."**

물론 단번에 잘 되진 못했습니다. 그래도 다시 시작하는 걸 반복하던 끝에 어느 정도 백엔드 개발이 진행되던 중...

## Spec-kit을 알게 되다

**Spec-kit**을 알게 되었습니다.

기존 문서를 Spec-kit에 맞춰서 마이그레이션 했더니, 생각보다 **기획 문서에 유실된 내용**이 있더라고요.

항상 수시로 변경사항이 발생할 때마다 업데이트를 했지만, 부족했나 봅니다.

유실된 내용을 채우면서, 기존 기획의 방향도 수정을 하게 되었습니다.

그렇게 문서를 다시 재정의하고 현재 소스를 수정 개발을 진행한 끝에 **지금의 백엔드**가 된 것입니다.

## 프론트는 수월하게

프론트는 API와 관련 문서의 완성도가 높으니까 수월하게 진행되는 중입니다.

나는 틀린 점을 지적해서 수정을 요청하는 거죠.

기술에 대한 지식이 없어서 수시로 **소스 코드 품질에 대해 검토하도록 요청**하고, 개발 가이드는 반드시 지켜야 할 사항들을 개발 규칙으로 정해뒀습니다.

기술 지식이 없어도 지금까지의 경험으로 **한 파일에 너무 많은 걸 담는다던가, 중복된 소스 코드나 재사용이 안 된다던가** 하는 정도는 볼 수 있으니까요. ㅎ

## 나의 역할

Fallingo가 어떻게 개발되는지 알겠죠?

나의 역할은:
- **기획에 맞는가?**
- **AI가 작성한 소스 코드를 끊임없이 의심하며 요청하는 것**
- **UI/UX에 대해 고민하고 결정하는 것**

물론 항상 AI에 더 나은 방법을 제안받아서 결정하고 있습니다.
