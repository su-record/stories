# vica 개발일지: 재개 안정성, SPEC 검증 정리, 실행 피드백 개선

이번 작업은 겉으로 보기에는 작은 수정들이지만, 실제로는 “한 번 돌린 작업이 끝까지 믿을 수 있게 이어지는가”에 초점을 맞춘 정리였습니다. 특히 원샷 실행, SPEC 검증, 게이트 예산, 워커 차단 안내처럼 사용자가 직접 체감하는 경계 지점들을 다듬었습니다.

## 세션이 없어도 이어갈 수 있게 만들기

가장 먼저 손본 부분은 재개 로직입니다. 기존에는 세션 정보가 없으면 디스크에 SPEC이 남아 있어도 정상적으로 이어가기 어려운 구멍이 있었습니다. 일반적인 흐름에서는 세션이 살아 있으니 잘 드러나지 않지만, 원샷 런처럼 한 번 실행하고 프로세스가 끊기는 방식에서는 이 문제가 꽤 치명적이었습니다.

이번에는 “세션이 없더라도 디스크에 유효한 SPEC이 있다면 재개 가능한 상태로 본다”는 쪽으로 기준을 보강했습니다. 덕분에 실행 중간에 끊겼거나, 다음 단계에서 다시 이어받아야 하는 상황에서도 불필요하게 처음부터 다시 시작하지 않게 됐습니다.

개인적으로 이런 류의 버그는 재현이 애매해서 더 찜찜합니다. 평소에는 잘 돌아가다가 특정 실행 방식에서만 허점이 보이기 때문입니다. 그래서 이번 수정은 단순한 예외 처리라기보다, vica가 상태를 판단하는 기준을 조금 더 현실적인 쪽으로 맞춘 작업에 가깝습니다.

## SPEC 검증 전 정리 과정을 더 안전하게

두 번째 축은 SPEC 체크 주변의 정리 로직입니다. 검증 전에 불필요한 흔적을 치우는 과정이 있었는데, 이 정리가 의도보다 넓게 작동할 가능성이 있었습니다. 특히 “파괴적 정리”에 해당하는 프리픽스 처리는 조심해야 합니다. 검증을 깨끗하게 만들려다가, 오히려 작성되면 안 되는 영역에 영향을 주면 안 되니까요.

이번에는 작성 금지 조항을 더 분명히 하고, 검증 직전에 결정론적으로 정화되는 흐름을 정리했습니다. 여기서 중요한 건 “어쩌다 깨끗해지는 것”이 아니라 “항상 같은 기준으로 깨끗해지는 것”입니다. SPEC 검증은 결과가 흔들리면 신뢰를 잃기 쉽기 때문에, 입력 상태를 안정적으로 만드는 일이 생각보다 중요합니다.

정리하자면, 검증 전에는 필요한 것만 예측 가능하게 치우고, 건드리면 안 되는 것은 명확히 보호하도록 방향을 잡았습니다. 이런 부분은 기능 추가보다 티가 덜 나지만, 나중에 디버깅 시간을 크게 줄여주는 투자라고 생각합니다.

## 실행 흐름의 작은 불편함 줄이기

마지막으로는 잔여 개선들을 묶어서 처리했습니다. 먼저 게이트 예산 계상을 더 정직하게 맞췄습니다. 게이트가 실제로 얼마를 쓰고 있는지, 어디까지를 비용으로 봐야 하는지 애매하면 실행 결과를 해석하기 어려워집니다. 이번 수정으로 예산 판단이 조금 더 실제 동작에 가깝게 맞춰졌습니다.

워커가 차단됐을 때의 안내도 다듬었습니다. 단순히 “막혔다”는 사실만 알려주면 사용자는 다음에 뭘 해야 할지 알기 어렵습니다. 그래서 차단 상황을 더 이해하기 쉽게 전달하도록 메시지 방향을 개선했습니다. 에러 메시지는 결국 UI의 일부라는 생각을 다시 하게 됩니다.

또 하나는 증명 경계에서 체크포인트 커밋을 남기는 부분입니다. 검증이나 증명 단계처럼 의미 있는 경계에서는 상태가 명확히 남아야 추적이 쉽습니다. 문제가 생겼을 때 “어디까지는 확실했는지”를 확인할 수 있기 때문입니다. 이번 변경으로 그런 경계가 조금 더 분명해졌습니다.

## 마무리

이번 3개 작업은 모두 vica의 실행 신뢰도를 높이는 데 맞닿아 있습니다. 세션이 없어도 재개할 수 있어야 하고, SPEC 검증은 안전하고 결정론적이어야 하며, 실행 중 차단이나 예산, 증명 경계도 사용자가 납득할 수 있게 드러나야 합니다.

크게 화려한 기능은 아니지만, 이런 부분들이 쌓여야 도구를 오래 믿고 쓸 수 있다고 봅니다. 다음 작업에서도 새 기능을 얹기 전에, 실행 흐름이 흔들리지 않는지 계속 점검해보겠습니다.
