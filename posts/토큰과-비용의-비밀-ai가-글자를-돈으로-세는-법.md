AI 요금은 글자 수가 아니라 모델이 잘라 낸 조각 수로 계산됩니다.

처음엔 긴 프롬프트만 줄이면 되는 줄 알았습니다. 실제로는 그보다 복잡했습니다.
모델은 문장을 사람이 읽는 단어 그대로 보지 않습니다. 토큰이라는 조각으로 읽습니다.

---

## 글자 하나가 그대로 1원이 되지 않습니다

![토큰과 비용의 비밀: AI가 글자를 돈으로 세는 법](/images/토큰과-비용의-비밀-ai가-글자를-돈으로-세는-법/fig1.png)


토큰은 문자, 단어, 단어 일부, 기호가 섞인 단위입니다.
영어 `automation`은 한 토큰일 수 있지만, 한국어 `자동화했습니다`는 여러 조각으로 나뉠 수 있습니다.

공백, 줄바꿈, JSON 괄호, 마침표도 토큰이 될 수 있습니다.
같은 1,000자라도 언어, 형식, 특수문자에 따라 토큰 수가 달라지는 이유입니다.

| 구분 | 모델이 읽는 것 | 비용에 미치는 영향 |
|---|---|---|
| 입력 토큰 | 시스템 프롬프트, 사용자 질문, 문서, 대화 기록 | 매 요청마다 먼저 과금됩니다 |
| 출력 토큰 | 모델이 새로 생성한 답변 | 길게 쓰게 할수록 바로 늘어납니다 |
| 컨텍스트 | 이전 대화와 참고 자료 | 필요 없는 기록도 함께 읽으면 비용이 붙습니다 |
| 재시도 | 실패 후 다시 보낸 요청 | 같은 입력을 반복해서 태웁니다 |

자주 놓치는 쪽은 출력입니다.
AI가 답을 생성하는 동안에도 토큰이 쓰이고, 생성된 모든 토큰에 가격이 붙습니다.

---

## 비용 공식은 단순하지만, 청구서는 그렇지 않습니다

토큰 비용 공식은 단순합니다.
입력 토큰 가격과 출력 토큰 가격을 따로 계산한 뒤 더하면 됩니다.

```text
비용 = 입력 토큰 / 1,000,000 × 입력 단가
     + 출력 토큰 / 1,000,000 × 출력 단가
```

입력 100만 토큰당 1달러, 출력 100만 토큰당 3달러인 모델이 있다고 가정해 보겠습니다.
한 번의 작업이 입력 20,000토큰, 출력 2,000토큰이면 비용은 0.026달러입니다.

| 항목 | 토큰 수 | 단가 | 비용 |
|---|---:|---:|---:|
| 입력 | 20,000 | $1 / 1M | $0.020 |
| 출력 | 2,000 | $3 / 1M | $0.006 |
| 합계 | 22,000 | - | $0.026 |

이 작업을 하루 1,000번 돌리면 하루 26달러입니다.
30일이면 780달러입니다. 프롬프트가 두 배로 커지면 월 1,560달러가 됩니다.

```python
# 정확한 값은 사용하는 모델의 tokenizer와 요금표로 다시 계산합니다.
def estimate_cost(input_tokens, output_tokens, input_price, output_price):
    return (input_tokens / 1_000_000 * input_price) + (output_tokens / 1_000_000 * output_price)

daily = estimate_cost(20_000, 2_000, 1, 3) * 1000
print(round(daily, 2))  # 26.0
```

2026년 2월 OpenRouter 기준 주간 처리량은 13.95조 토큰이었습니다.
2025년 3월 1.24조 토큰에서 1년 만에 10배 이상 늘었습니다.

단가가 내려가도 전체 사용량이 더 빨리 늘면 청구서는 줄지 않습니다.
2026년에 AI가 점점 싸진다는 상식이 흔들린 이유가 여기에 있습니다.

## 자동화 에이전트에는 숨은 반복문이 있습니다

채팅 한 번의 비용 구조는 단순합니다.
하지만 Google Gemini Spark 같은 노코드 자동화 도구는 태스크, 스킬, 스케줄을 따라 계속 움직입니다.

Gmail을 읽고, 캘린더를 확인하고, 서드파티 앱에 기록하고, 다시 요약합니다.
사용자가 자는 동안에도 백그라운드에서 24시간 돌면 비용은 질문 한 번의 문제가 아니게 됩니다.

sutory 같은 hermes 기반 콘텐츠 자동화 시스템도 같은 문제를 만납니다.
토리와 미니가 수집, 요약, 초안, 검수 단계를 돌면 각 단계가 컨텍스트를 다시 읽습니다.

| 단계 | 예시 입력 | 숨은 비용 |
|---|---|---|
| 수집 | 원문 5개, 검색 결과, 지시문 | 원문 전체를 매번 넣으면 입력이 커집니다 |
| 요약 | 원문과 스타일 가이드 | 고정 지시문이 반복 과금됩니다 |
| 초안 | 요약, 구조, 톤, 예시 | 출력 길이가 비용을 끌어올립니다 |
| 검수 | 초안 전체, 체크리스트 | 전체 재작성 요청이 토큰을 다시 씁니다 |

각 단계가 8,000 입력 토큰을 쓰고 4단계를 거치면 출력 전부터 32,000토큰입니다.
하루 100건이면 입력만 320만 토큰입니다.

## 토큰을 줄이는 프롬프트는 짧은 프롬프트가 아닙니다

짧게 쓰면 싸질 것 같지만, 모호한 지시는 재시도를 부릅니다.
출력 형식, 분량, 톤, 예시를 먼저 정해 두면 토큰 낭비가 줄어듭니다.

| 나쁜 지시 | 나은 지시 | 줄어드는 비용 |
|---|---|---|
| 자세히 설명해줘 | 800자 이하, 표 1개, 불릿 5개로 정리합니다 | 출력 토큰 |
| 위 자료 참고해줘 | 관련 문단 3개만 사용하고, 없으면 모른다고 답합니다 | 입력 토큰 |
| JSON으로만 줘 | 키는 title, summary, actions만 사용합니다 | 후처리 재시도 |
| 자연스럽게 써줘 | 평서체, 문장 20단어 이하, 예시는 2개만 둡니다 | 수정 요청 |

실무에서는 네 가지를 먼저 봅니다.
첫째, 매번 붙는 시스템 프롬프트와 스타일 가이드를 줄입니다.
둘째, 긴 문서는 전체가 아니라 관련 문단 3~5개만 넣습니다.

셋째, 전체 재작성 대신 변경된 문단만 요청합니다.
넷째, 출력 길이 제한과 중단 조건을 명시합니다.

서빙 단계에서도 차이가 큽니다.
2026년 5월 기준 오픈 모델은 양자화, 배칭, KV 캐시, 투기적 디코딩, 서빙 프레임워크에 따라 토큰당 비용이 크게 달라집니다.

사용자 프롬프트만 다듬어도 비용은 줄어듭니다.
직접 모델을 운영한다면 인프라 최적화도 같은 무게로 봐야 합니다.

## 청구서가 커졌을 때 먼저 볼 것

모델을 바꾸기 전에 로그부터 봐야 합니다.
평균값만 보면 놓치는 게 많습니다. p50, p95, 실패 재시도 횟수를 따로 봐야 합니다.

| 점검 항목 | 확인 질문 |
|---|---|
| 입력 토큰 | 매 요청에 같은 문서가 반복해서 붙는가 |
| 출력 토큰 | 필요한 길이보다 길게 생성하고 있는가 |
| 재시도 | 형식 오류 때문에 같은 요청을 다시 보내는가 |
| 컨텍스트 | 오래된 대화 기록을 계속 들고 가는가 |
| 스케줄 | 밤새 필요 없는 작업이 반복 실행되는가 |

토큰은 기술 용어이면서 동시에 비용 항목입니다.
청구서가 커졌다면 모델보다 입력 구조, 출력 길이, 재시도 횟수부터 보는 편이 빠릅니다.
