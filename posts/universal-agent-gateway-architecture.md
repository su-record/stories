## 봇을 세 개 만들면 문제가 세 번 생깁니다

![여러 채널을 하나의 게이트웨이로 모아 같은 에이전트에 연결](/images/universal-agent-gateway-architecture/01.png)
*하나의입구*


처음에는 텔레그램 봇 하나면 충분하다고 생각했습니다. 아니었습니다.

팀 알림은 슬랙에 있었고, 커뮤니티는 디스코드에 있었습니다. 개인 테스트는 텔레그램에서 했습니다. 같은 질문을 세 곳에서 받게 됐습니다.

문제는 모델이 아니었습니다. 입구가 문제였습니다. 채널마다 봇을 따로 만들면 프롬프트, 권한, 메모리, 로그가 모두 갈라집니다.

| 방식 | 에이전트 | 세션 | 메모리 | 운영 부담 |
|---|---:|---:|---:|---:|
| 플랫폼별 봇 | 3개 | 3개 | 3개 | 배포·로그·권한이 분리됨 |
| 게이트웨이 | 1개 | 1개 | 1개 | 채널만 어댑터로 분리됨 |

2026년 5월 Hermes Agent 멀티 플랫폼 가이드는 이 문제를 정면으로 다룹니다. Telegram, Discord, Slack을 같은 에이전트와 도구에 연결하는 방식입니다.

Hermes v0.16 문서도 같은 구조를 설명합니다. 여러 플랫폼을 동시에 켜도 에이전트, 세션, 메모리, 모델은 하나를 씁니다.

---

## 게이트웨이는 채팅 앱을 에이전트 뒤에 숨깁니다

![세션을 공유하기 전에 신원을 먼저 검증해 묶기](/images/universal-agent-gateway-architecture/02.png)
*신원먼저*


게이트웨이는 텔레그램, 디스코드, 슬랙을 서로 다른 제품으로 보지 않습니다. 모두 메시지 이벤트로 봅니다.

입력 형식은 다릅니다. 텔레그램은 chat_id를 보냅니다. 디스코드는 guild_id와 channel_id를 보냅니다. 슬랙은 team_id와 thread_ts를 보냅니다.

게이트웨이가 할 일은 이 차이를 표준 이벤트로 바꾸는 것입니다. 에이전트는 플랫폼 이름보다 사용자 의도와 세션을 봅니다.

```yaml
gateway:
  agent: ops-agent
  memory_store: postgres://agent-memory
  session_store: redis://agent-session

platforms:
  telegram:
    enabled: true
    token: ${TELEGRAM_BOT_TOKEN}
  discord:
    enabled: true
    token: ${DISCORD_BOT_TOKEN}
    intents: [messages, guilds]
  slack:
    enabled: true
    bot_token: ${SLACK_BOT_TOKEN}
    signing_secret: ${SLACK_SIGNING_SECRET}

routing:
  default_agent: ops-agent
  share_memory: true
  require_identity_binding: true
```

이 설정에서 중요한 줄은 `share_memory: true`입니다. 텔레그램에서 장애 원인을 물었던 사용자가 슬랙에서 이어서 배포 로그를 물을 수 있습니다.

Hermes Gateway는 20개 이상 메시징 플랫폼을 하나의 프로세스로 연결한다고 설명합니다. 리뷰 자료에서는 27개 이상 플랫폼 지원도 언급됩니다.

숫자보다 중요한 건 같은 상태를 유지하는 일입니다. 상태가 갈라지면 에이전트는 같은 사람을 세 사람으로 착각합니다.

## 같은 에이전트가 다르게 보이면 실패입니다

![같은 에이전트라도 플랫폼별 도구 권한은 섞지 않기](/images/universal-agent-gateway-architecture/03.png)
*권한분리*


게이트웨이를 붙였다고 끝나는 문제는 아닙니다. 텔레그램 답변과 슬랙 답변이 다르면 사용자에게는 다른 봇처럼 보입니다.

저는 세 가지를 먼저 고정합니다. 시스템 프롬프트, 도구 권한, 메모리 정책입니다. 이 셋이 흔들리면 채널 통합은 껍데기만 남습니다.

| 항목 | 고정해야 하는 값 | 깨졌을 때 증상 |
|---|---|---|
| 시스템 프롬프트 | 역할, 금지 행동, 응답 톤 | 채널마다 말투와 판단이 달라짐 |
| 도구 권한 | 조회·쓰기·배포 권한 | 슬랙에서는 배포하고 텔레그램에서는 실패함 |
| 메모리 정책 | 저장 범위, 만료 시간 | 이전 대화를 기억하거나 잊는 기준이 달라짐 |
| 감사 로그 | user_id, platform, tool_call | 사고 뒤에 원인을 추적하지 못함 |

운영 에이전트를 예로 들 수 있습니다. 디스코드에서는 고객 커뮤니티 질문을 받고, 슬랙에서는 내부 장애 대응을 합니다.

같은 에이전트라도 도구 권한은 달라야 합니다. 디스코드 사용자는 장애 목록을 볼 수 있습니다. 슬랙 SRE는 롤백 도구를 실행할 수 있습니다.

그래서 라우팅은 단순히 메시지를 넘기는 코드가 아닙니다. “누가, 어디서, 어떤 권한으로 말했는가”를 확정하는 절차입니다.

## 세션보다 먼저 신원을 묶어야 합니다

OpenClaw는 Telegram, WhatsApp, Discord, Slack을 하나의 Gateway와 세션 저장소로 연결합니다. 여기서 identity binding을 별도로 다룹니다.

이유는 분명합니다. 텔레그램의 `12345`와 슬랙의 `U12345`는 같은 사람이 아닙니다. 문자열이 비슷해도 신원이 증명된 것은 아닙니다.

잘못 묶으면 개인 메모리가 새어 나갑니다. 더 나쁘게는 내부 도구 권한이 외부 채널로 번질 수 있습니다.

```json
{
  "user": "su",
  "bindings": [
    {"platform": "telegram", "id": "tg_8342", "verified_at": "2026-07-19"},
    {"platform": "slack", "id": "U09ABCD", "verified_at": "2026-07-19"},
    {"platform": "discord", "id": "dc_5521", "verified_at": "2026-07-19"}
  ],
  "roles": ["operator"],
  "tool_scope": ["read_logs", "create_ticket"]
}
```

저는 바인딩 없는 사용자를 기본적으로 익명으로 봅니다. 익명 사용자는 장기 메모리를 쓰지 않습니다. 쓰기 도구도 실행하지 않습니다.

불편한 정책입니다. 대신 사고 범위를 줄일 수 있습니다. 게이트웨이는 편의를 늘리는 장치이지만, 사고 반경도 함께 넓힙니다.

## AI 에이전트는 싸졌지만, 연결 비용은 남았습니다

2026년 7월 기준 AI 시장에는 큰 자본이 들어왔습니다. GPU와 데이터센터 투자는 수백억 달러 단위로 움직입니다.

모델 가격은 내려가고 있습니다. 중국의 오픈소스 모델 경쟁도 가격을 압박합니다. DeepSeek 이후 저비용 모델 흐름은 더 강해졌습니다.

그렇다고 운영 비용이 사라지는 것은 아닙니다. 메시징 게이트웨이를 붙이면 로그, 권한, 세션, 감사, 장애 대응 비용이 생깁니다.

Anthropic은 2026년 7월 18일 자율형 에이전트 위험을 공식 경고했습니다. 핵심 인프라를 표적으로 삼는 통제 불가능한 행동을 문제로 봤습니다.

그래서 게이트웨이 설계의 첫 질문은 “몇 개 플랫폼을 붙일까”가 아닙니다. “어떤 행동을 절대 허용하지 않을까”입니다.

| 위험 | 방어선 |
|---|---|
| 외부 채널에서 내부 도구 호출 | 플랫폼별 tool allowlist |
| 사용자를 잘못 식별 | 인증 기반 identity binding |
| 메모리 유출 | 민감 정보 저장 금지와 TTL |
| 사고 추적 불가 | platform, user, session, tool_call 로그 |
| 에이전트 폭주 | rate limit, human approval, kill switch |

텔레그램, 디스코드, 슬랙에서 같은 에이전트가 일하는 구조는 이미 문서화되어 있습니다. Hermes와 OpenClaw가 그 방향을 보여줍니다.

남은 일은 연결 자체가 아닙니다. 같은 상태를 유지하면서도 권한을 섞지 않는 일입니다. 게이트웨이가 입구를 하나로 만들어도 책임이 사라지는 건 아닙니다.
