## 최근 작업: “다시 이어가기”와 “검증 자동화”를 더 믿을 수 있게 다듬었습니다

이번 작업은 크게 보면 **작업 재개 흐름**, **검증 스크립트 처리**, 그리고 **워커 실행 안정성**을 정리한 시간이었습니다. 겉으로는 작은 수정처럼 보일 수 있지만, 실제로는 vica가 긴 작업을 끊김 없이 이어가고, 모델이 만든 결과를 게이트가 더 정확하게 검증하도록 맞추는 데 초점을 뒀습니다.

### 설계 정합 상태에서 자연스럽게 이어가기

먼저 `resume` 흐름을 손봤습니다. 기존에는 이미 디스크에 SPEC이 남아 있어도 다시 clarify 단계나 SPEC 작성 단계로 들어갈 여지가 있었습니다. 사람 입장에서는 “이미 설계가 있는데 왜 또 물어보지?” 싶은 상황이었고, 자동화 관점에서도 불필요한 왕복이 생겼습니다.

이번에는 디스크에 있는 SPEC을 더 적극적으로 재사용하도록 정리했습니다. 설계가 이미 정합한 상태라면 clarify와 SPEC 작성 과정을 건너뛰고, 필요한 다음 단계로 바로 넘어가도록 했습니다. 또한 이미 완료된 페이즈는 다시 실행하지 않도록 skip 처리도 보강했습니다.

이 부분은 단순한 편의 기능이라기보다, vica가 “작업의 기억”을 제대로 활용하게 만드는 작업에 가깝습니다. 매번 처음부터 다시 설명하는 도구가 아니라, 이전 맥락을 읽고 합리적으로 이어가는 도구가 되어야 하니까요.

### 검증 heredoc을 파일로 바꿔 실행하기

두 번째로는 검증 커맨드에서 heredoc을 자동으로 파일화하는 기능을 넣었습니다. 이번 작업에서 꽤 중요하게 본 지점은 **모델이 의도한 검증 방식과 실제 게이트가 실행하는 방식의 정렬**이었습니다.

heredoc은 사람이 보기에는 간단하고 편하지만, 자동화된 검증 게이트에서는 다루기 까다로운 경우가 있습니다. 특히 quoting, shell 차이, stdin 처리 방식 때문에 모델이 생각한 검증과 실제 실행 결과가 어긋날 수 있습니다. 이런 어긋남이 쌓이면 결국 “왜 실패했는지 애매한 grind”가 생깁니다.

그래서 검증 heredoc을 임시 파일로 풀어낸 뒤 실행하는 쪽으로 바꿨습니다. 이렇게 하면 실행 단위가 명확해지고, 게이트가 무엇을 검증하는지도 더 분명해집니다. 개인적으로는 이번 변경이 꽤 마음에 듭니다. 화려한 기능은 아니지만, 자동화 시스템에서 신뢰도를 올리는 데는 이런 지저분한 경계면을 정리하는 일이 정말 중요하다고 느꼈습니다.

### 워커 실행 결함 보강

마지막으로 웨이브 3에서 드러난 워커 쪽 결함들을 정리했습니다. 핵심은 **읽기 전용 bash 처리**, **비용 상한**, 그리고 **스캐폴드 우선 순차 실행**입니다.

워커가 bash를 사용할 때 읽기 전용으로 다뤄야 하는 상황이 있는데, 이 경계가 느슨하면 의도하지 않은 변경 가능성이 생깁니다. 그래서 실행 권한과 사용 방식을 더 보수적으로 잡았습니다. 또한 비용 상한을 둬서 작업이 예상보다 커졌을 때 무한정 밀고 나가지 않도록 했습니다. 자동화가 편하려면, 동시에 멈출 줄도 알아야 한다고 생각합니다.

스캐폴드 우선 순차 실행도 같은 맥락입니다. 기반 구조가 먼저 잡히지 않은 상태에서 병렬적으로 많은 일을 벌이면, 결과가 흔들리기 쉽습니다. 그래서 먼저 뼈대를 세우고, 그 위에 후속 작업이 올라가도록 순서를 조정했습니다.

### 마무리

이번 변경들은 사용자에게 바로 보이는 큰 기능 추가라기보다는, vica가 더 차분하게 일하도록 만드는 내부 정비에 가깝습니다. 이미 있는 설계는 다시 쓰고, 검증은 더 재현 가능하게 만들고, 워커는 권한과 비용 안에서 움직이게 했습니다.

자동화 도구를 만들다 보면 결국 중요한 건 “똑똑해 보이는 한 번의 성공”보다 “덜 흔들리는 반복 실행”이라는 생각이 듭니다. 이번 커밋들은 그 방향으로 한 걸음 더 다가간 작업이었습니다.
