## vica 개발일지: SPEC이 현실을 놓치지 않도록 다듬기

이번 작업은 크게 보면 **SPEC 작성과 리뷰 단계에서 “시간”을 더 정확히 다루도록 만드는 수정**이었습니다. 겉으로는 작은 fix들이지만, 실제로는 vica가 계약처럼 다루는 문서에서 과거와 미래를 헷갈리지 않게 만드는 꽤 중요한 정리였습니다.

### SPEC 작성 단계의 날짜 접지 강화

먼저 SPEC 계층에 **temporal grounding**, 즉 날짜 접지를 넣었습니다. 문제가 된 지점은 이미 과거에 일어난 사건이 SPEC 안에서 마치 앞으로 해야 할 일처럼 계약되는 경우였습니다.

예를 들어 어떤 이슈가 이미 처리되었거나, 특정 날짜 기준으로 완료된 사실인데도 모델이 이를 “예정”, “추후 진행” 같은 형태로 정리해버리면 이후 단계가 전부 잘못된 전제 위에서 움직이게 됩니다. vica에서는 SPEC이 후속 판단의 기준점이 되기 때문에, 여기서 시간이 어긋나면 단순한 표현 실수가 아니라 실행 계획 자체가 틀어질 수 있습니다.

그래서 SPEC을 만들 때 현재 기준일을 명확히 접지하고, 과거 사건을 미래 계약처럼 다루지 않도록 막았습니다. 이번 수정은 “무엇을 할 것인가”만큼이나 “이미 무엇이 일어났는가”를 정확히 구분하는 쪽에 초점을 맞췄습니다.

### 최종 리뷰 심판에도 같은 시간 기준 적용

SPEC 단계만 고친다고 끝나는 문제는 아니었습니다. 최종 리뷰를 맡는 심판 단계에서도 비슷한 일이 생겼습니다. 여기서는 모델이 학습 과정에서 얻은 일반적인 프라이어, 그러니까 “보통은 이런 흐름일 것이다”라는 감각에 기대면서 실제 날짜 기반 진실을 가리는 문제가 있었습니다.

조금 솔직히 말하면, 이 부분은 꽤 AI다운 실패였습니다. 문맥상 그럴듯한 판단이 실제 사실보다 앞서버린 것입니다. 그래서 최종 리뷰 심판에도 날짜 접지를 적용했습니다. 이제는 리뷰 단계에서도 “그럴듯한 이야기”보다 “해당 날짜 기준으로 맞는 이야기”를 우선하도록 구조를 보강했습니다.

이 수정은 단순히 날짜 문자열 하나를 넘겨주는 정도가 아니라, 판단자가 어떤 시간축 위에서 판단해야 하는지를 명시해주는 작업에 가깝습니다. SPEC에서 바로잡은 시간 기준이 최종 검증에서도 유지되어야 전체 파이프라인이 일관되게 동작할 수 있기 때문입니다.

### 장문 SPEC 절단 문제와 maxTokens 조정

마지막으로 SPEC 작성에서 사용하는 maxTokens를 2000에서 8000으로 늘렸습니다. 클로드가 긴 SPEC을 작성할 때 출력이 중간에 잘리면서 fail-closed로 이어지는 문제가 있었습니다.

fail-closed 자체는 안전한 방향의 실패라서 나쁜 선택은 아니지만, 실제로는 충분히 작성 가능한 SPEC이 토큰 제한 때문에 실패 처리되는 상황이었습니다. 특히 최근처럼 SPEC 안에 날짜 기준, 상태 구분, 검증 조건이 늘어나면 2000 토큰은 생각보다 빠듯했습니다.

그래서 이번에는 제한을 넉넉히 늘려 장문 SPEC이 중간에 끊기지 않도록 했습니다. 물론 토큰을 늘리는 건 비용과 지연 시간 측면에서 부담이 될 수 있습니다. 하지만 SPEC은 후속 처리의 기준 문서이기 때문에, 여기서 잘려나간 정보 때문에 잘못 실패하는 것보다는 안정적으로 완성하는 편이 맞다고 판단했습니다.

### 정리

이번 변경들은 모두 vica가 더 “현실 기준”으로 판단하도록 만드는 작업이었습니다. 날짜를 명확히 고정하고, 과거와 미래를 섞지 않으며, 긴 SPEC도 끝까지 작성되도록 했습니다.

작게 보면 프롬프트와 설정값 조정에 가까운 수정들이지만, 실제 효과는 신뢰성 쪽에 있습니다. 특히 AI가 그럴듯한 말로 빈틈을 메우는 습관을 줄이고, 명시된 시간과 사실을 기준으로 움직이게 만드는 방향이라 의미가 컸습니다. 앞으로도 vica에서는 멋진 답변보다 덜 틀리는 답변, 그리고 계약으로 삼아도 안전한 문서를 만드는 쪽에 계속 무게를 두려고 합니다.
