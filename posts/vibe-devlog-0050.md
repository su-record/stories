# vibe 개발일지 #12 - 복잡성을 걷어내고 '하네스' 본질로 돌아가기 (25개 커밋)

**작업 기간**: 2026-06-30 ~ 2026-07-02

안녕하세요, `vibe`를 개발하고 있는 10년 차 프론트엔드 개발자 Su입니다. 

최근 Claude Code, Gemini CLI 등 프론티어 AI 모델들의 코딩 능력이 눈부시게 발전하고 있습니다. 이제 모델들은 굳이 복잡한 프롬프트 엔지니어링이나 세밀한 컨텍스트 압축 없이도 훌륭하게 코드를 짜냅니다. 

이러한 흐름 속에서 문득 `vibe` 코드를 되돌아보았습니다. **"Agent = Model + Harness"**가 우리의 철학인데, 정작 `vibe`가 모델의 역할(Orchestration, Context Compression)까지 지나치게 대신하려 하며 비대해져 있었더군요. 

그래서 지난 3일간, 과거의 내가 밤새워 짰던 코드들을 눈물 머금고 과감하게 잘라내는 **대대적인 '다이어트'와 아키텍처 체질 개선(v3)**을 단행했습니다. 

---

## 📝 이번 기간 작업 내용

이번 25개 커밋의 핵심 키워드는 **'단순화(Simplification)'**와 **'경량화(Slimming)'**입니다.

### 1. 자체 오케스트레이터 및 컨텍스트 도구 완전 제거 (-6,200+ LOC)
모델이 충분히 똑똑해진 현시점에서 자체 구현한 오케스트레이션은 오히려 독이 되었습니다.
* **관련 커밋**: `refactor!: remove self-built orchestrator layer (6.2k LOC)`, `refactor!: remove semantic tools, ContextCompressor, TokenBudgetTracker`
* **상세**: 6,200줄에 달하는 오케스트레이터 코드와 복잡한 토큰 버킷 트래커, 컨텍스트 압축기를 과감히 삭제했습니다. 이제 복잡한 라우팅은 모델의 선천적 능력에 맡깁니다.

### 2. 에이전트 및 스킬 통폐합 (76개 → 15개)
관리하기 힘들 정도로 파편화되어 있던 에이전트와 워크플로우를 목적 중심으로 압축했습니다.
* **관련 커밋**: `refactor(agents): consolidate 76 agent files into 15 goal-oriented agents`, `refactor(skills): collapse interview→plan→spec→review into single-pass SPEC`
* **상세**: 76개나 되던 에이전트 파일을 15개의 정예 에이전트(SSOT)로 통폐합했습니다. 또한, 기존의 지루했던 `인터뷰 -> 계획 -> 스펙 제안 -> 리뷰` 4단계 파이프라인을 단 한 번에 끝내는 **싱글 패스 SPEC(single-pass SPEC)**으로 개편하여 속도를 획기적으로 높였습니다.

### 3. '조용하고 안전한' 훅(Hooks) 시스템으로의 회귀
매 세션마다 개발자의 코드 생성을 방해하거나 강제하던 검증 게이트와 기능들을 Opt-in(선택 참여) 방식으로 변경했습니다.
* **관련 커밋**: `refactor(hooks): freeze evolution engine — opt-in only`, `refactor(hooks): drop heuristic detectors from code-check, unhook commit gate`
* **상세**: 자동으로 엔진을 진화시키던 'Evolution Engine'을 동결하고 원할 때만 실행되도록 바꿨습니다. 개발자가 코드를 커밋할 때마다 참견하던 휴리스틱 디텍터와 커밋 게이트, 시끄러운 경고 배너 훅도 제거했습니다. 개발을 방해하지 않는 것이 진짜 하네스니까요.

### 4. 핵심 검증 엔진(vibe.verify)의 다이어트
* **관련 커밋**: `refactor(skills): slim vibe.verify 28KB→6KB, tame auto-trigger sweep`
* **상세**: 자동 트리거 범위를 좁히고 불필요한 의존성을 정리하여, 핵심 검증 엔진의 크기를 **28KB에서 6KB로 약 78% 줄였습니다.**

## 💡 작업 하이라이트

### 76개의 에이전트를 15개로 통폐합, 그리고 v3 아키텍처로의 전환

이전 버전의 `vibe`는 지나치게 세분화된 에이전트 구조를 가지고 있었습니다. 피그마 디자인을 보고 스펙을 뽑는 에이전트 따로, 리뷰하는 에이전트 따로... 그러다 보니 컨텍스트 전송량도 늘어나고 유지보수가 지옥 같았습니다. 

이번 v3 아키텍처 개편을 통해 **"모델은 믿되, 결과물만 하네스(Harness)로 확실하게 묶어 검증한다"**는 본질로 회귀했습니다.

```
[구형 v2 아키텍처: 과도한 개입]
개발자 ──> [76개 에이전트 & 자체 오케스트레이터 (6.2k LOC)] ──> [컨텍스트 압축] ──> Model ──> 불안정한 결과

[신형 v3 아키텍처: 단순하고 확실한 하네스]
개발자 ──> Model (Claude Code 등 자체 역량 활용) ──> [vibe v3 하네스 (15개 에이전트 & 6KB Verify)] ──> 안심하고 배포 (Done)
```

이 개편 덕분에 불필요한 API 호출 비용이 대폭 절감되었고, 하이크/소넷/오푸스 등 모델 등급별로 구차하게 나누어 두었던 비용 라우팅 룰(`docs: retire Haiku/Sonnet/Opus cost-routing doctrine`)도 모두 던져버릴 수 있게 되었습니다.

## 📊 개발 현황

* **버전 변화**: `v2.16.3` ➔ `v2.16.4` (최적화 아키텍처 배포 완료)
* **코드 라인 수 변화**: 약 **-8,500줄** (쓸데없는 복잡성 제거)
* **에이전트 수**: 76개 ➔ **15개 정예 생존자(Survivors)**
* **핵심 파일 크기**: `vibe.verify` 기준 28KB ➔ **6KB** (78% 경량화)

## 🧑‍💻 개발 후기: 비우는 용기

개발자로서 애지중지 만들었던 기능을 삭제하는 것만큼 쓰린 일은 없습니다. '언젠가 쓰겠지' 하며 만들었던 토큰 트래커, 세부 오케스트레이터 코드를 지우면서 참 많은 생각이 들었습니다.

하지만 뛰어난 AI 코딩 도구들이 쏟아져 나오는 지금, `vibe`가 살 길은 '스스로 똑똑한 척하는 오케스트레이터'가 되는 것이 아니라, 개발자가 가져온 결과물이 깨지지 않도록 단단하게 잡아주는 **'가장 가볍고 확실한 하네스(Verification Harness)'**가 되는 것임을 뼈저리게 느꼈습니다.

훨씬 가벼워지고 빨라진 `@su-record/vibe` v3를 경험해 보세요. 이제 진정한 'Vibe Coding'에 안전벨트가 채워집니다.
