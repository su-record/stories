## 프롬프트 하나로 직원 셋을 만들 수 없었습니다

![프로필은 말투가 아니라 권한·기억·도구의 격리 단위](/images/creating-ai-employee-profile/01.png)
*역할은격리*


처음에는 개발, 글쓰기, 리서치를 한 채팅방에서 처리했습니다. 금방 문제가 생겼습니다.

코드 리뷰를 하다가 블로그 문체 지시가 끼어들었습니다. 리서치 메모는 배포 체크리스트를 밀어냈습니다. 나중에는 같은 질문에도 서로 다른 기준으로 답했습니다.

LLM은 수십억에서 수조 개 매개변수를 가진 모델입니다. 위키피디아, 뉴스, 웹, 서적, 코드로 사전 학습합니다.

3Blue1Brown의 LLM 설명 영상은 680만 회 이상 조회됐습니다. 이제 많은 사람이 모델이 대략 어떻게 작동하는지 이해하기 시작했습니다.

문제는 그다음입니다. 모델이 똑똑해져도 업무 경계가 없으면 조직 안에서는 사고가 납니다.

| 운영 방식 | 한 채팅방 운영 | 프로필 운영 |
|---|---|---|
| 역할 | 대화 중 계속 바뀜 | 개발, 콘텐츠, 리서치로 고정 |
| 권한 | 모든 도구가 한곳에 몰림 | API 키와 게이트웨이 분리 |
| 기억 | 모든 맥락이 섞임 | MEMORY.md, USER.md 분리 |
| 실패 범위 | 한 실수가 전체 업무에 전파 | 해당 역할 안에서만 손상 |
| 온보딩 | 매번 긴 설명 필요 | 프로필 실행으로 즉시 시작 |

AI 직원을 만든다는 말은 성격을 꾸민다는 뜻이 아닙니다. 책임, 권한, 기억을 분리한다는 뜻입니다.

---

## 프로필은 페르소나가 아니라 격리 단위입니다

![스킬은 절차이고 메모리는 장기 업무 맥락이다](/images/creating-ai-employee-profile/02.png)
*절차와기억*


Hermes Agent 문서에서 프로필은 독립 에이전트를 실행하는 단위입니다. config, API keys, memory, sessions, skills, gateway state를 나눕니다.

이 차이는 큽니다. 개발 에이전트는 GitHub와 테스트 실행 권한을 가집니다. 콘텐츠 에이전트는 Obsidian Wiki와 블로그 저장소만 봅니다.

Telegram으로 같은 사람에게 말을 걸어도 내부 상태는 다릅니다. 겉으로는 채팅이지만, 뒤에서는 서로 다른 직원이 답하는 구조입니다.

```text
.hermes/
  profiles/
    dev/
      config.yaml
      MEMORY.md
      USER.md
      sessions/
      skills/
        code-review.md
        test-run.md
    content/
      config.yaml
      MEMORY.md
      USER.md
      sessions/
      skills/
        blog-outline.md
        style-check.md
    research/
      config.yaml
      MEMORY.md
      USER.md
      sessions/
      skills/
        source-digest.md
        claim-log.md
```

핵심은 파일 구조 자체가 아닙니다. 같은 모델을 쓰더라도 직원의 근무지가 달라진다는 점입니다.

개발 프로필에는 운영 배포 키를 줄 수 있습니다. 콘텐츠 프로필에는 주면 안 됩니다. 리서치 프로필은 쓰기 권한 없이 읽기만 해도 충분할 수 있습니다.

프로필은 프롬프트보다 낮은 층에 있습니다. 프롬프트는 말투를 바꿉니다. 프로필은 접근 가능한 세계를 바꿉니다.

## 스킬과 메모리는 다른 층입니다

![기억은 자산이지만 섞이거나 오래되면 부채가 된다](/images/creating-ai-employee-profile/03.png)
*섞이면부채*


스킬은 업무 매뉴얼입니다. 필요할 때 로드되는 지식 문서이고, 절차적 메모리로 작동합니다.

Hermes Agent에서는 에이전트가 스킬을 만들고 수정할 수 있습니다. 편하지만 위험합니다.

잘못된 절차가 스킬에 저장되면 다음 실행에서도 반복됩니다. 한 번의 착각이 업무 규칙으로 굳어집니다.

| 항목 | 역할 | 예시 | 관리 기준 |
|---|---|---|---|
| USER.md | 사용자 프로필 | Su는 짧은 문장과 평서체를 선호 | 거의 고정 |
| MEMORY.md | 장기 업무 기억 | 블로그 frontmatter 규칙, 배포 주의점 | 주기적 검토 |
| skills | 절차와 체크리스트 | 코드 리뷰 7단계, 출처 검증 순서 | 변경 이력 필요 |
| sessions | 대화와 실행 기록 | 오늘 처리한 PR, 작성한 초안 | 역할별 보관 |

예를 들어 콘텐츠 프로필의 USER.md에는 문체 기준을 둡니다. ‘평서체, 짧은 문장, 과한 수식어 금지’ 같은 규칙입니다.

MEMORY.md에는 반복되는 작업 맥락을 둡니다. 블로그 카테고리, frontmatter 형식, 이전 글 링크 규칙이 여기에 들어갑니다.

스킬에는 실행 절차를 둡니다. 아래처럼 하나의 업무를 재사용 가능한 단위로 만듭니다.

```md
# blog-outline

1. 주제에서 독자가 이미 겪은 실패를 먼저 찾는다.
2. H2는 3개에서 5개로 제한한다.
3. 표를 최소 1개 넣는다.
4. 주장마다 수치, 파일명, 날짜 중 하나를 붙인다.
5. 마지막 문장은 슬로건으로 끝내지 않는다.
```

이렇게 나누면 메모리를 지우지 않고도 스킬만 고칠 수 있습니다. 절차는 그대로 두고 사용자 취향만 바꾸는 것도 가능합니다.

## 역할별 분리 예시: 개발, 콘텐츠, 리서치

프로필은 역할이 다를 때 만듭니다. 기분이나 말투가 다르다는 이유로 만들면 관리 비용만 늘어납니다.

제가 보는 기준은 세 가지입니다. 권한이 다르면 분리합니다. 장기 기억이 다르면 분리합니다. 성공 기준이 다르면 분리합니다.

| 프로필 | 주요 권한 | 장기 기억 | 스킬 | 주면 안 되는 권한 |
|---|---|---|---|---|
| dev | GitHub, 테스트, 스테이징 배포 | 코드 규칙, 장애 이력 | code-review, migration-check | 뉴스레터 고객 목록 |
| content | Obsidian, 블로그 저장소 | 문체, 카테고리, 발행 규칙 | blog-outline, style-check | 운영 서버 배포 키 |
| research | 웹 검색, PDF 읽기 | 신뢰 출처, 금지 출처 | source-digest, claim-log | 저장소 쓰기 권한 |

설정 파일에서도 역할이 보여야 합니다. 이름만 content-agent인데 모든 키를 갖고 있다면 분리한 것이 아닙니다.

```yaml
profile: content
model: gemini-3.5-pro
gateway: telegram-content
api_keys:
  - obsidian_wiki
  - github_blog_repo
memory:
  user: USER.md
  persistent: MEMORY.md
sessions: sessions/
skills:
  - blog-outline
  - style-check
  - fact-check
permissions:
  production_deploy: false
  database_write: false
```

2026년 7월 18일 기준으로 모델 경쟁은 더 복잡해졌습니다. Kimi K3, Gemini 3.5 Pro 같은 강한 모델이 계속 나옵니다.

그래도 운영 단위는 모델명이 아닙니다. 오늘 Gemini를 쓰고 내일 Kimi를 쓰더라도 콘텐츠 직원의 메모리와 권한은 유지되어야 합니다.

모델은 엔진입니다. 프로필은 사원증, 책상, 접근 카드, 업무 노트에 가깝습니다.

## 운영 기준: 기억은 자산이지만 섞이면 부채입니다

프로필을 나누면 처음에는 파일이 늘어납니다. config도 늘고 MEMORY.md도 늘고 스킬도 중복됩니다.

불편합니다. 그래도 한 채팅방에서 모든 업무를 처리하는 비용보다는 작습니다.

운영할 때는 아래 다섯 가지만 봅니다.

- API 키가 역할별로 최소화되어 있는가.
- 세션 로그가 프로필 밖으로 새지 않는가.
- 스킬 수정 권한에 검토 절차가 있는가.
- MEMORY.md에 사실과 취향이 섞이지 않는가.
- 30일 동안 쓰지 않은 스킬을 삭제했는가.

가장 많이 놓치는 부분은 기억입니다. AI가 기억한다고 해서 모두 좋은 기억은 아닙니다.

틀린 배포 절차, 오래된 문체 기준, 폐기된 API 주소도 똑같이 기억됩니다. 그래서 메모리는 저장보다 삭제가 중요합니다.

프로필을 나누면 성능보다 운영 문제가 먼저 보입니다. 그때부터 AI는 채팅창이 아니라 관리해야 할 직원 단위가 됩니다.
