AI 에이전트 협업은 역할 이름을 많이 붙이는 일이 아닙니다. 실패가 어디서 멈춰야 하는지 정하는 일에 가깝습니다.

## 표준 용어보다 운영 패턴이 먼저입니다

![협업은 역할명이 아니라 실패 중단선이다](/images/ai-agent-collaboration-patterns/01.png)
*실패 차단*


2026년 7월 1일 기준으로, ‘AI 에이전트 협업 패턴’이라는 고정된 표준 분류는 아직 뚜렷하지 않습니다. 검색 자료를 봐도 LangGraph로 에이전트를 직접 구현하는 예시는 있지만, 협업 패턴 자체를 체계적으로 정리한 출처는 많지 않았습니다.

그래서 기준을 조금 다르게 잡아야 합니다. 에이전트를 사람처럼 부르기보다, 어떤 입력을 받고 어떤 실패를 다음 단계로 넘기는지 봐야 합니다. 협업은 이름보다 인터페이스에 가깝습니다.

AI 에이전트는 보통 네 가지를 가집니다.

| 구성 요소 | 역할 | 실패 지점 |
|---|---|---|
| 태스크 | 해야 할 일 정의 | 목표가 넓으면 결과가 흔들림 |
| 스킬 | 도구 호출, API, 검색 | 권한과 입력 형식에서 실패 |
| 메모리 | 이전 작업과 상태 저장 | 오래된 정보가 재사용됨 |
| 스케줄 | 시간 기반 실행 | 실패가 밤새 반복됨 |

Google Gemini Spark는 이 구조를 제품으로 묶은 사례입니다. 자연어 지시로 태스크, 스킬, 스케줄을 만들고, Gmail과 캘린더 같은 Google Workspace, 서드파티 앱과 연결합니다.

중요한 점은 24시간 백그라운드 실행입니다. 사용자가 자는 동안에도 작업이 돌기 때문에, 설계가 잘못된 에이전트는 실패도 24시간 반복합니다.

## 다섯 가지 협업 패턴으로 나눠 봅니다

![에이전트 워크플로우는 긴 프롬프트보다 상태 그래프다](/images/ai-agent-collaboration-patterns/02.png)
*분기 운영*


에이전트 협업은 대개 다섯 가지 패턴으로 볼 수 있습니다. 제품 이름보다 이 표가 더 중요합니다. 실제 운영에서는 여러 패턴이 한 워크플로우 안에 섞이는 경우가 많습니다.

| 패턴 | 구조 | 잘 맞는 작업 | 위험 신호 |
|---|---|---|---|
| 디스패처-워커 | 한 에이전트가 작업을 분배 | 고객 문의 분류, 리드 배정 | 분류 기준이 자주 바뀜 |
| 플래너-실행자 | 계획과 실행을 분리 | 리서치, 보고서 작성 | 계획이 길고 실행 검증이 없음 |
| 작성자-검토자 | 초안과 리뷰를 분리 | 이메일, 제안서, 코드 리뷰 | 검토자가 같은 오류를 반복 |
| 블랙보드 | 공유 상태에 결과를 누적 | 다중 소스 조사, 장애 분석 | 누가 무엇을 덮어썼는지 모름 |
| 사람 승인 게이트 | 위험 작업 전 사람 확인 | 결제, 삭제, 외부 발송 | 승인 기준이 문서화되지 않음 |

처음 시작하는 팀은 보통 플래너-실행자 구조를 먼저 씁니다. 문제는 그다음에 생깁니다. 실행자가 실패했는데 플래너가 그 실패를 이해하지 못하는 경우입니다.

그래서 검토자와 사람 승인 게이트가 필요합니다. 특히 외부 발송, 데이터 삭제, 비용 발생 작업은 자동 실행보다 중단 조건을 먼저 정해야 합니다.

## Gmail 보고서 자동화로 보면 차이가 보입니다

![에이전트마다 입력·출력·권한·실패 처리 계약이 필요하다](/images/ai-agent-collaboration-patterns/03.png)
*계약 먼저*


예를 들어 매일 오전 7시에 영업 리드 요약을 만드는 워크플로우를 생각해 봅니다. 단일 에이전트에게 ‘Gmail을 읽고 중요한 리드를 정리해줘’라고 시키면 처음에는 그럴듯하게 보입니다.

하지만 세 번째 날쯤부터 문제가 드러납니다. 이미 처리한 메일을 다시 요약합니다. 캘린더 일정과 충돌하는 미팅을 제안합니다. 영업팀에 보내기 전에 민감한 가격 정보를 그대로 넣을 수도 있습니다.

협업 패턴으로 나누면 구조가 달라집니다.

| 단계 | 담당 에이전트 | 입력 | 출력 | 중단 조건 |
|---|---|---|---|---|
| 1 | 스케줄러 | 매일 06:30 | 실행 시작 | 전날 실패 미해결 |
| 2 | 수집자 | Gmail, CRM | 신규 메일 목록 | 중복률 20% 초과 |
| 3 | 분류자 | 메일 본문 | 리드 점수 | 확신도 0.7 미만 |
| 4 | 작성자 | 리드 점수 | 보고서 초안 | 개인정보 포함 |
| 5 | 검토자 | 초안 | 발송 가능 여부 | 가격, 계약 조건 포함 |
| 6 | 알림자 | 승인된 초안 | Slack, Gmail 발송 | 승인자 부재 |

Gemini Spark 같은 노코드 에이전트는 이 구조를 자연어로 만들 수 있습니다. 태스크는 ‘리드 요약’, 스킬은 ‘Gmail 읽기와 Slack 발송’, 스케줄은 ‘평일 06:30 실행’입니다.

다만 자연어로 만든다고 해서 통제가 사라져서는 안 됩니다. 권한, 중복 기준, 승인 조건은 문장으로 남겨야 합니다. 자동화의 품질은 프롬프트보다 중단 조건에서 갈리는 경우가 많습니다.

## 구현은 그래프에 가깝습니다

협업 에이전트는 긴 프롬프트 하나보다 상태 그래프에 가깝습니다. 각 노드는 역할을 맡고, 엣지는 다음 행동을 정합니다. LangGraph 같은 도구가 이 모델과 잘 맞습니다.

간단한 흐름은 이렇게 표현할 수 있습니다.

```python
def route(state):
    if state['duplicate_rate'] > 0.2:
        return 'stop'
    if state['confidence'] < 0.7:
        return 'human_review'
    if state['risk'] >= 7:
        return 'human_review'
    return 'send'

workflow = {
    'collect': ['classify'],
    'classify': ['draft'],
    'draft': ['review'],
    'review': [route]
}
```

이 코드에서 중요한 부분은 모델 호출이 아닙니다. 숫자로 된 중단 기준입니다. 중복률 20%, 확신도 0.7, 위험도 7 같은 기준이 있어야 운영자가 로그를 읽을 수 있습니다.

테스트도 같은 방식으로 하는 편이 낫습니다. 처음부터 1,000건을 자동 처리하면 원인을 놓치기 쉽습니다. 50건 샘플로 시작해 실패 유형을 먼저 세는 방식이 더 안전합니다.

| 측정 항목 | 첫 테스트 기준 | 보는 이유 |
|---|---:|---|
| 자동 처리율 | 60~70% | 너무 높으면 검토가 빠졌을 수 있음 |
| 사람 검토율 | 10~25% | 위험 작업이 걸러지는지 확인 |
| 중복 처리율 | 5% 이하 | 메모리와 상태 관리 검증 |
| 재시도 횟수 | 작업당 2회 이하 | 무한 루프 방지 |
| 오발송 건수 | 0건 | 외부 신뢰와 직결 |

## 협업은 프롬프트보다 계약입니다

좋은 에이전트 협업은 역할 설명이 길지 않습니다. 대신 입력, 출력, 권한, 실패 처리가 분명합니다. 사람 조직의 업무 인수인계와 비슷합니다.

각 에이전트에는 최소한 네 가지 계약이 필요합니다.

| 계약 항목 | 예시 |
|---|---|
| 입력 | 신규 Gmail 스레드 24시간치 |
| 출력 | 리드 점수, 근거 3개, 추천 액션 |
| 권한 | 읽기 가능, 외부 발송 불가 |
| 실패 처리 | 확신도 0.7 미만이면 사람 검토 |

이 계약이 없으면 에이전트 수가 늘어날수록 디버깅 비용도 커집니다. 한 에이전트의 환각이 다음 에이전트의 입력이 되고, 마지막 에이전트는 그럴듯한 보고서를 보낼 수 있습니다.

AI 에이전트 협업의 목적은 모든 일을 자동화하는 데 있지 않습니다. 반복 작업은 백그라운드로 보내고, 위험한 판단은 멈추게 만드는 데 있습니다. 패턴을 볼 때도 역할 분담보다 실패를 어디서 끊을지 먼저 확인해야 합니다.
