## vica 개발일지: 재시작에 강한 실행 흐름과 조금 더 작아진 책임들

이번 작업은 크게 두 축이었습니다. 하나는 데몬이 재시작되더라도 진행 중이던 런을 최대한 자연스럽게 복구하는 안정성 개선이고, 다른 하나는 점점 비대해지던 실행·서빙 쪽 코드를 작게 나누는 리팩터링이었습니다. 겉으로 보이는 기능 추가보다 내부 체력을 올리는 작업이 많았던 하루였습니다.

### 데몬 재시작 후 런 복구 확장

가장 중요한 변화는 데몬 재시작 시 큐에 있던 작업과 실행 중이던 작업을 복구할 수 있도록 한 부분입니다. 이전에는 프로세스가 내려가면 메모리 안에 있던 큐 상태나 in-flight 상태가 같이 사라지는 구조였고, 그래서 재시작 이후 “무엇을 다시 실행해야 하는가”를 안정적으로 판단하기 어려웠습니다.

이번에는 큐와 실행 중 상태를 영속화하는 방향으로 확장했습니다. 덕분에 데몬이 다시 올라왔을 때 저장된 상태를 읽고, 아직 완료되지 않은 런을 재실행 흐름에 태울 수 있게 되었습니다. 물론 이런 복구 로직은 항상 조심스럽습니다. 이미 끝난 작업을 다시 실행하면 안 되고, 반대로 실행이 필요했던 작업을 놓쳐서도 안 됩니다. 그래서 상태를 남기는 지점과 다시 큐에 올리는 기준을 최대한 명확히 가져가려고 했습니다.

개인적으로는 이 작업이 vica의 “운영 가능한 도구”로서의 성격을 조금 더 강하게 만들어준 것 같습니다. 단순히 잘 실행되는 것보다, 중간에 끊겨도 회복할 수 있는지가 실제 사용 경험에서는 훨씬 크게 다가오니까요.

### Serve 입력 전처리 분리

Serve 쪽도 계속 다이어트를 진행했습니다. 이번에는 입력 전처리 책임을 별도의 `InputIntake` 협력자로 분리했습니다. 기존 Serve 흐름은 요청을 받고, 입력을 해석하고, 전처리하고, 이후 실행 단계로 넘기는 역할까지 한곳에 꽤 많이 모여 있었습니다. 처음에는 빠르게 만들기 좋지만, 시간이 지나면 작은 변경도 전체 흐름을 건드리는 느낌이 강해집니다.

입력 전처리를 협력자로 빼면서 Serve는 조금 더 “흐름을 조율하는 역할”에 가까워졌고, 입력을 받아 정리하는 책임은 별도 단위에서 다룰 수 있게 되었습니다. 테스트나 추후 확장 면에서도 이쪽이 더 편합니다. 입력 형태가 늘어나거나 전처리 규칙이 바뀌더라도 Serve 본문을 계속 키우지 않아도 되기 때문입니다.

이런 분리는 당장 눈에 띄는 기능 변화는 아니지만, 나중에 디버깅할 때 차이가 큽니다. 문제가 생겼을 때 “요청 처리 문제인지, 입력 해석 문제인지, 실행 문제인지” 경계를 잡기가 쉬워집니다.

### runOne 축소와 래칫 조임

실행 경로의 핵심인 `runOne`도 계속 줄였습니다. 한때 거의 모든 책임을 끌어안고 있던 메서드였는데, 이번에 길이를 더 줄이고 래칫 기준도 조였습니다. 87줄에서 64줄까지 줄인 것은 단순한 숫자 이상의 의미가 있습니다. 메서드가 짧아졌다는 건, 그 안에서 동시에 생각해야 하는 조건과 상태가 줄었다는 뜻이기도 합니다.

물론 줄 수만 줄이는 리팩터링은 위험할 수 있습니다. 이름만 바꿔서 복잡도를 다른 곳으로 옮기는 경우도 있으니까요. 그래서 이번에는 흐름을 억지로 쪼개기보다, 자연스럽게 독립된 판단과 동작을 바깥으로 빼는 데 집중했습니다. 그리고 래칫을 조여서 다시 커지지 않도록 안전장치도 같이 걸었습니다. “이번에만 정리하고 끝”이 아니라, 앞으로도 이 수준을 유지하자는 의도를 코드에 남긴 셈입니다.

### 마무리

이번 세 커밋은 모두 vica를 더 단단하게 만드는 작업이었습니다. 데몬 재시작 복구는 사용자가 체감할 수 있는 안정성을 높였고, Serve와 runOne 리팩터링은 앞으로의 변경 비용을 낮췄습니다. 특히 복구 기능처럼 상태를 다루는 작업은 주변 코드가 복잡하면 금방 위험해지는데, 동시에 책임 분리를 진행한 덕분에 다음 단계로 나아갈 발판도 조금 더 좋아졌습니다.

아직 정리할 곳은 남아 있습니다. 하지만 실행 흐름의 중심부가 조금씩 작아지고, 재시작 같은 현실적인 상황을 견딜 수 있게 되는 과정이 보여서 꽤 만족스러운 작업이었습니다.
