## TUI에서 “보이는 품질”을 조금 더 챙긴 날

이번 작업은 크게 보면 **터미널 UI의 시각적 완성도 개선**과 **대화 흐름 회귀 수정**으로 묶을 수 있었습니다. 기능 자체를 크게 늘렸다기보다는, 실제로 `vica`를 켜고 사용할 때 “아, 이제 좀 덜 거슬린다” 싶은 부분들을 정리한 하루였습니다.

### 코드 하이라이팅과 라이브 영역 여백 정리

먼저 TUI 쪽에는 코드 하이라이팅을 넣었습니다. 이전에는 코드 블록이 일반 텍스트와 크게 다르지 않게 보여서, 응답을 읽을 때 구조를 빠르게 파악하기가 조금 불편했습니다. 특히 모델이 설정 예시나 스크립트, JSON 같은 내용을 길게 출력하면 어디가 설명이고 어디가 코드인지 눈으로 계속 분리해야 했습니다.

이번에는 터미널 안에서도 코드 영역이 더 명확하게 보이도록 하이라이팅을 적용했습니다. 웹 UI처럼 화려한 느낌을 목표로 했다기보다는, CLI 환경에서 읽기 흐름을 방해하지 않는 선에서 “코드는 코드답게” 보이도록 하는 데 초점을 맞췄습니다.

그리고 라이브 응답 영역 위쪽 여백도 손봤습니다. 이건 작은 변화지만 체감이 꽤 컸습니다. 응답이 바로 붙어서 시작되면 화면이 답답해 보이고, 이전 입력과 현재 출력의 경계가 흐릿해지는 문제가 있었는데요. 위쪽에 적당한 숨 쉴 공간을 주니 대화 단위가 조금 더 자연스럽게 나뉘었습니다. 터미널 UI는 픽셀 단위의 화려함보다 줄 간격과 여백이 훨씬 중요하다는 걸 다시 느꼈습니다.

### clarify 흐름 회귀 수정

이번에 가장 중요했던 수정은 `clarify` 흐름의 회귀였습니다. 원래 의도는 사용자의 요청이 모호할 때 자연스럽게 확인 질문을 던지는 것이었는데, 특정 상황에서 이 흐름이 너무 공격적으로 동작했습니다. 사용자는 간단히 뭔가를 시키고 싶은데, 프로그램이 마치 취조하듯 계속 캐묻는 느낌이 되어버렸고, 심지어 그 과정에서 런이 죽는 문제까지 있었습니다.

이건 사용자 경험상 꽤 치명적이었습니다. “더 좋은 답을 하기 위해 질문한다”는 기능이 “진행을 막는 장벽”처럼 느껴지면 안 되니까요. 그래서 clarify가 필요한 상황과 그렇지 않은 상황을 다시 정리하고, 잘못된 분기 때문에 실행이 중단되는 부분을 바로잡았습니다.

함께 발견된 문제로 질문이 여러 번 중복 노출되는 현상도 있었습니다. 같은 확인 질문이 세 번씩 보이면 아무리 좋은 기능이어도 신뢰가 떨어집니다. 이번 수정에서는 질문 출력 흐름을 정리해서 한 번만 명확하게 보이도록 했습니다. 이런 버그는 기능보다 인상에 더 큰 영향을 주는 경우가 많아서, 늦지 않게 잡아서 다행이었습니다.

### 이미지 프로토콜이 없는 터미널에서 로고 개선

마지막으로 로고 표시 품질도 개선했습니다. 이미지 프로토콜을 지원하는 터미널에서는 비교적 깔끔하게 이미지를 보여줄 수 있지만, 그렇지 않은 환경에서는 문자 기반으로 대체 표시를 해야 합니다. 문제는 이때 로고가 흐릿하거나, 여백 때문에 작아 보이거나, 형태가 어색해지는 경우가 있었다는 점입니다.

이번에는 불필요한 여백을 트림하고, 셀 단위 표시를 더 크게 가져가고, 서브셀 문자도 활용해서 터미널 문자 기반 출력의 밀도를 높였습니다. 완벽한 이미지 렌더링은 아니지만, 적어도 “대충 깨져 보이는 로고”가 아니라 “터미널에 맞춰 의도적으로 표현된 로고”에 가까워졌습니다.

특히 CLI 도구에서 첫 화면 로고는 생각보다 중요합니다. 기능과 직접 관련은 없어 보여도, 사용자가 처음 마주치는 인상이니까요. 작은 디테일이지만 프로젝트가 덜 실험작처럼 보이고, 조금 더 제품처럼 느껴지게 만드는 부분이라고 생각합니다.

### 정리

이번 커밋들은 모두 `vica`를 더 편하게 읽고, 더 안정적으로 쓰고, 더 보기 좋게 만드는 쪽에 가까웠습니다. 코드 하이라이팅과 여백은 읽는 경험을 개선했고, clarify 수정은 대화 흐름의 신뢰도를 회복했으며, 로고 렌더링 개선은 다양한 터미널 환경에서의 첫인상을 다듬었습니다.

큰 기능 하나를 만든 날은 아니었지만, 이런 날들이 쌓여서 도구가 실제로 손에 익는다고 믿습니다. 오늘은 특히 “터미널에서도 UX는 UX다”라는 말을 계속 떠올리게 되는 작업이었습니다.
