# 데이터는 정직하게, 자동화는 끝까지: vica 개발일지

이번 작업은 겉으로 드러나는 기능을 늘리기보다, **vica가 결과를 만들고 검증하는 과정에서 지켜야 할 신뢰성**을 다듬는 데 집중했습니다. 테스트나 데모를 위해 생성한 데이터가 실제 사실처럼 전달될 가능성을 막고, 자율 실행 모드가 중간 승인 단계에서 멈추지 않도록 보완했습니다. 여기에 긴 빌드 작업의 출력 처리 기준도 현실적으로 조정했습니다.

## 생성 데이터와 실제 사실의 경계를 강제했습니다

가장 중요하게 손본 부분은 데이터 정직성 게이트입니다. 기존 흐름에서는 시드 기반 난수처럼 재현 가능한 조작 데이터를 만들 수 있었는데, 문제는 그 데이터가 테스트용이라는 맥락을 잃은 채 실제 사실처럼 출하될 여지가 있었다는 점입니다.

재현 가능하다는 것은 테스트에 유용하다는 뜻이지, 데이터가 사실이라는 뜻은 아닙니다. 하지만 생성 과정이 안정적이고 결과의 모양도 그럴듯하면 이 차이를 놓치기 쉽습니다. 이번에는 단순히 안내 문구를 추가하는 수준이 아니라, **조작된 데이터를 사실로 취급하는 흐름 자체를 게이트에서 차단**하도록 근본 경로를 봉합했습니다.

이 변경으로 테스트 데이터나 예시 데이터의 사용이 금지되는 것은 아닙니다. 대신 출처와 성격이 불분명한 데이터를 실제 결과물로 넘기지 않는다는 원칙을 코드가 직접 지키게 됐습니다. 조금 엄격해 보일 수 있지만, 데이터 기반 결과를 다루는 도구라면 편의성보다 먼저 지켜야 할 기준이라고 판단했습니다.

## 자율 실행이 셸 승인에서 멈추지 않도록 했습니다

`--assume`을 사용하는 자율 실행은 사용자의 반복 개입 없이 작업을 완주해야 합니다. 그런데 브라우저 자동화나 E2E 검증 과정에서 셸 실행이 필요한 경우, 별도의 승인 게이트가 남아 있어 무인 실행이 중간에 멈출 수 있었습니다.

이번 수정에서는 자율 모드의 의도를 일관되게 적용해, 셸이 필요한 단계도 자동 승인하도록 연결했습니다. 덕분에 브라우저를 실행하고 검증 명령을 수행한 뒤 결과를 확인하는 전체 흐름을 사람의 추가 입력 없이 끝까지 진행할 수 있습니다.

물론 모든 실행에서 승인을 없앤 것은 아닙니다. 일반 모드의 안전장치는 유지하면서, 사용자가 명시적으로 자율 실행을 선택한 경우에만 그 선택이 셸 단계까지 이어지도록 의미를 맞췄습니다. 옵션 이름은 자율인데 실제로는 중간에 멈추던 어색함을 해소한 셈입니다.

## 긴 빌드 로그를 안정적으로 수용했습니다

빌드 턴의 출력 상한은 64,000으로 높였습니다. 최근 빌드와 E2E 작업은 의존성 설치, 번들링, 브라우저 로그 등이 한 번에 쌓이면서 기존 한도를 쉽게 넘길 수 있었습니다. 출력이 잘리면 실패 원인을 확인하기 어렵고, 자동 검증 역시 불완전한 로그를 기준으로 판단하게 됩니다.

아울러 이 상한이 전체 실행 동안 누적되는 값이 아니라 **호출할 때마다 리셋되는 한도**라는 점도 명확히 했습니다. 장시간 작업에서 이전 호출의 출력 때문에 다음 호출의 여유가 줄어드는 것으로 오해하지 않도록 동작 기준을 분명히 정리했습니다.

이번 변경들은 화려하지 않지만 서로 연결돼 있습니다. 정직한 데이터를 만들고, 자율 검증을 끝까지 수행하며, 그 과정을 충분한 로그로 남기는 것. vica가 “결과를 냈다”는 데서 그치지 않고, **믿을 수 있는 방식으로 결과를 냈다**고 말하기 위한 기반을 한층 단단하게 만든 작업이었습니다.
