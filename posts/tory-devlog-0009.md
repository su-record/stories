# tory 개발일지 #9 - 대규모 리팩토링 + 패션 트렌드 (10개 커밋)

**작업 기간**: 2026-03-05

## 📝 이번 기간 작업 내용

### Secretary 682줄 분해 및 인터페이스 추출 (2개 커밋)

IWorkerPool 인터페이스를 추출해 Router가 구체 클래스 대신 인터페이스에 의존하도록 바꿨다. 테스트에서 목(mock)을 주입하기 위한 준비다.

Secretary 모듈은 682줄 단일 파일을 7개 파일로 분해했다. 경로 핸들러, 트리거, RAG, 스킬 매칭이 각자 파일을 가진다. 단일 파일이 682줄이 된 시점에서 분해는 선택이 아니라 필수였다. 각 파일이 하나의 책임을 갖도록 경계를 그었다.

| 커밋 | 내용 |
|------|------|
| `refactor: IWorkerPool 인터페이스 추출 — Router가 구체 클래스 대신 인터페이스에 의존` | 의존성 역전 적용 |
| `refactor: Secretary 682줄 분해 — 경로 핸들러·트리거·RAG·스킬매칭 7개 파일 추출` | Secretary 모듈 분해 |

### Auth 프록시 및 SSE 공통화 (2개 커밋)

데스크톱 앱의 인증 흐름을 변경했다. Desktop이 Server에 직접 연결하던 구조를 Desktop→Daemon→Server 경유 구조로 바꿨다. API 기반 주소가 localhost:9100으로 통일된다. 인증 로직이 Daemon 한 곳에서 관리되므로 데스크톱과 서버 양쪽에 분산됐던 인증 코드를 줄일 수 있다.

SSE 파서는 3개 파서에서 중복되던 청크 처리 로직을 `sse_lines()` 헬퍼로 추출했다. 파서마다 같은 코드를 반복 유지할 필요가 없어졌다.

| 커밋 | 내용 |
|------|------|
| `refactor: Auth 프록시 — Desktop→Daemon→Server 경유 인증, API_BASE localhost:9100` | 인증 경로 중앙화 |
| `refactor: SSE 파서 공통화 — sse_lines() 헬퍼 추출, 3개 파서 중복 제거` | SSE 처리 헬퍼 추출 |

### 패션 트렌드 모니터링 (1개 커밋)

패션 트렌드 모니터링 기능을 추가했다. 트렌드 라우트, 스케줄러, 데스크톱 UI 컴포넌트로 구성된다. 스케줄러가 주기적으로 트렌드 데이터를 수집하고, 데스크톱 앱에서 결과를 확인할 수 있다. 도메인 특화 기능이지만 구조는 범용 트렌드 모니터링 프레임워크를 따른다.

| 커밋 | 내용 |
|------|------|
| `feat: 패션 트렌드 모니터링 — 트렌드 라우트 + 스케줄러 + Desktop UI 컴포넌트` | 패션 트렌드 수집·표시 구현 |

### 코드 리뷰 수정 및 품질 개선 (4개 커밋)

문서 업데이트 후 코드 리뷰 P1~P3를 처리했다. Auth 보안을 강화하고 SSE 메모리 사용을 최적화했으며 중복을 제거했다. Secretary 테스트 7건은 phase meeting 경로로 유도하는 텍스트 키워드를 보정해 통과시켰다. Store 마이그레이션 테스트의 기대값도 v4에서 v6로 업데이트했다. P1 품질 개선으로 main.ts와 store.ts를 분리하고 의존성을 정리했다.

## 💡 작업 하이라이트

**Secretary 682줄 분해 — 모듈 경계를 긋는 기준**

파일이 커진다고 무조건 분해하는 게 아니다. 분해는 책임이 명확히 나뉠 수 있을 때 의미가 있다. Secretary는 경로 핸들링, 트리거 판정, RAG 검색, 스킬 매칭이라는 네 가지 다른 일을 한 파일에서 처리하고 있었다. 각 책임이 독립적으로 변경될 수 있다는 판단이 섰을 때 분해를 결정했다.

7개 파일로 나누니 각 파일이 단일 책임을 갖는다. Secretary 코어는 조합 레이어만 남는다. 새 트리거 규칙을 추가하거나 RAG 검색 로직을 바꿀 때 다른 파일을 건드릴 필요가 없다.

IWorkerPool 인터페이스 추출도 같은 맥락이다. 구체 클래스에 의존하면 테스트에서 대체할 수 없다. 인터페이스를 경계로 삼으면 테스트용 구현체를 주입할 수 있다.
